# CLAUDE.md — точка входа для LLM

Ты работаешь над **AI Hospitality Platform** — мультитенантной SaaS-платформой, где AI является основным интерфейсом между гостями отеля, персоналом и внутренними сервисами. Это НЕ чат-бот: чат — лишь один из каналов, ядро — оркестрация гостиничных процессов.

## Обязательно перед любой задачей

1. Прочитай [FOUNDATION.md](FOUNDATION.md) — разделы 4 (Принципы P-1…P-12) и 12 (Правила R-1…R-12). Они обязательны, нарушение = блокер.
2. Термины бери из [docs/GLOSSARY.md](docs/GLOSSARY.md) — не изобретай синонимов.
3. Работая с модулем — сначала его `README.md` и `api.py`.
4. Архитектурно значимые решения — только через ADR ([docs/adr/](docs/adr/)).
5. Читай точечно, а не подряд: эта карта → нужные разделы FOUNDATION → README и `api.py` затронутого модуля → канонический пример. Файлы задачи перечислены в её карточке (PHASEN.md). Чтение репозитория «на всякий случай» — расход контекста без пользы.

## Ключевые факты (краткая карта)

- **Архитектура:** модульный монолит (ADR-001). Микросервисы запрещены без ADR.
- **Стек:** Python 3.12 + FastAPI + Pydantic v2 + SQLAlchemy 2 + PostgreSQL(+pgvector) + Redis; фронт — Next.js/TS (ADR-002).
- **Мультитенантность:** `tenant_id` + Postgres RLS в каждой тенантной таблице (ADR-003). Ни одного запроса вне контекста тенанта.
- **Интеграции:** только через порты и адаптеры (`integrations/`). Доменные модули не импортируют интеграции напрямую. У каждого порта — обязательный Fake-адаптер: полноценная локальная реализация для dev/CI/демо, рождается в одном PR с портом (ADR-007).
- **AI:** слой поверх бизнес-логики. Каждая способность AI — декларативный tool-обёртка над сервисом ядра. Промпты — код: версионируются и покрываются evals.
- **AI Gateway:** весь LLM-трафик — только через `ai/gateway` (маршрутизация моделей, лимиты, логирование стоимости и версий промптов). Прямые вызовы провайдеров запрещены.
- **Слои:** kernel (`platform/`, `shared/`) ← доменные модули (`modules/`) ← композиционный слой (`ai/`, `channels/`). Доменные модули никогда не импортируют `ai/`, `channels/` или `integrations/`.
- **Human-in-the-loop:** деньги, документы (АВР), изменение брони — только с подтверждением человека (P-9, NG-4).
- **Наблюдаемость:** структурированные JSON-логи с `tenant_id`/`correlation_id`, OpenTelemetry, коды ошибок из каталога. Код без логов/метрик — незавершён.

## Жёсткие запреты (из Non-Goals)

- Google Sheets/Excel — только экспорт, никогда не хранилище данных.
- Никаких неофициальных API WhatsApp.
- Не хранить платёжные данные карт.
- Не делать кастомный код под конкретный отель в ядре — только конфигурация тенанта.

## Правила работы

- Язык кода и идентификаторов — английский; документации — русский.
- Файл ≤ ~400 строк, строгая типизация (mypy strict), Pydantic-схемы на границах.
- Нетривиальная фича начинается со спеки в `docs/specs/`, а не с кода.
- Изменил модуль — обнови его README в том же PR.
- В БД — только UTC; наивные datetime запрещены. Локальное время — через часовой пояс отеля из конфига тенанта.
- Ревью кода выполняет отдельная LLM-сессия с чистым контекстом (FOUNDATION.md, раздел 13). Блокирующие правила — R-5, R-6, R-7, R-9; остальные — рекомендации.
- Задача противоречит FOUNDATION.md → остановись и сообщи, не решай молча (R-11).
- Завершая задачу, дай основателю «передачу смены»: что сделано и как проверено; что требуется от него (merge PR, секреты, доступы, ручные шаги); какая задача следующая по плану фазы (поле «Открывает» в PHASEN.md) и какой класс модели для неё нужен (правило 7 PROJECT_EXECUTION_PLAN).

## Контекст проекта

Основатель — не профессиональный разработчик; код пишут преимущественно LLM. Поэтому: простота сопровождения важнее элегантности, один канонический паттерн на каждую типовую задачу, никаких скрытых зависимостей и магии.
