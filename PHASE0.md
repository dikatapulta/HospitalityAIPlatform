# PHASE0 — Walking Skeleton: план задач

> **Статус:** действующий. Версия 1.0 от 2026-07-06.
> **Цель фазы:** сквозной поток «сообщение гостя в Telegram → AI → заявка → уведомление службы → выполнено», проходящий через полную инженерную инфраструктуру. Не фичи — фундамент и канонические примеры.
> **Формат:** одна задача = одна LLM-сессия (~2–4 часа) = один PR = зелёный CI = ревью отдельной сессией.
> **Правило:** задачи выполняются по порядку, если явно не указана возможность параллели. Если в ходе задачи обнаружено, что нужно «ещё чуть-чуть добавить» — это новая задача, а не расползание текущей.

## Карта зависимостей

```
0001 → 0002 → 0003 → 0004 ─┬→ 0005 → 0006 → 0007 → 0008 → 0009 ─┬→ 0010 → 0011
                            │                                      │
                            └──────── (0006 можно параллельно 0007–0009)
0011 → 0012 → 0013 ─┬→ 0014 → 0015 → 0016 → 0017 → 0018 → 0019 → 0020
                     └→ (0014 можно параллельно 0013)
```

Сквозные правила Foundation для **каждой** задачи (не повторяются ниже): P-1 (простота), P-12/R-10 (канон), R-3 (размер файлов, английский код), R-6 (mypy strict, Pydantic на границах), R-9 (DoD = зелёный CI). В полях задач указаны только правила, специфичные для неё.

### Task 0001 — Репозиторий и каркас проекта

- **Цель:** git-репозиторий с защищённым `main`, структура каталогов из FOUNDATION §15, каркас Python-проекта (pyproject, ruff, mypy strict, pytest) и один проходящий фиктивный тест.
- **Почему сейчас:** всё остальное — коммиты; без инструментов качества первый же код родится вне правил.
- **Файлы (создаются):** `.gitignore`, `pyproject.toml`, `ruff.toml`/конфиг в pyproject, `mypy.ini`/конфиг, `src/` каркас каталогов с `__init__.py`, `tests/test_sanity.py`, `README.md` (как запустить проверки).
- **Модули:** каркас всех, код — ни одного.
- **Правила Foundation:** структура §15 без отклонений.
- **Тесты:** `test_sanity` (импортируется пакет, 1==1).
- **CI:** ещё нет (появится в 0002); локально `ruff`, `mypy`, `pytest` зелёные.
- **DoD:** `git log` показывает первый коммит; локальные проверки проходят одной командой (`make check`).
- **Открывает:** Task 0002.

### Task 0002 — CI-пайплайн

- **Цель:** GitHub Actions (или аналог): ruff + mypy + pytest + порог покрытия на каждый PR; merge в `main` только через PR с зелёным CI.
- **Почему сейчас:** CI до кода — иначе enforcement придётся «натягивать» на уже написанное. Это материализация R-9.
- **Файлы:** `.github/workflows/ci.yml`, бейдж в `README.md`.
- **Правила:** R-9 (DoD = зелёный CI — с этого момента буквально).
- **Тесты:** прежние; проверка — CI падает на нарочно сломанном типе (проверить и откатить).
- **CI:** сам пайплайн.
- **DoD:** PR с ошибкой типа блокируется; PR без ошибок мержится.
- **Открывает:** Task 0003.

### Task 0003 — Импорт-линтер: границы слоёв

- **Цель:** import-linter с контрактами из FOUNDATION §5.1: kernel ← modules ← composition; запрет импорта `integrations/` из модулей; запрет циклов; заготовка правила «LLM-провайдеры только из `ai/gateway`». Включён в CI.
- **Почему сейчас:** главная защита конституции должна существовать раньше первого модуля, иначе первый модуль её и нарушит.
- **Файлы:** конфиг import-linter (в pyproject), правка `ci.yml`; фиктивные пакеты-маркеры слоёв, если нужны для контрактов.
- **Правила:** R-5 — эта задача и есть R-5.
- **Тесты:** нарочный тест-нарушение (модуль импортирует integrations) → линтер падает → нарушение удаляется.
- **CI:** job линтера границ обязателен.
- **DoD:** нарушение слоёв невозможно смержить.
- **Открывает:** Task 0004.

### Task 0004 — Локальная среда одной командой

- **Цель:** `docker compose up` (обёрнуто в `make dev`): Postgres 16 + pgvector, Redis, контейнер приложения с hot-reload. Одна команда — работающая среда.
- **Почему сейчас:** каждая следующая сессия начинается с запуска среды; трение здесь умножается на все будущие задачи.
- **Файлы:** `ops/docker-compose.yml`, `ops/Dockerfile`, `Makefile` (`dev`, `check`, `test`), `.env.example`.
- **Правила:** секреты не в репозитории (§11); скретч-данные не в проект.
- **Тесты:** smoke-скрипт: compose поднялся, Postgres и Redis отвечают.
- **CI:** сборка Docker-образа в пайплайне.
- **DoD:** с чистой машины: clone → `make dev` → среда работает; README описывает это в 5 строках.
- **Открывает:** 0005; параллельно можно готовить 0006.

### Task 0005 — Каркас приложения: FastAPI, настройки, health

- **Цель:** FastAPI-приложение в composition root; pydantic-settings (все настройки типизированы, читаются из env); `/health/live` и `/health/ready` (ping БД и Redis).
- **Почему сейчас:** health-эндпоинты нужны деплою (0006) и всей наблюдаемости; настройки — первый канон конфигурации.
- **Файлы:** `src/app.py` (composition root), `src/shared/config.py`, `src/shared/health.py`, тесты.
- **Правила:** P-7 (типизированные настройки); §10.6 (health).
- **Тесты:** health-эндпоинты через httpx TestClient; ready падает при недоступной БД.
- **CI:** прежние + новые тесты.
- **DoD:** `make dev` → `curl /health/ready` → 200 с JSON-статусами зависимостей.
- **Открывает:** 0006, 0007.

### Task 0006 — Staging и деплой с первого дня

- **Цель:** VPS со staging: деплой `main` автоматически (или одной командой), секреты через env на сервере, health-check после деплоя. Деплоится текущий hello-world каркас — и это правильно.
- **Почему сейчас:** «деплой — рутина, а не событие» дешевле всего сделать, пока деплоить нечего. Все следующие задачи сразу видны на staging.
- **Файлы:** `ops/deploy/` (compose для staging, скрипт деплоя), `docs/runbooks/deploy.md`, secrets-чеклист.
- **Правила:** §10.11 (инфраструктура кодом); §11 (секреты).
- **Тесты:** post-deploy smoke: health-эндпоинты staging отвечают.
- **CI:** deploy job на push в `main` (или ручной триггер).
- **DoD:** merge в main → изменение на staging без ручных шагов; runbook позволяет повторить настройку сервера с нуля.
- **Открывает:** непрерывную проверку всего последующего на staging. Может идти параллельно 0007–0009.

### Task 0007 — Логирование, correlation id, каталог ошибок

- **Цель:** structlog с JSON и обязательными полями (§10.1: timestamp, level, tenant_id, correlation_id, module, event); middleware correlation id; базовые классы ошибок с кодами (`AppError(code=...)`) и первые записи в каталоге ошибок.
- **Почему сейчас:** канон логирования и ошибок должен существовать до первого бизнес-кода — ретрофит логов по всем модулям не делает никто и никогда.
- **Файлы:** `src/shared/logging.py`, `src/shared/errors.py`, middleware в `src/shared/`, `docs/runbooks/errors.md` (первые коды: ERR-PLATFORM-001…), тесты.
- **Правила:** §10.1–10.2, §10.5, R-8.
- **Тесты:** запись лога содержит обязательные поля; correlation id из заголовка пробрасывается; AppError сериализуется в ответ API с кодом.
- **CI:** прежние.
- **DoD:** любой запрос виден в логах staging как структурированный JSON с correlation id.
- **Открывает:** 0008.

### Task 0008 — Слой БД: SQLAlchemy, Alembic, канонический паттерн сессии

- **Цель:** SQLAlchemy 2 (typed), Alembic, канонический паттерн сессии/транзакции (единственный способ получить сессию — §6), первая миграция: таблица `tenants`.
- **Почему сейчас:** паттерн сессии — фундамент RLS (0009); он должен быть единственным до того, как появится второй способ.
- **Файлы:** `src/shared/db.py` (canonical session), `src/platform/models.py` (Tenant), `alembic/` + первая миграция, тесты.
- **Правила:** §9 (только миграции; UTC-типы колонок), §6 (паттерн сессии).
- **Тесты:** миграция применяется на чистую БД; сессия отдаёт/возвращает соединение корректно; naive datetime в колонке — ошибка ревью (зафиксировать в каноне).
- **CI:** job «alembic upgrade head» на чистом Postgres (сервис в CI).
- **DoD:** `make dev` + `alembic upgrade head` → таблица tenants существует; в коде один способ работать с БД.
- **Открывает:** 0009.

### Task 0009 — Контекст тенанта и RLS + тест изоляции

- **Цель:** middleware контекста тенанта; `SET LOCAL` в транзакции по канону §6; RLS-политика на первой тенантной таблице; **обязательный тест изоляции** (тенант A не видит данных B, запрос без контекста — ошибка); tenant_id в каждой лог-записи.
- **Почему сейчас:** ретрофит тенантности — ошибка №1 из «фатальных» (ревью). Дальше каждая таблица копирует этот канон.
- **Файлы:** `src/shared/tenancy.py` (в карточке изначально стоял `platform/` — направление слоёв R-5 требует shared, см. докстринг модуля), RLS в миграции, правка `src/shared/db.py` и logging, `tests/test_tenant_isolation.py`.
- **Правила:** P-4, ADR-003 — задача реализует их буквально.
- **Тесты:** тест изоляции (двунаправленный); тест «без контекста — исключение»; тест утечки контекста через пул (два последовательных запроса разных тенантов).
- **CI:** тест изоляции помечен обязательным (падение = блокер навсегда).
- **DoD:** невозможно прочитать чужие данные даже нарочно написанным «плохим» запросом через канонический слой.
- **Открывает:** 0010, 0011.

### Task 0010 — События: outbox, шина, воркер (+ADR-005)

- **Цель:** мини-ADR-005 (выбор arq/celery — 30 минут, решение зафиксировать); таблица outbox; публикация `DomainEvent` в транзакции; воркер читает outbox и вызывает подписчиков; идемпотентность обработчиков; канонический пример события.
- **Почему сейчас:** P-6 — способ связи модулей; он нужен до второго модуля, иначе модули свяжутся напрямую.
- **Файлы:** `docs/adr/ADR-005-worker.md`, `src/shared/events.py`, миграция outbox, `src/worker.py`, тесты.
- **Правила:** P-6, P-8 (идемпотентность), §5.3 (воркер = тот же образ).
- **Тесты:** событие публикуется атомарно с бизнес-записью; повторная доставка не дублирует эффект; событие переживает падение воркера.
- **CI:** тесты воркера в пайплайне; воркер добавлен в compose (dev и staging).
- **DoD:** событие, опубликованное в API-процессе, обработано в воркере на staging; след в логах с общим correlation id.
- **Открывает:** 0011, 0012.

### Task 0011 — Модуль platform: конфиг тенанта и демо-тенант

- **Цель:** конфигурация тенанта с `schema_version` (§6); сид демо-тенанта «Demo Hotel» (профиль, часовой пояс, язык) для dev и staging; `make seed`.
- **Почему сейчас:** все последующие задачи тестируются на демо-тенанте; часовой пояс отеля нужен канону времени.

- **Правила:** §6 (schema_version), R-4 (первый паспорт модуля).
- **Тесты:** сид идемпотентен (повторный запуск не дублирует); конфиг валидируется схемой.
- **CI:** прежние + тесты.
- **DoD:** `make dev && make seed` → демо-тенант существует, `/health/ready` знает о БД, конфиг читается сервисом.
- **Открывает:** 0012.

### Task 0012 — Модуль requests: канонический доменный модуль

- **Цель:** модуль `requests/` в полной анатомии §5.2: `README.md`, `api.py` (публичный интерфейс), `models.py` (ServiceRequest, RequestCategory — с RLS), `service.py` (создание заявки, смена статуса по жизненному циклу), `events.py` (`request.created`, `request.status_changed`), `schemas.py`, `tests/`. Помечается как **canonical module**.
- **Почему сейчас:** это главный образец фазы — все будущие модули копируют его.
- **Файлы:** весь `src/modules/requests/`, миграция таблиц, тесты.
- **Правила:** §5.2 (анатомия), P-6 (события, не прямые вызовы), P-4 (RLS на новых таблицах — копия канона 0009).
- **Тесты:** жизненный цикл статусов (валидные/невалидные переходы); события публикуются; изоляция тенантов на новых таблицах.
- **CI:** импорт-линтер подтверждает: модуль не лезет в чужие внутренности.
- **DoD:** заявку можно создать вызовом сервиса в тесте; README модуля полон; в коде маркер canonical.
- **Открывает:** 0013; 0014 можно вести параллельно.

### Task 0013 — HTTP API заявок: канонический эндпоинт

- **Цель:** REST-эндпоинты `/api/v1/requests` (создать, список, сменить статус) поверх `requests/api.py`; канон: схемы запрос/ответ, коды ошибок из каталога, пагинация, OpenAPI, аутентификация статическим сервисным токеном из env (§11: эндпоинт рождается аутентифицированным; RBAC — Phase 1). Помечается **canonical endpoint**.
- **Почему сейчас:** канон эндпоинта нужен до кабинета персонала (Phase 1) и вообще любого API.
- **Файлы:** роутер в `src/modules/requests/`, подключение в composition root, тесты, новые коды в `docs/runbooks/errors.md`.
- **Правила:** P-7, §13.5 (версия `/v1/` с первого дня), R-8 (ошибки с кодами).
- **Тесты:** happy path + ошибки (несуществующая категория → ERR-REQ-001 и т.д.) + изоляция тенантов через API + запрос без токена / с неверным токеном → 401.
- **CI:** прежние.
- **DoD:** заявка создаётся curl-ом на staging и видна в списке; OpenAPI-схема корректна.
- **Открывает:** 0017 (частично); параллельно 0014–0016.

### Task 0014 — AI Gateway

- **Цель:** `ai/gateway`: единственная точка вызова LLM. Интерфейс провайдера, реализация Anthropic (одна модель, без маршрутизации), mock-провайдер для тестов; логирование каждого вызова в `LlmCallLog` (тенант, correlation id, хэш промпта, токены, стоимость, latency); таймауты/ретраи; правило импорт-линтера «anthropic импортируется только внутри gateway» включается по-настоящему.
- **Почему сейчас:** «ни один токен мимо gateway» (§7.2) должно быть физически верно с первого AI-вызова.
- **Файлы:** `src/ai/gateway/` (интерфейс, anthropic-адаптер, mock), миграция LlmCallLog, конфиг import-linter, тесты.
- **Правила:** §7.2 полностью; R-5 (правило линтера); NG — без маршрутизации моделей (запрещено раньше времени).
- **Тесты:** на mock: вызов логируется со стоимостью и хэшем промпта; ретрай при таймауте; бюджетный лимит тенанта (простейший: превышение → отказ).
- **CI:** линтер ловит прямой импорт anthropic вне gateway (проверить нарочным нарушением).
- **DoD:** единственный способ поговорить с LLM — gateway; каждый вызов виден в БД и логах.
- **Открывает:** 0015.

### Task 0015 — Оркестратор и первый инструмент (канонический tool)

- **Цель:** минимальный оркестратор §7.1: принимает нормализованное сообщение, контекст тенанта/гостя, вызывает LLM через gateway с одним инструментом `create_service_request` (класс `confirm_guest` — AI переспрашивает подтверждение). Промпт — файл в `ai/prompts/` с версией. Инструмент помечается **canonical tool**.
- **Почему сейчас:** канон «инструмент = обёртка над сервисом ядра» (P-5) — самый важный AI-паттерн проекта; он должен родиться на простейшем случае.
- **Файлы:** `src/ai/orchestrator.py`, `src/ai/tools/` (реестр + create_request), `src/ai/prompts/concierge_v1.md`, тесты.
- **Правила:** P-5, P-9 (класс подтверждения в контракте инструмента), §7.5 (промпт — код).
- **Тесты:** на mock-провайдере: сообщение «уберите номер» → вызов инструмента → заявка в БД; сообщение-вопрос → ответ без инструмента; отказ при неподтверждении.
- **CI:** прежние; evals — заготовка (3 сценария на mock, помечены как golden set v0).
- **DoD:** в тесте полный путь «текст → LLM(mock) → инструмент → сервис requests → заявка»; на staging то же с настоящей моделью вручную.
- **Открывает:** 0016.

### Task 0016 — Канал Telegram

- **Цель:** `channels/telegram`: вебхук с проверкой секрета, **идемпотентность** по update_id, нормализация в единый формат сообщения, модели Conversation/Message (с RLS), отправка ответов. Привязка чата к демо-тенанту (маппинг в конфиге).
- **Почему сейчас:** первый реальный вход в систему; идемпотентность канала (P-8) — канон для WhatsApp/Email в будущем.
- **Файлы:** `src/channels/telegram/`, `src/channels/base.py` (нормализованное сообщение — контракт для всех каналов), миграция conversations/messages, тесты.
- **Правила:** P-8 (дубликаты вебхука), P-7 (контракт нормализации), §8.4 (подписи вебхуков).
- **Тесты:** дубликат update не создаёт второго сообщения; неверный секрет → 403; нормализация форматов (текст; не-текст → вежливый отказ).
- **CI:** прежние.
- **DoD:** сообщение реальному боту на staging появляется как Message в БД с correlation id.
- **Открывает:** 0017.

### Task 0017 — Сквозная сборка: гость → AI → заявка → уведомление → выполнено

- **Цель:** соединить всё: входящее сообщение Telegram → оркестратор → инструмент → заявка → событие `request.created` → воркер-подписчик шлёт уведомление в staff-чат Telegram (простейшее «уведомление службы» без UI) → сотрудник командой в чате закрывает заявку → гость получает подтверждение.
- **Почему сейчас:** это и есть Walking Skeleton — момент, когда скелет впервые идёт.
- **Файлы:** подписчик уведомлений (в `channels/telegram` или тонкий `notifications`-подписчик), staff-команды бота, интеграционный тест сквозного потока.
- **Правила:** P-6 (уведомление — подписчик события, не вызов из requests), P-5.
- **Тесты:** интеграционный: от вебхука до уведомления на mock-провайдере; статусные переходы через staff-команды.
- **CI:** интеграционный тест в пайплайне.
