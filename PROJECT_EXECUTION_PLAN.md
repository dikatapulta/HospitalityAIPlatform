# PROJECT_EXECUTION_PLAN — Генеральный план строительства платформы

> **Статус:** действующий. Версия 1.0 от 2026-07-06. Основан на Architecture Freeze v1 (FOUNDATION.md).
> **Роль:** инженерный план строительства. Отвечает на вопросы: что строить, в каком порядке, почему именно так, когда можно переходить дальше. Это не roadmap продукта и не архитектура.
> **Связь с FOUNDATION.md:** конституция заморожена и не оспаривается этим планом. Фазы исполнения детализируют фазы раздела 16 конституции (мэппинг в разделе «Соответствие фаз»).

## Правила исполнения (действуют на всех фазах)

1. **Phase Gate.** Следующая фаза не начинается, пока не выполнены Exit Criteria текущей. Частичный переход запрещён — он порождает два недостроенных этажа вместо одного построенного.
2. **Одна задача = одна LLM-сессия = один PR.** Задача имеет номер, входит в план фазы, завершается зелёным CI и ревью отдельной сессией (FOUNDATION 13.1).
3. **Строим вглубь, а не вширь.** Лучше один сквозной работающий поток, чем пять начатых модулей.
4. **Ничего «на всякий случай».** Если задача не нужна для Exit Criteria текущей фазы и её можно безопасно отложить — она откладывается. Список «запрещено раньше времени» в каждой фазе обязателен к соблюдению.
5. **Деплой с первых дней.** Staging живёт с Phase 0; всё, что смержено, автоматически оказывается на staging. Продакшен появляется в Phase 1 и с этого момента деплой — рутина, а не событие.
6. **Мультитенантность не откладывается.** `tenant_id`, RLS и контекст тенанта существуют с первого коммита (ADR-003). Поздняя фаза «Multi-tenant SaaS» — это продуктизация онбординга и биллинг, а не добавление тенантности.
7. **Модель по классу задачи.** Задачи, создающие каноны или принимающие решения (ADR, canonical-примеры, security-критичный код: tenancy, идентичность, платежи), и ревью уровня A (правило 9) выполняются самой сильной доступной моделью. Задачи, копирующие существующий канон, допустимо выполнять дешёвой моделью: их качество держат канон и CI, а не модель. Правило: экономить там, где ошибку ловит CI; не экономить там, где ошибка сама становится каноном.
8. **Аудит на выходе фазы.** Exit Criteria каждой фазы включают независимый аудит отдельной LLM-сессией с чистым контекстом (сильная модель): проверка Exit Criteria, соответствия конституции и честности документации — по образцу Task 0020. Аудит включает трассировку 2–3 сквозных пользовательских сценариев (гость, сотрудник): каждый шаг цепочки — от входящего сообщения до последнего события — должен иметь владельца (модуль), контракт и тест; шаг без владельца — находка аудита. Каждый PHASEN.md завершается такой задачей. Найденное фиксируется через ADR/issue, не правится молча (R-11).
9. **Ревью каждого PR — независимой сессией; уровень ревьюера — по классу изменения.** Самопроверка автора ревью не является: контекст, породивший ошибку, её же и одобрит. Уровни:
   - **Уровень A (сильная модель):** миграции и схема БД; kernel (`shared/`, `platform/`); `ai/gateway` и промпты; поверхности безопасности (auth, RLS, вебхуки, секреты); новые или изменённые контракты (`api.py`, события, схемы, порты); новые модули, адаптеры, инструменты, эндпоинты, каналы; конфигурация CI/линтеров; FOUNDATION, ADR и канонические примеры.
   - **Уровень B (дешёвая модель, но обязательно отдельная сессия с чистым контекстом):** изменения внутри одного модуля, не трогающие его контракты и миграции; тесты; README модулей; мелкие фиксы.
   Порядок: автор указывает уровень в описании PR по списку выше; сомневаешься — уровень A. Ревьюер уровня B, увидевший в diff контракт, миграцию или границу слоя, обязан остановиться и эскалировать до уровня A. Вердикт ревью фиксируется комментарием в PR (approve / changes requested); основатель мержит только после approve — он принимает поведение (smoke-набор, staging), а не код. Занижение уровня, обнаруженное ревьюером или аудитом фазы, — блокер. Ролей ревью больше не становится: уровень кода — это правило, уровень архитектуры — аудит фазы (правило 8), уровень релиза — smoke-набор + staging + фиче-флаг (FOUNDATION §13.1, §13.6).

## Phase 0 — Walking Skeleton

**Главная цель:** сквозной поток «сообщение гостя в Telegram → AI → заявка → уведомление службы → выполнено» проходит через **полную** инженерную инфраструктуру: CI с enforcement, тенант-контекст с RLS, наблюдаемость, AI Gateway, события, staging. Не фичи — фундамент и канонические примеры, которые будут копировать все последующие фичи (P-12).

- **Инженерные задачи:** детально — [PHASE0.md](PHASE0.md) (Task 0001–0020).
- **Модули:** `shared/`, `platform/` (тенант, конфиг), `requests/` (единственный доменный модуль), `ai/` (gateway, orchestrator, один tool), `channels/telegram`.
- **Разрешённые интеграции:** Telegram (официальный Bot API), Anthropic API. Других интеграций в Phase 0 нет — первые порты и их Fake-адаптеры появляются с Phase 1 (ADR-007); LLM в тестах — сценарный mock-провайдер gateway.
- **Запрещено раньше времени:** любой UI; RAG и pgvector-логика; второй канал; второй доменный модуль; маршрутизация моделей в Gateway (одна модель); реальные адаптеры PMS/карт/погоды; оптимизация чего бы то ни было.
- **Документы:** README модулей `requests/`, `ai/`, `channels/`; runbook деплоя и восстановления; каталог ошибок (первые коды).
- **ADR:** ADR-005 (воркер фоновых задач).
- **Закрываемые риски:** «конституция — только текст» (появляется enforcement); «LLM-сессии строят по-разному» (появляются каноны); «деплой как событие» (появляется рутина).
- **Exit Criteria:** сквозной сценарий проходит на staging; smoke-набор v0 зелёный в CI; импорт-линтер контролирует слои и Gateway; тест тенант-изоляции существует и зелёный; восстановление из бэкапа отрепетировано один раз; все канонические примеры помечены.
- **Что открывается:** любая следующая фича — копирование канона, а не изобретение; можно строить Phase 1 параллельными сессиями без разрушения архитектуры.

## Phase 1 — Core Hotel Loop (пилотный отель: операционка)

**Главная цель:** реальный отель ежедневно использует платформу для заявок. Ценность с минимальным AI-риском: приём и маршрутизация заявок — это то, где ошибка стоит дёшево и заметна сразу.

- **Инженерные задачи:** все категории заявок как конфигурация (housekeeping, инженерия, IT, F&B, жалобы) с SLA и эскалациями; модель `Guest`/`GuestIdentity`; эскалация диалога человеку (и обратно); **канал Web-чат гостя** (перенесён из Phase 2, issue #79: демография пилота — ~70% гостей из стран, где Telegram/WhatsApp непопулярны или заблокированы, пилот только на Telegram дал бы нерепрезентативную метрику; §5.1 «Web (гость)», приоритет выше WhatsApp): статический QR в номере — только точка входа, идентифицирует отель/комнату, не гостя; аутентификация — персональный per-stay токен, выданный при заселении и истекающий при выезде (механизм — ADR-008); без валидной привязки чат отвечает только на публичные вопросы и не создаёт заявок (тест: истёкшая сессия не может действовать); кабинет персонала v1 (Next.js: лента заявок, статусы, «взять в работу»); база знаний отеля v1 + RAG (AI отвечает на вопросы об отеле с опорой на источники); модуль `feedback/`; RBAC v1 + `docs/RBAC.md`; `docs/PII_REGISTRY.md` с первой PII-миграцией; golden set evals v1 (заявки + вопросы об отеле; языки — демография тенанта, ADR-010: для пилота 6 языков + сценарии злоупотребления web-каналом + **класс «адверсориал-ввод на неродном языке»** из ai-аудита 23.07); чек-лист онбординга отеля v0 (выполняется разработчиком вручную); продакшен-окружение.
- **Модули:** `guests/`, `feedback/`, `concierge/` (только знания отеля, без города), расширение `requests/`, `channels/web`.
- **Разрешённые интеграции:** Email (уведомления), Fake PMS (ADR-007) или CSV-импорт проживаний. Параллельно (бизнес-процессы, не код): подача заявки на WhatsApp BSP, начало общения с Oracle по OHIP — у обоих долгий lead time.
- **Запрещено раньше времени:** платежи; документы/АВР; рекомендации по городу и внешние карты; запись в PMS; самообслуживание онбординга; аналитика сверх простых счётчиков.
- **Документы:** RBAC.md, PII_REGISTRY.md, runbook эскалаций, обновление README затронутых модулей.
- **ADR:** **ADR-006 (хостинг и резидентность данных РК — провайдер Kazteleport; до продакшена, дедлайн локализации 25.08.2026)**; ADR-008 (модель идентичности и доступа — **до** RBAC v1 и web-чата, инварианты в FOUNDATION §11); старт исследования для ADR-004 (PMS-синхронизация).
- **Закрываемые риски:** «продукт не нужен отелю» (проверяется на живых людях раньше, чем построено дорогое); «AI галлюцинирует правила отеля» (RAG + evals); «утечка PII» (реестр + маскирование до реальных данных гостей).
- **Exit Criteria:** пилотный отель живёт на платформе ≥ 4 недель; ≥ 70% заявок проходит без участия ресепшена; Web-чат по QR работает на пилоте (гость без Telegram может создать заявку); эскалация к человеку срабатывает надёжно; evals зелёные; продакшен с бэкапами (шифрованными, issue #81) и алертами в Telegram.
- **Что открывается:** доверие отеля и реальные данные об использовании — сырьё для решений Phase 2–3; отработанный деплой в прод.

## Phase 2 — AI Concierge (гостевой опыт)

**Главная цель:** AI становится ценным для гостя, а не только для персонала: город, маршруты, погода, такси, late checkout, допуслуги.

- **Инженерные задачи:** знания о городе в базе знаний; порты `MapsPort`, `WeatherPort`, `TaxiPort` + первые реальные адаптеры; инструменты late checkout и заказа услуг (класс `confirm-staff`: подтверждает ресепшен, PMS обновляется вручную) — опираются на привязку `GuestIdentity` → `Stay` через per-stay токен, построенную в Phase 1 (веб-чат, ADR-008): без неё инструменты не знают, чью бронь трогают; upsell-подсказки; канал WhatsApp (официальный BSP, заявка из Phase 1; отдельный выделенный номер под ИИ-канал — не номер, которым уже пользуется отель/персонал); полировка мультиязычности; расширение golden set (консьерж-сценарии).
- **Модули:** `stays/` (late checkout, услуги), расширение `concierge/`.
- **Разрешённые интеграции:** Google Maps, 2GIS, Weather API, такси-провайдер, WhatsApp BSP.
- **Запрещено раньше времени:** автоматическая запись в PMS; платежи (заказ услуг — «оплата на ресепшене»); Wallet-пассы; персонализация сверх `GuestFact`.
- **Документы:** README новых адаптеров (лимиты API, получение ключей), обновление каталога ошибок кодами интеграций.
- **ADR:** выбор такси-провайдера и картографии по городам (если неочевидно — мини-ADR).
- **Закрываемые риски:** «интеграционная архитектура — теория» (порты проверяются тремя непохожими адаптерами); «деградация не работает» (нет погоды/карт — консьерж продолжает работать, проверяется тестами).
- **Exit Criteria:** гости реально пользуются консьержем (метрика диалогов/проживание); две категории допродаж работают через `confirm-staff`; отключение любого внешнего API не ломает диалог; WhatsApp в проде.
- **Что открывается:** доказанные допродажи → экономическое основание для платежей (Phase 3); паттерн адаптера отточен для сложных интеграций (Phase 4).

## Phase 3 — Business Automation (деньги и документы)

**Главная цель:** платформа начинает трогать деньги и юридически значимые документы — с человеком в контуре на каждом необратимом шаге.

- **Инженерные задачи:** модуль `documents/`: загрузка PDF/фото реквизитов → извлечение → валидация схемой (БИН, ИИК/IBAN) → **подтверждение бухгалтером** → генерация АВР → экспорт в Google Sheets (`SpreadsheetExportPort`); защита от prompt injection в пайплайне документов (недоверенный ввод, изоляция парсинга в воркере); платежи Kaspi (`PaymentPort`, токены — NG-3); Wallet-пассы (Apple/Google); неизменяемость финансовых записей (append-only + аудит).
- **Модули:** `documents/`, расширение `stays/` (оплачиваемые услуги).
- **Разрешённые интеграции:** Kaspi, Google Sheets, Apple/Google Wallet.
- **Запрещено раньше времени:** прямая интеграция с 1С (сначала Sheets-экспорт докажет корректность данных); автоматический выпуск АВР без подтверждения (NG-4 — навсегда); хранение карточных данных (NG-3 — навсегда).
- **Документы:** runbook инцидентов с платежами; обновление PII_REGISTRY (реквизиты компаний).
- **ADR:** ADR платёжного потока (кто инициирует, идемпотентность, возвраты).
- **Закрываемые риски:** «двойное списание» (идемпотентность платежей); «юридически неверный документ» (human-in-the-loop); «injection через загруженный PDF» (валидация + изоляция).
- **Exit Criteria:** бухгалтерия пилотного отеля принимает экспорт без ручных исправлений ≥ 2 недель; первый оплаченный через платформу заказ; пентест-сценарии на injection в документы пройдены.
- **Что открывается:** выручка через платформу → аргумент для продаж SaaS; данные для интеграции с 1С (Phase 4) уже выверены.

## Phase 4 — Deep Integrations (тяжёлые внешние системы)

**Главная цель:** заменить ручные/CSV-адаптеры настоящими: Opera PMS, 1С, Channel Manager. Самая рискованная интеграционная фаза — поэтому она идёт тогда, когда порты уже отточены, а продукт уже ценен и без неё.

- **Инженерные задачи:** реализация ADR-004 (синхронизация PMS: вебхуки + reconciliation-джоб + разрешение конфликтов); адаптер Opera (OHIP); адаптер 1С (замена Sheets-экспорта, порт тот же); Channel Manager; приоритетные очереди LLM-трафика в Gateway; AI-черновики инцидент-репортов; модуль `analytics/` v1 (витрины для менеджера/директора).
- **Модули:** `analytics/`; замена адаптеров без изменения ядра — главный экзамен архитектуры.
- **Разрешённые интеграции:** Opera/OHIP, 1С, Channel Manager.
- **Запрещено раньше времени:** двусторонняя запись в PMS до стабильного одностороннего чтения + reconciliation; выключение Sheets-экспорта до месяца параллельной работы с 1С.
- **Документы:** runbook расхождений PMS-синхронизации (главный операционный документ фазы).
- **ADR:** финализация ADR-004; ADR по 1С-обмену.
- **Закрываемые риски:** «копия данных PMS расходится с истиной» (reconciliation + алерты на расхождения); «смена адаптера требует переписывания» (проверяется заменой Sheets→1С).
- **Exit Criteria:** проживания синхронизируются с Opera без ручных сверок ≥ 4 недель; расхождения детектируются автоматически; 1С принимает документы; старые адаптеры удалены (не закомментированы — удалены).
- **Что открывается:** платформа технически готова к отелям «с улицы» — у них будут свои PMS и свои бухгалтерии.

## Phase 5 — Multi-tenant SaaS (тиражирование)

**Главная цель:** подключение отеля перестаёт требовать разработчика. Тенантность уже есть — фаза строит продукт вокруг неё.

- **Инженерные задачи:** мастер онбординга (self-service: профиль → знания → интеграции → персонал → чек-лист); биллинг на основе метеринга (данные копятся с Phase 0); тарифы и лимиты; оффбординг (экспорт + удаление) как работающий код; инструменты миграции конфигов тенантов; изоляционное нагрузочное тестирование (noisy neighbor); базовый суппорт-процесс (кто отвечает отелю №7 в 3 часа ночи).
- **Модули:** расширение `platform/` (billing, onboarding); admin-панель платформы.
- **Разрешённые интеграции:** платёжный провайдер для подписки SaaS.
- **Запрещено раньше времени:** marketplace и plugin-API (NG-8); индивидуальные доработки под отдельного клиента в ядре (NG-7).
- **Документы:** публичная документация для отелей; SLA-политика.
- **ADR:** модель биллинга; политика поддержки версий конфигурации.
- **Закрываемые риски:** «каждый новый отель — проект» (self-service); «нельзя выставить счёт» (метеринг→биллинг); «уход клиента — кризис» (оффбординг).
- **Exit Criteria:** отель подключается без коммитов в репозиторий; ≥ 5 живых тенантов; первый счёт выставлен и оплачен; оффбординг отрепетирован.
- **Что открывается:** продажи ограничены маркетингом, а не инженерией.

---

## Phase 6 — Enterprise Platform (масштаб и команда)

**Главная цель:** платформа выдерживает enterprise-клиентов (сети отелей) и приход профессиональной команды.

- **Инженерные задачи:** мульти-отельные тенанты (сети) — по ADR; высокая доступность (реплики БД, zero-downtime deploy); внешний аудит безопасности/пентест; соответствие процедур комплаенсу (доказуемая изоляция, аудит-трейл для due diligence); онбординг людей-разработчиков (FOUNDATION уже написан для них — проверить, что он честен); при доказанном спросе — ADR о plugin-API/marketplace (снятие NG-8 осознанным решением).
- **Запрещено раньше времени:** ничего из этой фазы не делается раньше — всё здесь дорого и оправдано только масштабом.
- **Exit Criteria:** enterprise-сделка проходит технический due diligence; новый разработчик-человек самостоятельно доставляет фичу в первую неделю.
