# ADR-001: Модульный монолит вместо микросервисов

- **Статус:** accepted
- **Дата:** 2026-07-06

## Контекст

Платформа задумана как долгоживущая (10–15 лет) мультитенантная SaaS. Разработку и сопровождение ведут преимущественно LLM; основатель — не профессиональный разработчик. «Enterprise» интуитивно ассоциируется с микросервисами.

## Рассмотренные варианты

1. **Микросервисы.** Плюсы: независимый деплой и масштабирование. Минусы: распределённые транзакции, сетевые сбои, оркестрация, дублирование инфраструктуры, сложность отладки — эксплуатационная нагрузка, непосильная для команды без выделенных SRE; LLM хуже работают с логикой, размазанной по репозиториям и сетевым границам.
2. **Классический монолит без границ.** Плюсы: просто начать. Минусы: за годы превращается в клубок скрытых зависимостей — ровно то, что запрещают требования проекта.
3. **Модульный монолит.** Одно приложение, внутри — модули с машинно-проверяемыми границами (импорт-линтер), общение через публичные API модулей и события.

## Решение

Вариант 3: модульный монолит. Границы модулей проектируются так, как проектировались бы границы сервисов, поэтому при доказанной необходимости отдельный модуль можно выделить в сервис без переписывания ядра.

## Последствия

- Один деплоймент, одна БД, простая эксплуатация и отладка (один trace, один лог-поток).
- Дисциплина границ обеспечивается CI (import-linter), а не договорённостями.
- Пересмотр: если конкретный модуль упрётся в масштабирование или потребует независимого релизного цикла — выделение через новый ADR.
