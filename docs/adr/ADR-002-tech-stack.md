# ADR-002: Технологический стек

- **Статус:** accepted
- **Дата:** 2026-07-06

## Контекст

Код пишут преимущественно LLM; каждая новая сессия не должна выбирать технологии заново (P-12: один способ сделать вещь). Нужны: сильная экосистема AI и интеграций, строгая типизация контрактов, минимальная эксплуатационная сложность.

## Рассмотренные варианты

1. **TypeScript/NestJS full-stack.** Один язык на фронт и бэк, но слабее экосистема AI/данных, тяжелее ORM-дисциплина.
2. **Python + FastAPI.** Лучший язык для LLM-генерации, Pydantic даёт явные схемы на границах (P-7), богатейшая экосистема AI и парсинга документов.
3. **Go.** Отличная эксплуатация, но беднее экосистема AI и дороже скорость итераций.

## Решение

- **Бэкенд:** Python 3.12+, FastAPI, Pydantic v2, SQLAlchemy 2 + Alembic, mypy strict.
- **Данные:** PostgreSQL 16+ с pgvector (данные + outbox + векторы знаний — одна СУБД), Redis (кэш, rate limits), объектное хранилище для файлов.
- **Фоновые задачи:** воркер на том же кодовом образе (arq или celery — зафиксировать отдельным мини-ADR в Фазе 0).
- **LLM:** Claude (Anthropic) как основной, строго за провайдер-абстракцией слоя `ai/`.
- **Фронтенд (персонал/админ):** Next.js + TypeScript.
- **Наблюдаемость:** structlog + OpenTelemetry + Sentry + Prometheus/Grafana.
- **Инфраструктура v1:** Docker Compose на VPS/managed-сервере с учётом резидентности данных РК; managed PostgreSQL. Kubernetes — только через ADR.

## Последствия

- Все примеры, каноны и шаблоны кода пишутся под этот стек.
- Смена любого элемента стека — новый ADR со ссылкой на этот.
