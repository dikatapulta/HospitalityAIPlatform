# ADR-003: Модель мультитенантности — общая схема + tenant_id + RLS

- **Статус:** accepted
- **Дата:** 2026-07-06

## Контекст

v1 — один отель, но цель — тысячи тенантов без изменения ядра. Утечка данных между отелями — фатальный для SaaS-репутации инцидент, поэтому изоляция не может держаться только на дисциплине кода.

## Рассмотренные варианты

1. **БД на тенанта.** Максимальная изоляция, но миграции и эксплуатация на сотнях БД — непосильно для маленькой команды.
2. **Схема на тенанта.** Средний вариант; миграции по сотням схем всё ещё болезненны.
3. **Общая схема, `tenant_id` в каждой тенантной таблице + Postgres Row-Level Security.** Одна миграция на всех, изоляция принудительно на уровне СУБД: даже ошибочный запрос без фильтра не вернёт чужих данных.

## Решение

Вариант 3. Правила:
- Каждая тенантная таблица имеет NOT NULL `tenant_id` с RLS-политикой.
- Контекст тенанта устанавливается middleware в начале запроса (`SET LOCAL`), включая фоновые задачи; выполнение запроса без контекста — ошибка.
- `tenant_id` — обязательное поле каждой лог-записи и метрики.
- Лимиты на тенанта (rate limits, бюджеты LLM) — часть platform-модуля.

## Последствия

- Дешёвые миграции и эксплуатация; изоляция проверяется автоматическими тестами (запрос под тенантом A не видит данных B).
- Для будущих VIP-клиентов с требованием физической изоляции возможен выделенный инстанс приложения+БД с той же схемой — без смены модели данных (новый ADR при появлении такого клиента).
