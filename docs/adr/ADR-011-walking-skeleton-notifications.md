# ADR-011: Сквозная доставка Phase 0 — уведомления, staff-команды, привязка заявки к диалогу

- **Статус:** accepted
- **Дата:** 2026-07-13
- **Автор:** LLM-сессия (Task 0017, Opus)

## Контекст

Task 0017 впервые соединяет скелет целиком: сообщение гостя в Telegram → заявка →
уведомление службе → закрытие сотрудником → подтверждение гостю. Все доменные
кирпичи готовы (Task 0012–0016) и **менять их нельзя** без веской причины — цель
задачи в том числе проверить, что контракты стыкуются (R-11). Возникают три
архитектурно значимых решения композиционного слоя, которых ещё нет в репозитории:

1. **Где живут подписчики-уведомления** и как не нарушить P-6 (уведомление —
   подписчик события, не вызов из `requests`).
2. **Как сотрудник закрывает заявку** без UI (кабинет персонала — Phase 1).
3. **Как подтверждение возвращается гостю**, если событие `request.status_changed`
   не знает канал и чат гостя.

Ограничения: P-2 (модули общаются через public API/события), P-5 (AI — слой над
сервисами), P-6 (события для побочных эффектов), P-8 (идемпотентность), §5.1
(направление слоёв: `channels/` — композиционный слой, видит доменные модули;
обратное запрещено), NG: нет модуля `guests/` и идентичностей в Phase 0.

## Рассмотренные варианты

### A. Куда положить подписчиков-уведомления

1. **Прямой вызов из `requests.service` после создания заявки** — ❌ нарушает P-6,
   связывает домен с каналом, ломает §5.1 (`modules → channels` запрещён).
2. **Отдельный модуль `notifications/`** — чисто, но в Phase 0 единственный канал
   доставки — Telegram, и уведомление физически шлётся его отправителем; отдельный
   модуль-посредник сейчас — лишний слой без второго потребителя.
3. **Тонкие подписчики в `channels/telegram/notifications.py`, регистрируются
   composition root воркера** — ✅ уведомление действительно принадлежит каналу
   (шлёт его `TelegramSender`), P-6 соблюдён (подписчик события), §5.1 соблюдён
   (`channels` → `modules.requests.api` разрешён). PHASE0 прямо допускает этот
   вариант («в `channels/telegram` или тонкий `notifications`-подписчик»).

### B. Как сотрудник закрывает заявку

1. **HTTP API + кабинет** — это Phase 1, не walking skeleton.
2. **Reply на сообщение-уведомление** (Telegram reply → resolve по
   `external_message_id`) — удобнее для персонала, но требует хранить обратный
   индекс «message_id уведомления → request_id» и разбирать reply-контекст; лишняя
   механика для скелета.
3. **Явные команды `/verb <request_id>` в staff-чате** — ✅ просто, детерминированно,
   по одной команде на переход `STATUS_TRANSITIONS` (карта переходов модуля requests
   не обходится — скелет честно проходит жизненный цикл new→assigned→in_progress→done).

### C. Как вернуть подтверждение гостю (обратная адресация)

1. **Добавить в событие `request.status_changed` ссылку на канал/чат** — ❌ домен
   получил бы знание о каналах; событие описывает доменный факт, а не маршрут
   доставки. Нарушает P-2/P-5 и смысл события.
2. **Поле `conversation_id`/источник на `ServiceRequest`** — ❌ доменная таблица
   ссылается на таблицу канала; `requests` начинает знать про `channels`.
3. **Канал ведёт свою таблицу привязки `request_origins(request_id →
   conversation_id)`** — ✅ обратная адресация — забота композиционного слоя;
   домен не тронут; `request_id` хранится как непрозрачная ссылка из события
   (без FK на `service_requests` — канал не связывает свою схему с чужой). В
   Phase 1 вытесняется резолвом идентичности гостя (`guests/` + `GuestIdentity`).

Смежное: **состояние гейта подтверждения P-9 (`pending_action`)** между двумя
вебхуками. Вариант «хранить в Redis/памяти» — теряется при рестарте и вводит вторую
СУБД состояния; выбрано хранить в колонке `conversations.pending_action` (диалог и
так в Postgres, состояние живёт с диалогом).

## Решение

Всё новое — в композиционном слое, доменные модули не меняются.

1. **Подписчики-уведомления живут в `channels/telegram/notifications.py`** и
   регистрируются в `hospitality/worker.py` (`register_subscribers`), как
   `include_router` в `app.py`. Уведомление службе — подписчик `request.created`;
   подтверждение гостю — подписчик `request.status_changed` (только `→ done`). P-6
   соблюдён: `requests` о них не знает.
2. **Сотрудник закрывает заявку командами `/assign` `/start` `/done` `/cancel
   <request_id>` в staff-чате** (`TELEGRAM_STAFF_CHAT_ID`). Канал по chat_id
   отличает staff-чат от гостевого и в staff-чате трактует текст как команду →
   `requests.change_request_status`, а не как реплику гостю оркестратору.
3. **Канал ведёт тенантную таблицу `request_origins`** (`request_id →
   conversation_id`, RLS-канон 0002, `request_id` без FK). Подписчик
   подтверждения находит по ней чат гостя. Состояние гейта P-9 — колонка
   `conversations.pending_action` (JSONB, NULL = нет ожидания).

Идемпотентность подписчиков (P-8) — через исходящий `Message` с естественным
ключом (`staff:request_created:<id>`, `guest:request_done:<id>`): повторная
доставка события не порождает второго уведомления.

## Последствия

- **Проще:** скелет проходит целиком без правок домена; новый потребитель события —
  подписчик в своём слое + строка в `register_subscribers`, модуль `requests`
  неизменен. Канонический паттерн «исходящее уведомление гостю в его канале»
  построен и переиспользуется (напоминания о выезде и т.п. — DISCUSSION_LOG).
- **Сложнее / долги Phase 1:**
  - `request_origins` и `TELEGRAM_STAFF_CHAT_ID` — временная замена идентичности
    гостя и кабинета персонала. При появлении `guests/` + `GuestIdentity` (Phase 1)
    обратная адресация переезжает на резолв идентичности; `request_origins`
    помечается устаревшим (ADR superseded).
  - staff-команды в чате — заглушка UI; RBAC нет (любой в staff-чате может
    закрывать заявки) — приемлемо для одного демо-чата Phase 0, закрывается RBAC v1
    (§17.7, ADR-008).
  - Доставка «best-effort + at-least-once»: подписчик при сбое отправки
    поднимает исключение → воркер ретраит с backoff (ADR-009); ключ идемпотентности
    гасит дубль на штатной пере-доставке. Окно двойной отправки (успех отправки →
    сбой записи) редкое и задокументировано — точная exactly-once доставка в внешний
    мессенджер не является целью Phase 0.
- **Миграция:** 0009 — колонка `conversations.pending_action` + таблица
  `request_origins` (RLS-канон).
- **Пересмотреть, когда:** появляется второй канал доставки уведомлений (вынести
  подписчиков в `notifications/` над портом отправки) или модуль `guests/`
  (заменить `request_origins` идентичностью гостя).
