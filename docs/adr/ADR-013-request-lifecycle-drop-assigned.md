# ADR-013: Жизненный цикл заявки без статуса `assigned`

- **Статус:** accepted (реализация смержена: PR #85, 2026-07-19; FOUNDATION §5.2
  обновлён там же)
- **Дата:** 2026-07-18
- **Автор:** LLM-сессия (spec 0021, Fable)

## Контекст

Жизненный цикл заявки зафиксирован в FOUNDATION §5.2 как
`new → assigned → in_progress → done/cancelled` и реализован в
`modules/requests` (Task 0012). Живой прогон на staging (issue #75) показал:
персонал маленького отеля (служба 1–2 человека) не различает «назначено» и
«в работе» и спотыкается о лишний шаг `/assign` перед `/start` — в логах серия
`ERR-REQUESTS-003` от команд не в том порядке. Штраф платится на каждой заявке,
а выгоды состояние не приносит: в Phase 0 нет ни пользователей-исполнителей,
ни очередей, ни SLA — «assigned» ничего не означает.

Ограничения: P-9 (каждая смена статуса — осознанное действие человека), P-1
(сопровождаемость важнее полноты «на вырост»), NG-8 (не строим машинерию впрок),
§13.5 (политика совместимости контрактов: удаление значения — брейк).

## Рассмотренные варианты

1. **Оставить `assigned`, обучить персонал.** Плюсы: ничего не менять.
   Минусы: пилот уже показал, что трение реально, а состояние пусто по смыслу;
   каждый новый сотрудник отеля будет спотыкаться заново; «обучить пользователей
   вместо упрощения продукта» — путь в NG-7-мышление.
2. **Сделать `assigned` необязательным** (добавить ребро `new → in_progress`,
   `/assign` оставить). Плюсы: контракт не ломается. Минусы: два пути для
   одного действия (нарушение P-12), мёртвое состояние остаётся в карте
   переходов, тестах, статистике и головах; путаница «а надо ли /assign»
   никуда не девается.
3. **Убрать `assigned` из жизненного цикла** (выбран). Цикл:
   `new → in_progress → done/cancelled`. Плюсы: минимальная модель, понятная
   персоналу («взять» → «закрыть»); меньше кода, тестов и кодов ошибок; Phase 1
   вводит назначение как **атрибут** `assignee` (кто взял/кому передали), а не
   как проходной статус — это ортогонально стадии работы и не возвращает шаг.
   Минусы: миграция данных; брейк контракта HTTP API (значение исчезает из
   enum) и FOUNDATION §5.2.

## Решение

Статус `assigned` удаляется из `RequestStatus` и `STATUS_TRANSITIONS`; жизненный
цикл — `new → in_progress → done/cancelled`, отмена возможна из любого
нетерминального статуса. «Назначение исполнителя» — будущий атрибут заявки
(Phase 1, кабинет персонала), не статус. P-9 сохраняется: переход
`new → in_progress` — явное действие человека («Взять в работу» / `/start`).

## Последствия

- **Проще:** персонал делает два действия вместо трёх; карта переходов, коды
  ошибок и тесты сокращаются; кнопочный UX (spec 0021, Проблема 2) строится на
  двух активных статусах.
- **Миграция данных:** `UPDATE service_requests SET status='in_progress' WHERE
  status='assigned'` (колонка — VARCHAR, `native_enum=False`: ALTER TYPE не
  нужен). Downgrade различие не восстанавливает (информация «назначено, но не
  начато» теряется необратимо) — зафиксировать в миграции.
- **Контракты:** HTTP API `POST /api/v1/requests/{id}/status` перестаёт
  принимать `assigned`. Это осознанное исключение из §13.5 «контракты только
  расширяются»: Phase 0, все потребители API — сам репозиторий (staff-чат,
  smoke, сиды); внешних интеграций и версионируемых клиентов нет. Такое
  исключение допустимо только до первого внешнего потребителя — после пилота
  подобный брейк потребует мажорной версии контракта.
- **Документация:** FOUNDATION §5.2 (строка цикла) поправлен в PR реализации
  (#85); README `modules/requests` и `channels/telegram` — там же; `/assign`
  в staff-чате отвечает подсказкой «шаг упразднён — используйте /start N»
  (переучивание, не молчание). Статус этого ADR переведён в accepted в PR #78
  после мержа реализации.
- **Пересмотреть, если:** Phase 1+ принесёт реальную потребность в стадии
  «принято диспетчером, но не начато» (например, крупный отель с диспетчерской).
  Тогда возвращать её следует не статусом, а моделью очереди/назначений поверх
  атрибута `assignee` — и через новый ADR.
