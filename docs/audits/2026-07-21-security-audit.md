# Аудит безопасности и надёжности — 21.07.2026

**Состояние кода:** main @ `dd20156`. **Метод:** пять параллельных read-only аудиторов
(AI-поверхность, Telegram-канал, RLS/мультитенантность, секреты/инфраструктура, outbox/надёжность)
плюс два контрольных прогона; находки перекрёстно подтверждены. Код не менялся.

**Вердикт.** Фундамент изоляции зрелый: RLS-канон (ENABLE+FORCE, NULLIF-политики, WITH CHECK)
применён ко всем тенантным таблицам, аутентификация fail-closed и constant-time, секретов в
git-истории нет, tenant_id недостижим из гостевого ввода. Активной межтенантной утечки нет.
Реальные риски перед пилотом — доступность/злоупотребление (открытый бот без rate-limit),
надёжность доставки уведомлений (outbox) и отсутствие идентичности гостя.

## Находки

### 🔴 Критические

**C1. Открытый бот + нет rate-limit + гонка бюджета → любой скрипт гасит консьержа всего отеля и жжёт деньги.**
Любой `chat_id` попадает в единственный тенант без allowlist (`channels/telegram/service.py:100-109`),
троттлинга нет (`channels/telegram/router.py`), единственный тормоз — общий дневной бюджет тенанта
~$5 (`shared/config.py:77`), проверка которого — TOCTOU: стоимость пишется в `LlmCallLog` после
ответа провайдера, поэтому параллельные вебхуки проходят гейт вместе (`ai/gateway/service.py:200-226`).
Исчерпание бюджета одним злоумышленником = ERR-AI-002 для всех гостей отеля. Секрет не нужен —
достаточно username бота. Подтверждено тремя аудиторами независимо.

### 🟠 Высокие

**H1. Живые секреты в рабочем `.env` внутри чекаута публичного репозитория.**
`.env` untracked и в `.gitignore`, git-история чиста (проверены все ветки и объекты), но в рабочей
копии лежат действующие ANTHROPIC_API_KEY, TELEGRAM_BOT_TOKEN, TELEGRAM_WEBHOOK_SECRET.
Один `git add -f` / swap-файл / tarball от утечки — и ключи публичны.

**H2. Сетевой I/O (LLM + Telegram) внутри одной batch-транзакции outbox, держащей FOR UPDATE.**
`deliver_pending_events` (`shared/events.py:198-223`) выбирает до 50 событий `FOR UPDATE SKIP LOCKED`
и в той же открытой транзакции гоняет переводы через LLM и HTTP к Telegram. Транзакция висит
минутами, держит блокировки и соединение; один зависший вызов или `idle_in_transaction_session_timeout`
откатывает всю пачку, включая уже доставленные, — они переотправятся.

**H3. «Отравленные» события молча выбрасываются после 10 попыток — без dead-letter и алерта.**
`shared/events.py:246-276`: после исчерпания попыток строка больше не выбирается, след — одна
лог-строка `event_delivery_exhausted`. Суммарный бэкофф ≈ 18 минут: недоступность Telegram/LLM
дольше — и гость никогда не узнает про выполненную заявку, персонал не увидит новую.

**H4. Спуфинг номера комнаты: `room_number` — свободный текст гостя без привязки чата к комнате.**
`ai/tools/create_service_request.py:104-107` → `modules/requests/service.py:99-145`: валидация только
по длине. «Почините сейф в 402» с чужой комнатой уходит персоналу как есть. Ограничение Phase 0
(модуля идентичности гостя нет), но нигде не зафиксировано как принятый риск.

### 🟡 Средние

**M1. Вся изоляция держится на обратимом `SET ROLE`, логин-роль — суперпользователь.**
`shared/db.py:106-112`; для суперпользователя FORCE RLS недействителен, `RESET ROLE`/`DISCARD ALL`
(например, дефолт pgbouncer) молча возвращает суперюзера. Нужна отдельная non-superuser LOGIN-роль
на staging/prod (уже гейт продакшена — issue #49).

**M2. Реестр `tenants` без RLS, app-роли выдан полный DML; `load_tenant_config` принимает любой tenant_id.**
`alembic/versions/0002_tenant_rls_canon.py:78`, `platform/config.py:95-102`. Активной утечки нет
(все вызовы передают `current_tenant_id()`), но любой будущий эндпоинт с клиентским id раскроет
чужой конфиг. Снять UPDATE/DELETE-грант или добавить политику.

**M3. Общеизвестный дефолт `service_token = "dev-service-token"` в коде** (`shared/config.py:38`).
Staging-compose форсит `${SERVICE_TOKEN:?}`, но голый uvicorn/локальный compose падают на дефолт.

**M4. Дубли уведомлений: check → send → record неатомарны** (`channels/telegram/notifications.py:166-175`).
Крах между отправкой и записью ключа идемпотентности (или откат пачки из H2) → переотправка.

**M5. Текст под влиянием гостя попадает в system-промпт confirmation-gate** (`ai/orchestrator.py:238-252`).
Ущерб ограничен: исполняется сохранённый `pending_action`, максимум — авто-подтверждение собственной
заявки. Убрать текст модели/гостя из system-секции.

**M6. Staff-авторизация — только членство в чате, без ролей и атрибуции** (`channels/telegram/service.py:77`).
Любой участник staff-группы закрывает любые заявки; «кто сделал» — только в логах. Принято для
демо-чата, станет проблемой с ростом (связано с #48).

**M7. Нет структурной гарантии, что новая тенантная таблица получит RLS.**
RLS включается копипастом `_apply_tenant_rls` в каждой миграции, а `ALTER DEFAULT PRIVILEGES`
(`0002_tenant_rls_canon.py:74-77`) заранее выдаёт app-роли полный DML на любую будущую таблицу:
забытый блок = таблица без RLS, читаемая между тенантами. CI-тест гоняет только канарейку;
мета-теста по `pg_class.relrowsecurity` для всех таблиц с `tenant_id` нет.

**M8. `/health/ready` — бесплатный DoS-рычаг на публичном URL.**
`shared/health.py:60-72`: свежий `asyncpg.connect` + новый Redis-клиент на каждый запрос, без
auth/rate-limit, доступен на staging.necturn.com. Цикл curl исчерпывает `max_connections` Postgres.

### 🟢 Низкие

- **L1.** Сырой 500 при исчерпании retry дневного номера (`modules/requests/service.py:111-155`).
- **L2.** `attempts` инкрементится и при успехе (`shared/events.py:277`) — исказит будущие метрики.
- **L3.** `find_open_requests_by_daily_number` не ограничен служебным днём (`service.py:190-207`) —
  `/done 5` может закрыть вчерашнюю #5. Связано с #83.
- **L4.** Swagger/OpenAPI (`/docs`, `/openapi.json`) открыты на публичном туннеле (`app.py:26`).
- **L5.** `POSTGRES_PASSWORD` в staging-compose не форсится через `:?` (БД наружу не смотрит).
- **L6.** `int(token)` в staff-командах без границ → int32 overflow → 500 (`staff.py:363-373`).
- **L7.** Нет лимита длины гостевого текста перед LLM/хранением (вектор стоимости).
- **L8.** Ответ на note-prompt определяется подстрокой русского текста (`staff.py:341-353`).
- **L9.** `session_scope` после rollback продолжает запросы без tenant-GUC (`store.py:52-62`) —
  fail-closed, но живой footgun при копировании паттерна.
- **L10.** Политика `platform_dispatch` для INSERT не ограничивает `tenant_id` строки
  (`0003_outbox_events.py:75-77`).
- **L11.** `/ready` обходит app-пул и RLS-роль (`shared/health.py:24-49`) — см. M8.
- **L12.** Webhook-секрет — единственный якорь доверия (нет IP-allowlist как defense-in-depth).
- **L13.** Устаревшее `ufw allow 8000/tcp` в `ops/deploy/bootstrap-server.sh:36-38` — порт сейчас
  не опубликован, но правило переживёт возврат `ports:`-маппинга и обойдёт Cloudflare TLS.

### Проверено и подтверждено сильным (не трогать)

Изоляция тенантов недостижима из гостевого ввода; аргументы AI-тулов Pydantic-валидируются, резолв
против данных тенанта; human-in-the-loop структурный (код, не промпт), денежных/документных тулов
нет (P-9/NG-4); переходы статуса под `FOR UPDATE` + карта переходов; дневной номер race-safe;
идемпотентность по `update_id`; секреты в git-истории отсутствуют; ошибки не текут внутренностями;
образ non-root; staging БД/Redis не опубликованы; CI-секреты через `secrets.*`.

## Дополнение 21.07 (вечер): седьмой прогон, отдельное окно LLM

### 🟠 H5. `LLM_MODEL` вне прайс-листа → отказ консьержа + платные неучтённые вызовы + бюджет вечно $0

`ai/gateway/service.py`: `provider.complete()` (деньги уже списаны) выполняется **до**
`_compute_cost`, который для модели не из `MODEL_PRICING_USD_PER_MTOK` (ровно 3 ключа-алиаса
без дат) бросает `ValueError` — до записи `LlmCallLog(OK)`. `ValueError` — не `AppError` →
HTTP 500 на каждое сообщение гостя; строка стоимости не пишется → `_ensure_tenant_budget`
вечно суммирует $0 и никогда не срабатывает. Триггер — поставить `LLM_MODEL` в датированный id
(`claude-haiku-4-5-20251001`), новую модель или опечатку; провайдер возвращает свой id модели,
и матчится именно он. Комментарий в коде («вызов падает до обращения к провайдеру») — неверен.
Проект часто меняет модель (bake-off Haiku↔Sonnet) — footgun правдоподобный.
**Фикс дёшев:** валидировать `LLM_MODEL` против прайс-листа на старте приложения (fail-fast)
и логировать вызов даже при неизвестной модели (cost NULL + алерт), чтобы бюджет не слеп.

### 🟡 M9. `make smoke` молча уничтожает dev-БД

`ops/smoke.sh:16-19`: `trap cleanup EXIT` → `compose down --volumes` выполняется и при успехе,
на том же `ops/docker-compose.yml` и томе `db_data`, что использует `make dev`. Запуск
`make smoke` после `make dev` необратимо стирает локальную БД разработчика. Только dev,
но легко нарваться. Фикс: отдельный project-name/том для smoke либо не трогать volumes.

### 🟡 M10. Необратимая миграция автоприменяется на деплое без бэкапа; CI-проверка обратимости — ложная зелень

`alembic/versions/0011_drop_assigned_status.py`: `downgrade()` — `pass` (различие
assigned/in_progress теряется навсегда). `ops/deploy/deploy.sh` гонит `alembic upgrade head`
на живой БД без backup-шага, а CI-проверка `downgrade base && upgrade head` идёт по пустому
Postgres — теряющий данные downgrade проходит зелёным. Сегодня латентно, но страховки нет,
а CI создаёт ложное ощущение её наличия. Фикс: `pg_dump` перед `alembic upgrade` в deploy.sh
(связано с #81 — бэкапы); в CI прогонять downgrade по засеянной БД или явно маркировать
необратимые миграции.

### 🟡 M11. У воркера нет liveness: зависший или крэш-лупящийся воркер невидим

`worker.py:61`: `register_subscribers()` вне try/except — ошибка конфигурации роняет процесс,
`restart: unless-stopped` даёт тихий крэш-луп с копящимися событиями; healthcheck'а у воркера
в `docker-compose.staging.yml` нет. Вместе с H2/H3 это значит: уведомления перестают доставляться,
и никто не узнаёт. Фикс — в один issue с H3 (алерт/dead-letter); связан с #89.

### Уточнение к H3 (в issue волны 2)

Retention чистит только `processed_at IS NOT NULL`, а исчерпанные события остаются навсегда →
неограниченный рост `outbox_events` (нарушение §9). Добавить в объём issue H3.
