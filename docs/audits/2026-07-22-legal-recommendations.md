# План приведения проекта в соответствие законодательству РК

Дата: 22.07.2026. База: юраудит (`2026-07-22-legal-audit-rk.md`), аудит безопасности (`2026-07-21-security-audit.md`), операционный аудит (`../audit/OPERATIONAL_AUDIT.md`), инвентаризация репозитория. Принцип: минимальный достаточный комплаенс для пилота одного отеля, без энтерпрайз-избыточности.

## Обзор: четыре пакета

| Пакет | Что | Объём | Когда |
|---|---|---|---|
| П1. Документы | 6 доков в репо + 1 вне | ~1 день LLM-работы | до пилота |
| П2. Код | consent-gate, маскирование, ретеншн + хардненинг из аудита 21.07 | 3–4 небольших PR | до пилота |
| П3. Инфраструктура | ADR-006 (хостинг РК), Task 0019 (бэкапы), мониторинг | 1 ADR + 1–2 PR + переезд | к запуску пилота |
| П4. Организационное | ИП/ТОО, ответственное лицо, договор с отелем | вне репо | до первых денег |

## П1. Документы (юридический слой)

1. **`docs/PII_REGISTRY.md`** — реестр ПД (заявлен в FOUNDATION §7.6, не создан). Таблица: категория данных → таблица/поле → цель → срок хранения → кому передаётся (Anthropic, Telegram, Cloudflare, хостер) → основание. Источник фактуры — инвентаризация из юраудита, реестр фактически наполовину готов.
2. **`docs/legal/privacy-policy.md`** (kk/ru/en) — политика конфиденциальности гостя: кто оператор, что собираем (тексты диалога, chat_id, номер комнаты — и что НЕ собираем: имена/телефоны/платежи), трансграничная передача, сроки, права субъекта, контакт. Публикуется на статичной странице (staging/prod домен) — ссылка из бота.
3. **`docs/legal/consent-text.md`** (kk/ru/en) — текст согласия для `/start`: обработка ПД + прямое упоминание трансграничной передачи (ст. 16) + право возражения против автоматизированной обработки (ст. 19-1) + ссылка на политику. Версионируется (consent_version) — при изменении текста согласие перезапрашивается.
4. **`docs/legal/dpa-hotel-template.md`** — раздел договора с отелем о поручении обработки: отель = собственник базы, платформа = оператор; перечень субоператоров; обязанность отеля информировать гостей при заселении; распределение ответственности при инциденте.
5. **`docs/runbooks/breach-response.md`** — регламент утечки: критерии инцидента → уведомление КИБ МИИЦР в 1 рабочий день (шаблон текста: контакты ответственного, принятые меры, затронутые субъекты — по приказу № 232/НҚ) → фиксация таймлайна → ротация секретов. Сокрытие = отдельный штраф, поэтому регламент «когда сомневаешься — уведомляй».
6. **`docs/runbooks/subject-requests.md`** — обращения гостя: доступ / исправление / блокирование / удаление / возражение (ст. 19-1 — ответ за 3 рабочих дня). Фаза 0: ручная SQL-процедура по conversation_id (удаление messages + анонимизация service_requests). Фаза 1: staff-инструмент.
7. **Вне репо: приказ о назначении ответственного лица** за организацию обработки ПД (основатель) — оформляется вместе с ИП/ТОО; шаблон — в `docs/legal/`.

## П2. Код

### Новое (комплаенс-фичи)

1. **Consent-gate в `/start`** (совместить с OP-30 «приветствие»):
   - Вариант A (рекомендую): inline-кнопка «Согласен / Келісемін / I agree» до первого диалога; факт — `conversations.consent_at` + `consent_version`; без согласия LLM не вызывается (заодно экономит токены на ботах-сканерах).
   - Вариант B: implied consent — «продолжая диалог, вы соглашаетесь…» одной строкой в приветствии. Дешевле по UX-трению, слабее как доказательство.
   - В обоих: язык текста — по языку гостя (механизм guest_language уже есть).
2. **Маскирование платёжных паттернов** во входящем тексте (regex 13–19 цифр + Luhn-чек) в normalize-пайплайне канала — до записи в `messages.text` и до LLM. Замена на `[card ****1234]`. Закрывает OP-24/L-6 юраудита.
3. **Ретеншн `messages`/`conversations`**: периодическая зачистка в воркере — канон уже есть (`OUTBOX_RETENTION_DAYS`), скопировать паттерн: `MESSAGES_RETENTION_DAYS` (варианты: 30 — минимум для разбора инцидентов; **90 — рекомендую**; 180 — максимум разумного). Для `service_requests`: не удалять (операционная история отеля), а **анонимизировать** свободный текст (`details`, `resolution_note`) спустя тот же срок; агрегаты/статусы остаются для аналитики.
4. **Лимит длины гостевого текста** (L7) — вместе с rate-limit #41.

### Хардненинг (из аудита 21.07, юридически = «принятие мер»)

5. **#41 rate-limit + #46 атомарный бюджет** — волна 1 аудита, без изменений.
6. **#49**: non-superuser LOGIN-роль (M1) + убрать дефолт `dev-service-token` (M3).
7. **Закрыть публичную поверхность**: `docs_url=None`/`openapi_url=None` вне dev (L4), `/health/ready` за auth или облегчить (M8), убрать `ufw allow 8000` (L13).
8. **H1 (ручные шаги)**: унести `.env` из чекаута, ротировать ANTHROPIC_API_KEY и токен бота.

## П3. Инфраструктура

1. **ADR-006 «Хостинг и резидентность РК»** — зафиксировать: продакшн целиком в РК; staging остаётся на Hetzner с синтетическими данными; LLM-вызовы наружу под согласием; бэкапы у второго независимого провайдера; шифрованный зарубежный бэкап допустим (шифроблоб без ключа ≠ доступные ПД).
   - Провайдер (решение основателя): **Kazteleport** (ЦОД Halyk, Tier III, SLA 99.9% — рекомендую как якорь надёжности) / Servercore Алматы / VK Cloud KZ. PS.kz — не для продакшн-БД.
2. **Task 0019 (бэкапы)** — теперь юридическое требование, не только опс: pg_dump по cron → шифрование (age) → S3-совместимое хранилище второго KZ-провайдера; `docs/runbooks/restore.md` + одна живая репетиция с таймингом.
3. **Шифрование диска** прод-VPS (LUKS или средствами провайдера) — прагматичное закрытие криптотребования СТ РК для хранения.
4. **Внешний uptime-мониторинг** `/health` (UptimeRobot/healthchecks.io, бесплатный тир) → алерт в Telegram.

## П4. Организационное (вне репо)

1. **ИП или ТОО до первых денег с отеля.** ИП (упрощёнка) — дёшево и быстро, ок для платного пилота; ТОО — нужно для Astana Hub (0% КПН/НДС) и солиднее для сетевых отелей. Рекомендация: ИП сейчас, ТОО + Astana Hub при выходе за 2–3 отеля.
2. **Приказ об ответственном лице** (п. П1.7).
3. **Договор с пилотным отелем** с DPA-разделом (П1.4); обязанность отеля — информировать гостей при заселении (табличка/анкета).
4. **К 25.08.2026**: проверить пороги уведомительного реестра операторов (закон № 231-VIII); подписаться на обновления подзаконки МИИЦР.

## Чего НЕ делать (избыточно для Фазы 0/пилота)

- Биометрическая аутентификация пользователей — обязательна только для баз > 100 тыс. записей (следить при росте: ~100 заселений/день × диалоги — горизонт годы).
- Интеграция с гос. сервисом контроля доступа (ст. 8-1) — только при взаимодействии с объектами информатизации госорганов; у нас их нет.
- Сертифицированные СЗИ, ISO 27001, штатный DPO, DLP-системы — несоразмерно масштабу; закон этого от малого оператора не требует.
- Горячий standby-сервер — см. раздел отказоустойчивости юраудита.

## Порядок и упаковка в задачи

Блокеры пилота (порядок): П2.8 (секреты, руками, сегодня) → ADR-006 → П1 (доки, одна задача) → П2.1–2.3 (consent+маскирование+ретеншн, 2 PR) → Task 0019 + переезд прод-контура → П2.5–2.7 (хардненинг — уже волны 1–3 аудита 21.07). П4 — параллельно, календарно к пилоту.

Открытые решения основателя: (1) провайдер прод-хостинга; (2) вариант consent-gate A/B; (3) срок ретеншна 30/90/180; (4) ИП vs ТОО. Рекомендации — в тексте выше.
