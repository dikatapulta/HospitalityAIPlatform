1. **Зависшие docs-PR**: решения уже действуют в коде main, а их фиксация (ADR-012, план с веб-чатом в Phase 1, наблюдаемость 0018/0019) живёт только в открытых PR #67, #84, #51, #52. Для сессии, читающей main, эти решения «не приняты».

## Расхождения по убыванию опасности

### H-1. deploy.md: финальная проверка деплоя всегда падает — и толкает ломать security-модель

- **Док:** `docs/runbooks/deploy.md:135-136` (шаг A9 «Проверить») — `curl http://<IP>:8000/health/live` / `/health/ready`. Туда же `deploy.md:52` и `ops/deploy/bootstrap-server.sh` — `ufw allow 8000/tcp`.
- **Код:** `ops/deploy/docker-compose.staging.yml` — у `app` только `expose: "8000"`, наружу не опубликован ни один порт (шапка файла это прямо декларирует); единственный вход — `https://staging.necturn.com` через cloudflared.
- **Что врёт:** задокументированный способ убедиться, что деплой жив, гарантированно даёт connection refused на здоровом стеке.
- **Чем грозит:** основатель или LLM-сессия сочтёт деплой сломанным и начнёт «чинить» — откатывать, править firewall, публиковать порт 8000 наружу, что сломает модель «вход только через туннель». Смежно: security-аудит L13 (устаревший `ufw allow` в bootstrap) — здесь шире: сам приёмочный шаг битый.

### H-2. errors.md: рецепты восстановления LLM-лимитов на staging — плацебо

- **Док:** `docs/runbooks/errors.md:269` (ERR-AI-001 → «поднять `LLM_TIMEOUT_SECONDS`»), `errors.md:282` (ERR-AI-002 → «поднять `LLM_TENANT_DAILY_BUDGET_USD`»).
- **Код:** `ops/deploy/docker-compose.staging.yml:55-61,120-126` — в контейнеры `app`/`worker` пробрасываются только `ANTHROPIC_API_KEY` и `LLM_MODEL`; `LLM_TIMEOUT_SECONDS`, `LLM_MAX_ATTEMPTS`, `LLM_TENANT_DAILY_BUDGET_USD`, `WORKER_*` — нет. `.env` в образ не попадает (`.dockerignore`), так что `Settings` в контейнере видит только env контейнера.
- **Что врёт:** правка `/opt/hospitality/.env` + передеплой ничего не меняют — лимиты остаются дефолтными ($5, 30 с).
- **Чем грозит:** в день инцидента «бюджет исчерпан» консьерж всего отеля лежит до полуночи UTC, а задокументированный фикс молча не работает. Это единственная находка, где чинить надо **код** (compose), а не док.

### H-3. `.env.example` рекомендует Haiku — отменённое решение с устаревшим обоснованием

- **Док:** `.env.example:24-26` — «Haiku 4.5 — выбор bake-off'а … равное качество с Sonnet 5, втрое дешевле. `LLM_MODEL=claude-haiku-4-5`».
- **Код:** `src/hospitality/shared/config.py:64-72` — дефолт `claude-sonnet-5` с обоснованием (живой прогон при фиксе #71: на казахском Haiku лезет в русский); `ops/deploy/docker-compose.staging.yml:61,126` — тот же дефолт.
- **Что врёт:** канонический шаблон env утверждает отменённое решение как действующее, с аргументом «дешевле», который экономика пилота делает соблазнительным.
- **Чем грозит:** любой, кто скопирует `.env.example` (локальный запуск, evals, новый стенд), явно переопределит правильный дефолт и вернёт модель с известным P1-дефектом казахского языка. Усугубляется H-4: обоснование выбора Sonnet в main недоступно.
