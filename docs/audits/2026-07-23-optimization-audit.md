# Оптимизационный аудит — 23.07.2026

**Состояние кода:** main @ `4f5f525`. Код не менялся; писался только этот файл.
**Скоуп:** мёртвый код, дублирование, неэффективные запросы к БД, избыточные слои, зависимости.
Токен-экономика LLM и качество AI-ответов — параллельные окна аудита, здесь не разбираются.
**Метод:** vulture (min-confidence 60/70) + ручная верификация каждой находки по коду,
pip-audit по freeze текущего окружения, сверка запросов с индексами миграций,
трассировка вызовов grep'ом. Находки аудитов 21–22.07 (`2026-07-21-security-audit.md`,
`docs/audit/OPERATIONAL_AUDIT.md`, `2026-07-22-legal-audit-rk.md`) не дублируются —
смежные помечены ссылкой.

## Вердикт

**Кодовая база для Phase 0 на удивление чистая.** Мёртвого кода фактически нет (vulture
по не-тестовому коду даёт только ложные срабатывания: FastAPI-роуты, Pydantic-валидаторы,
TypeDecorator-методы), неиспользуемых зависимостей нет, pip-audit — ноль уязвимостей,
абстракции с одним потребителем (gateway-слоистость, registry на один инструмент,
`channels/base.py`) — осознанные и документированные контракты роста, а не оверинжиниринг.
Разделение `platform/` vs `shared/` — не дубликат, а зафиксированное направление слоёв.

Реальный оптимизационный долг один: **индексы не соответствуют фактическим путям
запросов** горячих таблиц (`messages`, `outbox_events`, `service_requests`,
`llm_call_log`). Сегодня это невидимо (десятки строк), но `messages` и `llm_call_log`
не имеют retention — деградация гарантирована самим ростом. Плюс горсть лишних
roundtrip'ов, чинящихся построчно. Всё лечится одной миграцией и одним мелким PR.

## Находки (по убыванию ценности)

### O-1. Обратный поиск по `messages.external_message_id` — скан всей таблицы на каждое действие персонала

`channels/telegram/store.py:217-235` (`load_request_id_for_staff_message`):
`WHERE external_message_id = ? AND direction='outbound' AND idempotency_key LIKE 'staff:%' ORDER BY created_at DESC LIMIT 1`.
Индексы `messages` (`channels/telegram/models.py:121,124` + миграция 0008): только
`tenant_id`, `conversation_id`, `uq(tenant_id, idempotency_key)`. Колонка
`external_message_id` (`models.py:134`) не индексирована — под RLS запрос сканирует все
сообщения тенанта. Вызывается на **каждую** reply-команду персонала, каждый ответ на
note-prompt и каждый reply-резолв (`staff.py:341-361`). Retention у `messages` нет
(OP-24) — рост неограничен.

- **Выигрыш:** O(N) → O(log N); через месяцы пилота — с сотен мс до <1 мс на каждое действие персонала.
- **Фикс:** индекс `(tenant_id, external_message_id)`, можно partial `WHERE direction = 'outbound'`.
- **Риск:** низкий — только миграция.

Туда же в миграцию: составной `(conversation_id, created_at)` под
`load_dialog_history` (`store.py:153-180`, top-20 сейчас сортирует все сообщения
диалога через `ix_messages_conversation_id`).

### O-2. Outbox: исчерпанные события навсегда остаются в горячей выборке поллера

`shared/events.py:199-218`: воркер раз в секунду выбирает
`WHERE processed_at IS NULL AND attempts < 10 AND ... ORDER BY occurred_at LIMIT 50`.
Две беды:

1. `cleanup_processed_events` (`events.py:289-317`) удаляет только
   `processed_at IS NOT NULL`. Событие с `attempts >= 10` (ERR-EVENTS-002) остаётся
   с `processed_at NULL` **навсегда** и попадает в скан каждого полла, отфильтровываясь
   по `attempts` уже после чтения. Каждое «отравленное» событие — вечный налог на
   ежесекундный цикл.
2. Под сам полл нет подходящего индекса: `ix_outbox_events_processed_at` покрывает
   `IS NULL`, но сортировка по `occurred_at` — отдельный sort поверх выборки
   (в миграциях 0003/0004 отложено осознанно).

- **Выигрыш:** полл с ростом мусора остаётся O(batch), а не O(мусор); минус вечный sort.
- **Фикс:** partial-индекс `(occurred_at) WHERE processed_at IS NULL` + решение о судьбе
  исчерпанных строк. Судьба строк — это ровно dead-letter из sec-аудита **H3**: решать
  одной задачей, не двумя.
- **Риск:** индекс — низкий; retention исчерпанных — нужно продуктовое решение (терять ли факт).

Смежное: сетевой I/O внутри FOR UPDATE-транзакции пачки — уже найдено как **sec H2**,
здесь не дублируется; фикс H2 автоматически снимет и «минутные» транзакции воркера.

### O-3. Redis — целый инфраструктурный компонент с единственным потребителем: собственным health-чеком

Единственный импорт `redis` во всём src — `shared/health.py:13`; единственное
использование — проверка «жив ли Redis» в `/health/ready`. Функционально Redis не
используется нигде: ни кэша, ни очередей, ни rate-limit. При этом он поднимается в
`ops/docker-compose.yml` и `ops/deploy/docker-compose.staging.yml`, тянет зависимость
`redis>=5.0` и — главное — **его падение валит readiness приложения, которому он не нужен**.

- **Выигрыш:** минус контейнер на staging (память/поверхность), минус ложный сигнал
  readiness; либо честная готовность к первому реальному использованию (#41 rate-limit —
  первый кандидат).
- **Фикс (вилка):** (а) если rate-limit #41 будет на Redis — оставить всё как есть,
  находка закрывается сама; (б) если #41 сделают на Postgres — убрать Redis из compose
  и readiness до первого реального потребителя (ADR-002 позволяет вернуть, когда
  понадобится). Решить при спеке #41, не раньше.
- **Риск:** низкий; смежно с sec **M8** (health-чек как DoS-рычаг) — чинить одним заходом.

### O-4. `find_open_requests_by_daily_number` — уникальный индекс не работает из-за порядка колонок

`modules/requests/service.py:198-207`: `WHERE daily_number = ? AND status IN (new, in_progress)`.
Единственный индекс с `daily_number` — `uq(tenant_id, service_day, daily_number)`
(миграция `0010:40-44`), но `service_day` в запросе отсутствует и стоит в индексе
**вторым** — используется лишь префикс `tenant_id`: скан всех заявок тенанта на каждый
`/done N` / `/take N`.

- **Выигрыш:** скан → точечный lookup.
- **Фикс:** не новый индекс, а фикс sec **L3**/#83 (ограничить поиск текущим служебным
  днём): добавление `service_day = today` в WHERE автоматически включает существующий
  уникальный индекс. Две находки закрываются одной строкой.
- **Риск:** низкий; меняет семантику `/done N` для вчерашних заявок — ровно так, как
  того уже требует L3.

### O-5. `_resolve_tenant` — SELECT тенанта на каждый вебхук при статичном slug

`channels/telegram/service.py:100-109`: slug приходит из env (`TELEGRAM_TENANT_SLUG`),
неизменен всё время жизни процесса, но резолвится отдельной платформенной транзакцией
на **каждое** входящее сообщение.

- **Выигрыш:** −1 транзакция (checkout + BEGIN + SELECT + COMMIT) на каждый вебхук.
- **Фикс:** кэш `slug → tenant_id` в памяти процесса (простой module-level dict;
  инвалидация не нужна — id по slug не меняется).
- **Риск:** минимальный. Учесть, что в Phase 1 маппинг бот→тенант уедет в конфиг
  (план в `service.py:16`) — кэш не должен помешать; словарь per-slug не помешает.

### O-6. `_category_name` грузит все категории тенанта ради одной

`channels/telegram/notifications.py:290-295`: `list_categories()` + поиск по id в
Python на каждое staff-уведомление. Нужен `get_category(id)` в API модуля requests.

- **Выигрыш:** копеечный сейчас (категорий 3), но это единственный настоящий
  «выбрать всё и отфильтровать в Python» в коде — убрать, пока не скопировали как канон.
- **Риск:** минимальный (+1 функция в `modules/requests/api.py`).

### O-7. `set_pending_action` — SELECT + UPDATE вместо одного UPDATE

`channels/telegram/store.py:136-143`: `session.get(Conversation)` + ORM-мутация = два
statement'а на каждый ход гостя с гейтом. Одиночный
`UPDATE conversations SET pending_action = ... WHERE id = ...` — один.

- **Выигрыш:** −1 запрос на ход гостя. **Внимание:** issue #91 (гонка параллельных
  сообщений) собирается вешать lock на Conversation — фикс делать внутри #91, а не
  отдельно, иначе дважды перепишете одно место.
- **Риск:** низкий.

### O-8. Бюджетный гейт — SUM по журналу за день перед каждым LLM-вызовом

`ai/gateway/service.py:207`: `SELECT coalesce(sum(cost_usd), 0) WHERE created_at >= day_start`
на каждый вызов — O(вызовов за день) на вызов, квадратично за сутки. Индексы
`llm_call_log` — `tenant_id` и `created_at` раздельно (0007), составного нет; retention
журнала нет. На пилотных объёмах — единицы мс, но сам гейт уже признан негодным
(sec **C1**/#46, TOCTOU): при переделке на атомарную квоту (резервирование до вызова)
SUM исчезнет сам. Отдельно не чинить; в миграцию из O-1 добавить составной
`(tenant_id, created_at)` — он пригодится и квоте, и будущим отчётам.

### O-9. `staff.py` — 426 строк, превышен лимит «файл ≤ ~400» (R-правило)

`channels/telegram/staff.py` — 426 строк, единственный файл src сверх лимита.
Естественный шов: `_resolve_target` / `_resolve_request` / `_resolve_by_daily_number` /
`_is_uuid` (резолв цели команды, ~70 строк) → отдельный модуль.

- **Выигрыш:** соответствие канону; файл перестанет расти в задаче #83 (кнопки выбора при неоднозначности).
- **Риск:** нулевой (перенос без изменений); делать попутно первым же PR, трогающим staff.py, — отдельного PR не заслуживает.

### O-10. Зависимости: чисто; три мелких заметки

- **pip-audit** (по freeze из venv, 60 пакетов): **уязвимостей нет**.
- **Неиспользуемых объявленных зависимостей нет:** все 11 из `pyproject.toml`
  импортируются (uvicorn — рантайм-сервер, tzdata — данные zoneinfo). Redis — см. O-3:
  используется, но лишь health-чеком.
- **Устаревание минорное:** anthropic 0.116→0.117, fastapi 0.139.0→0.139.2, mypy
  2.1→2.3 и т.п. — рутинные бампы, ничего критичного; отдельной задачи не нужно.
- **Гигиена окружения (не репозитория):** в локальном venv стоит `sentry-sdk`, не
  объявленный в pyproject (артефакт работы над PR #51 в общем venv двух сессий);
  локальный venv — Python 3.13.5 при 3.12 в Dockerfile и CI. Оба расхождения безвредны,
  но «работает локально — падает в CI» при следующем мажорном расхождении начнётся
  отсюда. Лечится `make venv` с зафиксированной версией — по желанию.

### O-11. Известный налог, чинить НЕ надо (зафиксировать и жить): транзакция-на-функцию

Канон проекта — каждая store-функция открывает свой `session_scope()` (checkout + BEGIN +
`SET LOCAL app.tenant_id` + COMMIT). Один ход гостя = ~10–12 отдельных транзакций
(~30–40 сетевых roundtrip'ов, ~10–30 мс при локальной БД) на фоне 2–5 с LLM-вызова.
Это цена читаемости для LLM-мейнтейнеров — осознанная и правильная в Phase 0.
Находка фиксируется, чтобы будущий аудитор не «оптимизировал» канон: объединять
транзакции стоит только когда профилировщик покажет, что это заметно, — не раньше.

## Чего НЕ делать (упрощение навредит)

1. **Не сливать `platform/` и `shared/`.** Похожие имена файлов (`config.py`, `events.py`,
   `seed.py`) — разные вещи: env-настройки vs конфиг тенанта, канонический пример
   событий модуля vs outbox-механизм. Направление `platform → shared` зафиксировано
   import-linter'ом (контракт 1).
2. **Не удалять «неиспользуемые» значения контрактов:** `ConfirmationClass.AUTO`/`CONFIRM_STAFF`
   (`ai/tools/base.py:20-22`) — это контракт P-9, а не мёртвый код; `ReplyTo.text: None`
   и `MessageKind`-структура (`channels/base.py`) — зарезервированы под WhatsApp-адаптер
   (задокументировано в DISCUSSION_LOG).
3. **Не схлопывать `ai/gateway/` (8 файлов, 724 строки).** Каждый файл — один слой
   (порт/провайдер/mock/журнал/схемы); mock-провайдер — обязательство ADR-007 (CI без
   ключей). Слияние сэкономит ~50 строк импортов и сломает канон.
4. **Не убирать `registry.py` при одном инструменте** — это точка роста, на которую
   ссылается канон «новый инструмент = запись здесь + модуль-обёртка».
5. **Не «оптимизировать» `ensure_conversation`** (`store.py:33-63`): SELECT-first
   оптимален для повторных сообщений (подавляющий случай); INSERT-first ухудшит happy-path.
6. **Не трогать retry-цикл дневного номера** (`modules/requests/service.py:111-155`):
   `max+1` + уникальный индекс + retry — осознанный race-safe дизайн, подтверждён
   sec-аудитом («не трогать»).
7. **Не дедуплицировать conftest'ы модулей** — реимпорт фикстур из `tests/conftest.py`
   и есть канонический приём (файлы по 36–54 строки, докстринги это фиксируют).
8. **Не удалять `tools/publish_demo_event.py` и `ai/evals/bakeoff.py`** — ручные
   CLI/офлайн-инструменты, на которые ссылаются runbooks и coverage-omit соответственно.
9. O-11 — не объединять транзакции store-слоя.
