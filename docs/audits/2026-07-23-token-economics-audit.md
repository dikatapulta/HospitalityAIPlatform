- Стоимость типового диалога «заявка с подтверждением» (Sonnet 5, без кэша):
  **~$0.018–0.02**; длинный диалог с уточнениями — до ~$0.04.
- До 31.08.2026 действует интро-цена Sonnet 5 ($2/$10 вместо $3/$15) — фактический
  счёт Anthropic на ~33% ниже, чем `cost_usd` в `llm_call_log` (в прайс-листе
  `service.py` намеренно стандартная цена, чтобы COGS не занижался — это правильно,
  но в сверке счёта надо помнить).

Три правки (ниже) снижают стоимость диалога ещё в ~2.5–3 раза, до ~$0.006–0.008,
без потери качества.

## Анатомия стоимости диалога

Один ход гостя = один полноразмерный вызов, где input доминирует (~85–95% цены):

- **Фиксированный префикс ~1.2–1.6k токенов на каждом ходу**: system-промпт
  `concierge_v3` (~2.7 КБ ≈ 650 ток.) + JSON-схема инструмента
  `create_service_request` (развёрнутые описания полей ≈ 600–800 ток.). У хода
  подтверждения — свой префикс той же величины (`confirmation_gate_v1` +
  pending_action + схема `resolve_confirmation`).
- **История до 20 сообщений** (`MAX_HISTORY_MESSAGES`, фикс #74) — перечитывается
  полным тарифом на каждом ходу; к концу диалога input растёт до ~4k токенов.
- Output мал (24–260 ток.): `max_tokens=1024` адекватен, на цену не влияет
  (оплата по факту).

Полный сценарий «заявка от иностранца»: ход-заявка (~$0.008) + ход-подтверждение
(~$0.008) + перевод сути для персонала (~$0.0006) + 2 перевода статусов гостю
(~$0.0012) ≈ **$0.018**. Локальному гостю (ru) переводы гостю не нужны — ~$0.017.

## Находки-экономии (по убыванию эффекта)

### 1. Prompt caching не используется вовсе — экономия ~50–65% стоимости диалога

`anthropic_provider.py` не ставит ни одного `cache_control`. Каждый ход платит
полный input-тариф за неизменный префикс (system + tools) и за всю историю,
которая уже обсчитывалась на прошлом ходу. Кэш-чтение стоит ~0.1× input-цены
(запись 1.25×) — для многоходового диалога это сокращение input-части на
~70–85%, т.е. **~50–65% полной стоимости диалога**.

- **$ в месяц:** при среднем сценарии ~$15–25/мес; при потолке — до ~$80/мес.
  Эффект растёт линейно с трафиком — к продакшену это главная правка.
- **Риск: нулевой для качества** (кэш — механика API, ответы не меняются).
  Технические нюансы: (а) брейкпоинт ставить по multi-turn паттерну — на
  последний блок сообщений, тогда кэшируется префикс+история целиком; один
  только префикс (~1.2–1.6k ток.) может быть ниже минимума кэшируемости и
  молча не закэшироваться; (б) `_compute_cost` сейчас считает только
  `input_tokens`/`output_tokens` — при включении кэша обязательно расширить
  прайс-лист и `LlmCallLog` полями `cache_read/creation_input_tokens`, иначе
  стоимость будет молча занижаться (нарушение §7.2); (в) у диалога и
  классификатора разные system → это два разных кэш-префикса, кэшировать оба;
  (г) проверять включение по `usage.cache_read_input_tokens > 0`.

### 2. Haiku 4.5 для классификатора подтверждения и переводов — экономия ~25–30%

Сейчас все 4 типа вызовов идут на Sonnet 5. Два из них — не творческий диалог:

- **Классификатор подтверждения** (`_classify_confirmation`) — структурная
  задача с forced tool (enum из трёх значений + короткая реплика). Это ~половина
  всех полноразмерных вызовов диалога. Haiku в 3 раза дешевле на input
  ($1 vs $3/MTok) → экономия ~2/3 стоимости этих вызовов.
  **Риск: средний, управляемый.** Вердикт (confirm/decline/other) на Haiku
  надёжен (форс-вызов, свободный текст невозможен; fallback `OTHER` безопасен
  по P-9). Риск только в поле `reply` на казахском (память проекта: Haiku на kk
  сползает в русский, фикс #71/#72). Митигация: прогнать 6-язычный golden-set
  (§7.7, ADR-010) именно на классификаторе; либо роутить kk-диалоги на Sonnet,
  остальные на Haiku. Потребует снять Non-Goal «нет маршрутизации» Task 0014 —
  малым ADR или строчкой в спеке.
- **`translate_for_staff`** (→ русский) — риска kk нет вообще (целевой язык
  русский), переключать безопасно сразу. **`translate_for_guest`** — Haiku для
  всех языков, кроме `kk` (для kk оставить Sonnet). В абсолютных $ переводы
  почти ничто (~$0.0006/вызов), но правка тривиальна и попутна.

- **$ в месяц:** при среднем сценарии ~$8–14/мес (почти всё — классификатор).

### 3. Урезать историю для классификатора подтверждения — экономия ~10%

`_handle_confirmation_reply` передаёт классификатору всю историю (до 20
сообщений), хотя вердикт «да/нет/другое» определяется последним обменом +
pending_action (который и так подаётся отдельно в system). 2–4 последних
сообщений достаточно; input вызова падает с ~2–2.5k до ~1–1.2k токенов.

- **$ в месяц:** ~$3–5/мес при среднем сценарии.
- **Риск: низкий.** Кейс «other» (гость сменил тему) всё равно уходит в
  `_handle_new_request` с полной историей. Проверить golden-set'ом кейсы
  отложенного подтверждения («да» после уточняющего вопроса).

### Не рекомендуется сейчас

- **Дедупликация переводов** (кэш `(text, lang)` в Redis для повторяющихся
  статусных фраз) — экономия <$1/мес при текущем трафике; отложить до Phase 1.
- **Сокращение system-промпта** — `concierge_v3` уже компактен (~650 ток.),
  каждый абзац — фикс конкретного бага (#31, #71). Резать — риск регрессии
  ради ~$2/мес; после включения кэша (правка 1) вопрос исчезает совсем.
- **Окно истории 20 → меньше для основного диалога** — не трогать: окно уже
  результат фикса #74, дальнейшее сжатие бьёт по связности диалога.
