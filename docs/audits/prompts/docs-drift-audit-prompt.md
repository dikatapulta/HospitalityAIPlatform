# Промпт: аудит дрейфа документации (параллельное окно 3/3)

Ты — аудитор с чистым контекстом. Найди все расхождения между документацией и фактическим кодом. Контекст критичен: код проекта пишут LLM-сессии, которые читают FOUNDATION/README/ADR как истину — каждое расхождение станет багом, который будущая сессия внесёт «по инструкции».

**Направление.** README каждого модуля vs его фактический `api.py` и поведение; FOUNDATION.md (особенно принципы P, правила R, разделы про наблюдаемость и данные) vs реализация; все ADR в `docs/adr/` — соответствует ли код принятым решениям, нет ли решений, реализованных иначе; PHASE0.md/PHASEN-карточки vs то, что реально в main; GLOSSARY vs термины в коде; runbooks vs фактические команды/пути (например, упоминаемые make-таргеты, которых нет). Отдельно: обещания «Phase 1» в комментариях кода — сверить, не противоречат ли текущему плану.

**Не дублируй** находки `docs/audits/` и `docs/audit/OPERATIONAL_AUDIT.md` — смежное указывай ссылкой.

**Правила изоляции (параллельно работают другие сессии):**
- Только чтение. Никаких правок (даже «мелких фиксов доков»), коммитов, `git checkout`.
- Не запускать приложение/docker/make check.
- Чужие untracked-файлы в `git status` — норма, не трогать.
- Писать ровно один файл: `docs/audits/2026-07-23-docs-drift-audit.md`.

**Формат отчёта:** вердикт (масштаб дрейфа) → расхождения по убыванию опасности (каждое: док-файл:строка vs код-файл:строка, что именно врёт, чем грозит будущей LLM-сессии) → предлагаемая упаковка фиксов (что чинить в доках, что — в коде).
