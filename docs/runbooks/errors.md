# Каталог ошибок (FOUNDATION §10.5)

Каждая **ожидаемая** ошибка платформы имеет код `ERR-<MODULE>-NNN` и статью в этом файле.
Алерт или ответ API содержит код → код содержит диагноз. Это база для «автоматического
определения вероятной причины» без чтения кода.

## Правила

- Код выбрасывается только через `AppError(code=..., message=..., status_code=...)`
  из `hospitality.shared.errors` (канон, R-8). `message` виден клиенту — без секретов
  и внутренних деталей.
- Новый код = константа в коде + статья здесь, **в том же PR**. Код без статьи — блокер ревью.
- Формат: `ERR-<MODULE>-NNN`, где `<MODULE>` — имя модуля заглавными (PLATFORM, REQUESTS, …),
  `NNN` — порядковый номер внутри модуля, начиная с 001. Номера не переиспользуются:
  удалённый код помечается здесь как «изъят», а не заменяется новым смыслом.
- Структура статьи: **что значит → вероятные причины → что проверить**.

## Как найти ошибку в логах

Каждый ответ с ошибкой содержит `correlation_id` (он же — в заголовке ответа
`X-Correlation-ID`). По нему находится весь след запроса:

```bash
# staging (подробности подключения — docs/runbooks/deploy.md)
docker compose -f /opt/hospitality/docker-compose.staging.yml logs app | grep <correlation_id>

# локально
make dev-logs | grep <correlation_id>
```

## ERR-PLATFORM-001 — необработанная внутренняя ошибка

- **Что значит:** запрос упал с исключением, которое не является `AppError`, — программная
  ошибка или отказ инфраструктуры. Клиент получил HTTP 500 без деталей; полная трасса —
  в логах, событие `unhandled_error` с тем же `correlation_id`.
  Под uvicorn на тот же инцидент в логах будет и вторая запись с той же трассой от
  `uvicorn.error` («Exception in ASGI application»): `ServerErrorMiddleware` перевыбрасывает
  исключение после отправки ответа — это дубль, а не второй инцидент. Канонический
  источник для подсчёта и будущих алертов (§10.8) — только событие `unhandled_error`.
- **Вероятные причины:** баг в свежевыкаченном коде; недоступность Postgres/Redis во время
  запроса; несовместимость данных (миграция не применилась).
- **Что проверить:**
  1. Логи по `correlation_id` (см. выше) — событие `unhandled_error` содержит трассу стека.
  2. `/health/ready` — если `postgres`/`redis` в статусе `error`, причина инфраструктурная.
  3. Не совпал ли всплеск ошибок с последним деплоем (`docker compose ps`, тег образа) —
     если да, откат по docs/runbooks/deploy.md.

## ERR-PLATFORM-002 — запрос не прошёл валидацию

- **Что значит:** тело/параметры запроса не соответствуют Pydantic-схеме эндпоинта (P-7).
  Клиент получил HTTP 422; в ответе поле `details` — список нарушений (какое поле, что не так).
  Это ошибка вызывающей стороны, не платформы.
- **Вероятные причины:** клиент (фронт, интеграция, AI-инструмент) отправил устаревший или
  неверный формат; контракт эндпоинта изменился без синхронизации с потребителями.
- **Что проверить:**
  1. `details` в ответе или лог-событие `request_validation_failed` по `correlation_id`
     (в лог пишутся только `loc`/`msg`/`type` — сырой ввод клиента может содержать PII
     и в логи не попадает; сам ввод виден в поле `details` ответа клиенту).
  2. Если ошибки массовые после релиза — сверить версию контракта у клиента и сервера
     (изменение схемы без обратной совместимости — нарушение P-7, фиксировать как инцидент).

## ERR-PLATFORM-003 — HTTP-ошибка фреймворка

- **Что значит:** запрос не дошёл до бизнес-логики — маршрутизация или сам эндпоинт
  ответили стандартным `HTTPException`: неизвестный путь (404), неверный HTTP-метод (405),
  в будущем — 401/403 авторизации. `message` — стандартный текст фреймворка
  («Not Found», «Method Not Allowed»); статус — из исключения. Это ошибка вызывающей
  стороны, не платформы.
- **Вероятные причины:** опечатка в URL у клиента; вызов эндпоинта, который ещё не
  реализован или переименован; неверный метод (GET вместо POST).
- **Что проверить:**
  1. Событие `http_request` по `correlation_id` — фактические `method`, `path`, `status_code`.
  2. Если ошибки массовые после релиза — не переименован ли путь/метод без синхронизации
     с потребителями (см. ERR-PLATFORM-002, тот же протокол).

## ERR-PLATFORM-004 — тенант не найден

- **Что значит:** операция сослалась на `tenant_id`, которого нет в реестре `tenants`
  (`load_tenant_config`/`store_tenant_config`, platform/config.py).
- **Вероятные причины:** устаревший/чужой `tenant_id` у вызывающей стороны (например,
  id из другой среды: dev-идентификатор на staging); тенант удалён оффбордингом.
- **Что проверить:**
  1. Откуда пришёл `tenant_id` — след по `correlation_id`.
  2. Есть ли тенант в реестре: `SELECT id, slug FROM tenants` (доступ — владельцем схемы,
     см. runbook deploy). Демо-тенант пересоздаётся `make seed` (dev) и деплоем (staging).

## ERR-PLATFORM-005 — конфигурация тенанта не задана

- **Что значит:** тенант существует, но `tenants.config` пуст — онбординг не завершён,
  а код уже требует конфиг (часовой пояс, язык и т.д.).
- **Вероятные причины:** тенант создан напрямую в обход онбординга (служебный
  `demo-smoke` из publish_demo_event — ожидаемо без конфига); онбординг прерван.
- **Что проверить:**
  1. Что за тенант: `SELECT slug, config IS NULL FROM tenants WHERE id = '<tenant_id>'`.
  2. Демо-тенант `demo-hotel` — дозаполнит `make seed` (dev) или деплой (staging);
     боевой тенант — завершить онбординг (мастер профиля — Phase 1).

## ERR-PLATFORM-006 — конфигурация тенанта не соответствует схеме

- **Что значит:** JSON в `tenants.config` не проходит схему `TenantConfig`
  (platform/config.py) — данные и код разошлись. Подробности нарушения — в причине
  исключения по `correlation_id`.
- **Вероятные причины:** схему изменили несовместимо без повышения `schema_version`
  и скрипта миграции конфигов (§6) — главный подозреваемый; правка колонки руками
  мимо `store_tenant_config`.
- **Что проверить:**
  1. `schema_version` в БД против `TENANT_CONFIG_SCHEMA_VERSION` в коде.
  2. Если разошлись после релиза — откатить релиз или написать недостающий скрипт
     миграции конфигов всех тенантов (дисциплина §6, как Alembic для БД).

## ERR-PLATFORM-007 — запрос не аутентифицирован

- **Что значит:** запрос к аутентифицированному эндпоинту (`/api/v1/*`) пришёл без
  валидного сервисного токена — контекст тенанта не установлен. Клиент получил
  HTTP 401 с заголовком `WWW-Authenticate: Bearer`. Отсутствующий и неверный токен
  намеренно неразличимы в ответе (не подсказываем подбирающему).
- **Вероятные причины:** нет заголовка `Authorization: Bearer <токен>`; токен клиента
  не совпадает с `SERVICE_TOKEN` окружения (ротация без синхронизации клиентов);
  `SERVICE_TOKEN_TENANT_SLUG` указывает на тенанта, которого нет в реестре
  (сид не выполнен / опечатка) — тогда токен верный, а 401 всё равно возвращается.
- **Что проверить:**
  1. Заголовок запроса: `curl -H "Authorization: Bearer $SERVICE_TOKEN" ...` —
     схема именно `Bearer`, токен без кавычек и переводов строк.
  2. Логи по `correlation_id`: событие `service_token_tenant_missing` означает
     причину №3 — проверить `SELECT slug FROM tenants` и переменную
     `SERVICE_TOKEN_TENANT_SLUG` в `.env`; тенанта досоздаст сид
     (`make seed` локально, деплой — на staging).
  3. Совпадение токена клиента с `SERVICE_TOKEN` в `.env` окружения
     (staging: `/opt/hospitality/.env`, см. runbook deploy).

## ERR-TELEGRAM-001 — неверный секрет вебхука

- **Что значит:** запрос на `/channels/telegram/webhook` пришёл без заголовка
  `X-Telegram-Bot-Api-Secret-Token` или с секретом, не совпавшим с
  `TELEGRAM_WEBHOOK_SECRET` окружения (Task 0016, §8.4). Клиент получил HTTP 403;
  тело/апдейт не разбираются, Message не создаётся. Пустой `TELEGRAM_WEBHOOK_SECRET`
  отвергает ВСЕ запросы (fail-closed, §11) — это тоже даёт этот код.
- **Вероятные причины:** вебхук зарегистрирован в Telegram (`setWebhook`) без
  `secret_token` или с другим значением, чем в `.env`; секрет ротировали на сервере,
  но не переустановили вебхук; сканер/чужой POST на публичный путь; на сервере
  `TELEGRAM_WEBHOOK_SECRET` не задан (вебхук закрыт до настройки).
- **Что проверить:**
  1. Совпадает ли `TELEGRAM_WEBHOOK_SECRET` в `.env` сервера с `secret_token`,
     переданным при регистрации вебхука (см. docs/runbooks/telegram.md).
  2. Текущая регистрация: `getWebhookInfo` у Bot API — на тот ли URL и настроен ли
     секрет вообще.
  3. Логи по `correlation_id`: событие `http_request` со `status_code=403` на пути
     `/channels/telegram/webhook` — массовый поток без валидного секрета обычно
     означает чужие POST'ы, а не сбой Telegram.

## ERR-EVENTS-001 — доставка события не удалась (будет повтор)

- **Что значит:** подписчик доменного события упал; событие осталось в `outbox_events`,
  воркер повторит доставку (at-least-once, ADR-005). Лог-событие `event_delivery_failed`
  в логах worker содержит `event_name`, `event_id`, `attempts` и трассу; `correlation_id` —
  тот же, что у публикации.
- **Вероятные причины:** баг в обработчике; временная недоступность зависимости обработчика;
  полезная нагрузка не проходит валидацию схемы события (контракт изменён без совместимости —
  нарушение §13.5).
- **Что проверить:**
  1. Логи worker по `correlation_id`/`event_id` — трасса в `event_delivery_failed`.
  2. Если ошибка повторяется от попытки к попытке — это не «мигание», ждите ERR-EVENTS-002
     и чините обработчик.

## ERR-EVENTS-002 — повторы доставки исчерпаны

- **Что значит:** событие не доставлено `worker_max_delivery_attempts` раз и больше
  не берётся в работу (строка в `outbox_events`: `processed_at IS NULL`,
  `attempts` = пределу, диагноз — в `last_error`). Эффект подписчика НЕ произошёл.
- **Вероятные причины:** устойчивый баг обработчика; «ядовитое» событие
  (нагрузка не соответствует текущей схеме).
- **Что проверить:**
  1. `last_error` строки события (доступ — владельцем схемы, см. runbook deploy).
  2. После фикса обработчика вернуть событие в очередь:
     `UPDATE outbox_events SET attempts = 0, next_attempt_at = NULL WHERE id = '<event_id>';`
     — воркер подхватит его в следующем цикле. Сбрасывать оба поля — канон
     (ADR-009, backoff между попытками): обычно `next_attempt_at` к моменту
     исчерпания уже в прошлом (последняя попытка случилась после истечения его
     окна), но если `worker_max_delivery_attempts` понижали, поле может быть в
     будущем — тогда без сброса строка не возьмётся в работу до его истечения.

## ERR-EVENTS-003 — итерация воркера упала целиком

- **Что значит:** цикл воркера не смог прочитать outbox (сбой уровня инфраструктуры,
  а не отдельного обработчика). Процесс жив: логирует `worker_iteration_failed`,
  ждёт период опроса и повторяет.
- **Вероятные причины:** Postgres недоступен; миграции не применены (нет таблицы
  `outbox_events` или роли `hospitality_app`).
- **Что проверить:**
  1. Логи worker (`docker compose ... logs worker`).
  2. `/health/ready` приложения — статус `postgres`.
  3. Применены ли миграции: `docker compose ... run --rm --no-deps app alembic upgrade head`.

## ERR-EVENTS-004 — retention-очистка outbox упала

- **Что значит:** периодический вызов `cleanup_processed_events()` (раз в
  `worker_cleanup_interval_seconds`, ADR-009) не смог удалить обработанные строки
  `outbox_events` старше `outbox_retention_days`. Цикл доставки не затронут: ошибка
  логируется, воркер продолжает доставлять события как обычно и повторит очистку
  на следующем цикле.
- **Вероятные причины:** Postgres недоступен в момент очистки (тот же класс причин,
  что у ERR-EVENTS-003); блокировка строк outbox долгой транзакцией (например,
  зависшая доставка держит `FOR UPDATE` на те же строки).
- **Что проверить:**
  1. Логи worker (`outbox_cleanup_failed`) — трасса исключения.
  2. `/health/ready` приложения — статус `postgres`.
  3. Если ошибка повторяется от цикла к циклу — таблица `outbox_events` растёт
     быстрее, чем ожидалось retention-политикой; проверить объём строк с
     `processed_at IS NOT NULL` вручную.

## ERR-REQUESTS-001 — категория заявки не найдена

- **Что значит:** `create_request` получил `category_id`, которого нет среди категорий
  ТЕКУЩЕГО тенанта. Чужая и несуществующая категории неразличимы намеренно (RLS, P-4):
  обе дают этот код и HTTP 404.
- **Вероятные причины:** вызывающая сторона (тест, AI-инструмент, будущий HTTP-клиент)
  передала устаревший/чужой идентификатор; категории тенанта ещё не засеяны
  (Task 0011 — сид демо-тенанта); вызов выполнен в контексте не того тенанта.
- **Что проверить:**
  1. Логи по `correlation_id`: `tenant_id` в контексте — тот ли тенант?
  2. Категории тенанта: `SELECT id, key FROM request_categories` из тенантной сессии
     (с Task 0013 — через API списка категорий).

## ERR-REQUESTS-002 — заявка не найдена

- **Что значит:** `change_request_status`/`get_request` не нашли заявку с данным `id`
  у текущего тенанта. Чужая и несуществующая заявки неразличимы (RLS, P-4). HTTP 404.
- **Вероятные причины:** устаревший идентификатор; вызов в контексте не того тенанта.
- **Что проверить:** те же шаги, что у ERR-REQUESTS-001, но по `service_requests`.

## ERR-REQUESTS-003 — недопустимый переход статуса заявки

- **Что значит:** запрошенный переход отсутствует в жизненном цикле
  `new → assigned → in_progress → done/cancelled` (карта — `STATUS_TRANSITIONS`
  в `modules/requests/service.py`): например `new → done`, повторное закрытие
  или любой переход из терминального статуса. Заявка не изменена, событие
  не опубликовано. HTTP 409.
- **Вероятные причины:** гонка двух операторов/процессов (заявку уже перевели);
  клиент показывает устаревший статус; ошибка логики вызывающей стороны.
- **Что проверить:**
  1. Текущий статус заявки и её историю событий `request.status_changed`
     в `outbox_events` по `request_id`.
  2. Если это гонка — повторить операцию от актуального статуса.

## ERR-REQUESTS-004 — ключ категории уже занят

- **Что значит:** `create_category` получил `key`, уже существующий у этого тенанта
  (уникальность пары `tenant_id, key`). HTTP 409.
- **Вероятные причины:** повторный вызов сида без идемпотентности; попытка завести
  категорию-дубликат вручную.
- **Что проверить:** существующие категории тенанта (`request_categories`);
  если нужна та же категория — использовать существующую строку, а не новую.

## ERR-AI-001 — LLM-провайдер не ответил (таймауты исчерпали ретраи)

- **Что значит:** AI Gateway сделал `LLM_MAX_ATTEMPTS` попыток вызова провайдера,
  каждая упала по таймауту (`LLM_TIMEOUT_SECONDS`). В `llm_call_log` осталась
  строка со статусом `timeout`. HTTP 503.
- **Вероятные причины:** деградация/инцидент у провайдера (status.anthropic.com);
  сетевые проблемы между сервером и API; слишком низкий `LLM_TIMEOUT_SECONDS`
  для длинных ответов.
- **Что проверить:**
  1. Логи по `correlation_id`: события `llm_call_timeout` (номер попытки) — все ли
     вызовы страдают или единичный.
  2. Статус-страницу провайдера; при инциденте канал обязан деградировать в
     не-AI путь (FOUNDATION §7.8) — заявки создаются без LLM.
  3. Если таймауты только на длинных запросах — поднять `LLM_TIMEOUT_SECONDS`.

## ERR-AI-002 — дневной бюджет LLM тенанта исчерпан

- **Что значит:** сумма `cost_usd` в `llm_call_log` тенанта за текущие UTC-сутки
  достигла `LLM_TENANT_DAILY_BUDGET_USD`; вызов отвергнут ДО обращения к провайдеру
  (§7.2: бюджеты и лимиты тенантов). HTTP 429 с `Retry-After` до конца UTC-суток.
- **Вероятные причины:** всплеск диалогов у тенанта; слишком низкий бюджет для
  реальной нагрузки; цикл в вызывающем коде, жгущий токены.
- **Что проверить:**
  1. `SELECT count(*), sum(cost_usd) FROM llm_call_log WHERE created_at >= date_trunc('day', now())`
     из тенантной сессии — сколько вызовов и на что ушло (по `prompt_hash` видно,
     один ли промпт крутится в цикле).
  2. Если расход легитимен — поднять `LLM_TENANT_DAILY_BUDGET_USD` (Phase 0: лимит
     общий для всех тенантов; пер-тенантный — конфиг тенанта, Phase 1).

## ERR-AI-003 — ошибка LLM-провайдера (не таймаут)

- **Что значит:** провайдер ответил ошибкой API (401/403/429/5xx, невалидный запрос,
  сеть). Не ретраится (ретрай — только по таймауту); в `llm_call_log` — строка со
  статусом `error`. HTTP 502.
- **Вероятные причины:** невалидный/отозванный `ANTHROPIC_API_KEY`; исчерпан
  rate limit аккаунта у провайдера; инцидент провайдера.
- **Что проверить:**
  1. Логи по `correlation_id`: событие `llm_call_failed` содержит текст ошибки SDK.
  2. Ключ и лимиты в консоли провайдера (docs/runbooks/secrets.md — ротация).

## ERR-AI-004 — вызов инструмента не соответствует контракту

- **Что значит:** модель попросила вызвать AI-инструмент (Task 0015, §7.3) с
  аргументами вне его контракта — неизвестный `category_key` (не из enum,
  выданного модели), невалидные поля или несуществующий инструмент. HTTP 422.
  Оркестратор НЕ отдаёт эту ошибку гостю: он превращает её в эскалацию к
  сотруднику (`needs_human`), заявка при этом не создаётся.
- **Вероятные причины:** у тенанта не настроены категории заявок (пустой enum);
  рассинхрон: категорию удалили между сборкой инструмента и исполнением; модель
  отклонилась от enum (низкоресурсный язык, слабая модель — риск, который ловит
  golden-set §7.7).
- **Что проверить:**
  1. Логи по `correlation_id`: события `tool_execution_failed` / `unknown_tool_call`
     (имя инструмента и код).
  2. Есть ли у тенанта категории заявок (`list_categories`); при пустом наборе —
     завести категории (сид демо-тенанта / кабинет персонала Phase 1).
  3. Если код повторяется на конкретном языке — это сигнал для ревизии выбора
     модели/промпта по данным bake-off'а (spec 0015).

## ERR-OPS-001 — /health/ready недоступен (алерт)

- **Что значит:** алертер (`hospitality.tools.alerter`, Task 0018) несколько опросов
  подряд (порог `ALERT_READY_FAILURE_THRESHOLD`) не получил 200 от `/health/ready`:
  либо недоступна зависимость (Postgres/Redis — какие именно, видно в `checks` в
  тексте алерта), либо приложение не отвечает вовсе (`connection error`).
  Код выбрасывается не через `AppError` — это алерт платформенного уровня,
  correlation id у внешнего опроса нет.
- **Вероятные причины:** упал контейнер `db`/`redis`/`app`; кончилось место на
  диске; VPS перегружен; деплой дольше пары минут (после него придёт ✅).
- **Что проверить:** пошаговая диагностика — docs/runbooks/alerts.md#err-ops-001.

## ERR-OPS-002 — всплеск ошибок 5xx (алерт)

- **Что значит:** прирост суммы `http_requests_total{status="5xx"}` из `/metrics`

- **Что проверить:** пошаговая диагностика — docs/runbooks/alerts.md#err-ops-002.
