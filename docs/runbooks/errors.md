# Каталог ошибок (FOUNDATION §10.5)

Каждая **ожидаемая** ошибка платформы имеет код `ERR-<MODULE>-NNN` и статью в этом файле.
Алерт или ответ API содержит код → код содержит диагноз. Это база для «автоматического
определения вероятной причины» без чтения кода.

## Правила

- Код выбрасывается только через `AppError(code=..., message=..., status_code=...)`
  из `hospitality.shared.errors` (канон, R-8). `message` виден клиенту — без секретов
  и внутренних деталей.
- Новый код = константа в коде + статья здесь, **в том же PR**. Код без статьи — блокер ревью.
- Формат: `ERR-<MODULE>-NNN`, где `<MODULE>` — имя модуля заглавными (PLATFORM, REQUESTS, …),
  `NNN` — порядковый номер внутри модуля, начиная с 001. Номера не переиспользуются:
  удалённый код помечается здесь как «изъят», а не заменяется новым смыслом.
- Структура статьи: **что значит → вероятные причины → что проверить**.

## Как найти ошибку в логах

Каждый ответ с ошибкой содержит `correlation_id` (он же — в заголовке ответа
`X-Correlation-ID`). По нему находится весь след запроса:

```bash
# staging (подробности подключения — docs/runbooks/deploy.md)
docker compose -f /opt/hospitality/docker-compose.staging.yml logs app | grep <correlation_id>

# локально
make dev-logs | grep <correlation_id>
```

---

## ERR-PLATFORM-001 — необработанная внутренняя ошибка

- **Что значит:** запрос упал с исключением, которое не является `AppError`, — программная
  ошибка или отказ инфраструктуры. Клиент получил HTTP 500 без деталей; полная трасса —
  в логах, событие `unhandled_error` с тем же `correlation_id`.
  Под uvicorn на тот же инцидент в логах будет и вторая запись с той же трассой от
  `uvicorn.error` («Exception in ASGI application»): `ServerErrorMiddleware` перевыбрасывает
  исключение после отправки ответа — это дубль, а не второй инцидент. Канонический
  источник для подсчёта и будущих алертов (§10.8) — только событие `unhandled_error`.
- **Вероятные причины:** баг в свежевыкаченном коде; недоступность Postgres/Redis во время
  запроса; несовместимость данных (миграция не применилась).
- **Что проверить:**
  1. Логи по `correlation_id` (см. выше) — событие `unhandled_error` содержит трассу стека.
  2. `/health/ready` — если `postgres`/`redis` в статусе `error`, причина инфраструктурная.
  3. Не совпал ли всплеск ошибок с последним деплоем (`docker compose ps`, тег образа) —
     если да, откат по docs/runbooks/deploy.md.

## ERR-PLATFORM-002 — запрос не прошёл валидацию

- **Что значит:** тело/параметры запроса не соответствуют Pydantic-схеме эндпоинта (P-7).
  Клиент получил HTTP 422; в ответе поле `details` — список нарушений (какое поле, что не так).
  Это ошибка вызывающей стороны, не платформы.
- **Вероятные причины:** клиент (фронт, интеграция, AI-инструмент) отправил устаревший или
  неверный формат; контракт эндпоинта изменился без синхронизации с потребителями.
- **Что проверить:**
  1. `details` в ответе или лог-событие `request_validation_failed` по `correlation_id`
     (в лог пишутся только `loc`/`msg`/`type` — сырой ввод клиента может содержать PII
     и в логи не попадает; сам ввод виден в поле `details` ответа клиенту).
  2. Если ошибки массовые после релиза — сверить версию контракта у клиента и сервера
     (изменение схемы без обратной совместимости — нарушение P-7, фиксировать как инцидент).

## ERR-PLATFORM-003 — HTTP-ошибка фреймворка

- **Что значит:** запрос не дошёл до бизнес-логики — маршрутизация или сам эндпоинт
  ответили стандартным `HTTPException`: неизвестный путь (404), неверный HTTP-метод (405),
  в будущем — 401/403 авторизации. `message` — стандартный текст фреймворка
  («Not Found», «Method Not Allowed»); статус — из исключения. Это ошибка вызывающей
  стороны, не платформы.
- **Вероятные причины:** опечатка в URL у клиента; вызов эндпоинта, который ещё не
  реализован или переименован; неверный метод (GET вместо POST).
- **Что проверить:**
  1. Событие `http_request` по `correlation_id` — фактические `method`, `path`, `status_code`.
  2. Если ошибки массовые после релиза — не переименован ли путь/метод без синхронизации
     с потребителями (см. ERR-PLATFORM-002, тот же протокол).

## ERR-EVENTS-001 — доставка события не удалась (будет повтор)

- **Что значит:** подписчик доменного события упал; событие осталось в `outbox_events`,
  воркер повторит доставку (at-least-once, ADR-005). Лог-событие `event_delivery_failed`
  в логах worker содержит `event_name`, `event_id`, `attempts` и трассу; `correlation_id` —
  тот же, что у публикации.
- **Вероятные причины:** баг в обработчике; временная недоступность зависимости обработчика;
  полезная нагрузка не проходит валидацию схемы события (контракт изменён без совместимости —
  нарушение §13.5).
- **Что проверить:**
  1. Логи worker по `correlation_id`/`event_id` — трасса в `event_delivery_failed`.
  2. Если ошибка повторяется от попытки к попытке — это не «мигание», ждите ERR-EVENTS-002
     и чините обработчик.

## ERR-EVENTS-002 — повторы доставки исчерпаны

- **Что значит:** событие не доставлено `worker_max_delivery_attempts` раз и больше
  не берётся в работу (строка в `outbox_events`: `processed_at IS NULL`,
  `attempts` = пределу, диагноз — в `last_error`). Эффект подписчика НЕ произошёл.
- **Вероятные причины:** устойчивый баг обработчика; «ядовитое» событие
  (нагрузка не соответствует текущей схеме).
- **Что проверить:**
  1. `last_error` строки события (доступ — владельцем схемы, см. runbook deploy).
  2. После фикса обработчика вернуть событие в очередь:
     `UPDATE outbox_events SET attempts = 0, next_attempt_at = NULL WHERE id = '<event_id>';`
     — воркер подхватит его в следующем цикле. Обязательно сбросить и
     `next_attempt_at` (ADR-009, backoff между попытками): если оставить старое
     значение, строка не возьмётся в работу, пока не истечёт backoff-окно
     последней (неудачной) попытки.

## ERR-EVENTS-003 — итерация воркера упала целиком

- **Что значит:** цикл воркера не смог прочитать outbox (сбой уровня инфраструктуры,
  а не отдельного обработчика). Процесс жив: логирует `worker_iteration_failed`,
  ждёт период опроса и повторяет.
- **Вероятные причины:** Postgres недоступен; миграции не применены (нет таблицы
  `outbox_events` или роли `hospitality_app`).
- **Что проверить:**
  1. Логи worker (`docker compose ... logs worker`).
  2. `/health/ready` приложения — статус `postgres`.
  3. Применены ли миграции: `docker compose ... run --rm --no-deps app alembic upgrade head`.

## ERR-EVENTS-004 — retention-очистка outbox упала

- **Что значит:** периодический вызов `cleanup_processed_events()` (раз в
  `worker_cleanup_interval_seconds`, ADR-009) не смог удалить обработанные строки
  `outbox_events` старше `outbox_retention_days`. Цикл доставки не затронут: ошибка
  логируется, воркер продолжает доставлять события как обычно и повторит очистку
  на следующем цикле.
- **Вероятные причины:** Postgres недоступен в момент очистки (тот же класс причин,
  что у ERR-EVENTS-003); блокировка строк outbox долгой транзакцией (например,
  зависшая доставка держит `FOR UPDATE` на те же строки).
- **Что проверить:**
  1. Логи worker (`outbox_cleanup_failed`) — трасса исключения.
  2. `/health/ready` приложения — статус `postgres`.
  3. Если ошибка повторяется от цикла к циклу — таблица `outbox_events` растёт
     быстрее, чем ожидалось retention-политикой; проверить объём строк с
     `processed_at IS NOT NULL` вручную.
