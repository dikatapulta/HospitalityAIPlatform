# Ревью PR вторым GitHub-аккаунтом

> FOUNDATION §13.1: «ревью — отдельной LLM-сессией с чистым контекстом». Этот
> документ — только механика *как* независимая сессия оставляет реальный,
> зачитываемый GitHub'ом approve, а не просто текстовый вердикт в комментарии.

## Как запускать ревью-сессию

1. **Новое окно/чат**, не продолжение того, где писался код — иначе «чистый
   контекст» нарушен: сессия не должна помнить рассуждений автора, только
   читать diff, README и FOUNDATION с диска.
2. Модель — сильная (см. стратегию выбора моделей), не Sonnet: ревью в одном
   классе с канонами и ADR, цена ошибки выше, чем в рутинной копии паттерна.
3. Промпт ставит роль ревьюера явно (что читать, какие правила блокирующие) —
   см. пример в истории PR #20/PR #19.
4. Сессия сама вызывает `gh pr review` под вторым аккаунтом (ниже) — вердикт
   пишется прямо в теле review, а не в отдельный комментарий.

## Почему нужен второй аккаунт

GitHub не даёт аккаунту формально approve-ить свой же PR. Все PR в этом
репозитории открываются от `dikatapulta`, поэтому реальный approve обязан
прийти от другой личности — `supertazick-oss`. Без этого «ревью отдельной
сессией» оставалось бы вежливой конвенцией, а не машинной гарантией.

## Как это работает технически

- `supertazick-oss` — коллаборатор репозитория с правами **Write**.
  Read/Triage недостаточно: GitHub засчитывает approve в
  `required_pull_request_reviews` только от аккаунтов с правом push
  (задокументированное поведение, не наша прихоть) — Write шире, чем хотелось
  бы для чисто ревьюерской роли, но это минимальный уровень, на котором approve
  вообще учитывается веткой main.
- Авторизация изолирована от основного `gh`: отдельная конфиг-директория
  `GH_CONFIG_DIR=~/.config/gh-reviewer`, токен — в системном keychain (не файл).
  Обычный `gh auth status` (аккаунт `dikatapulta`) это не трогает.
- Токен — classic PAT со скоупами `repo` + `read:org`. `gh auth login
  --with-token` требует оба скоупа для валидации; сами API-вызовы ревью
  используют куда меньше. Срок — 90 дней (таблица секретов —
  [secrets.md](secrets.md)).
- Защита `main` (`branches/main/protection`) требует
  `required_pull_request_reviews.required_approving_review_count: 1` — без
  approve `gh pr merge` физически заблокирован (`mergeStateStatus: BLOCKED`).

## Как ревьюер оставляет вердикт

```
GH_CONFIG_DIR=~/.config/gh-reviewer gh pr review <N> --approve --body-file <файл>
# или при блокирующих находках:
GH_CONFIG_DIR=~/.config/gh-reviewer gh pr review <N> --request-changes --body-file <файл>
```

Содержательный вердикт (что проверено, R-5/R-6/R-7/R-9, DoD) пишется в теле
review так же, как раньше писался в комментарий текстом — поменялся только
механизм регистрации, не содержание ревью.

## Мерж

Approve снимает блокировку `main`, но сам `gh pr merge` — сознательно ручной
шаг основателя, не автоматика LLM-сессии (необратимое действие в общей ветке).

## Ротация / компрометация токена

См. [secrets.md](secrets.md) — раздел «Локальные токены». При подозрении на
утечку: Revoke токена на странице `supertazick-oss` → выпустить новый с теми
же скоупами → `GH_CONFIG_DIR=~/.config/gh-reviewer gh auth login --with-token`.
