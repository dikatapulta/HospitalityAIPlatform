# Спека 0015 — Оркестратор и первый инструмент (canonical tool)

- **Задача:** Task 0015 (PHASE0.md), FOUNDATION §7.1 (оркестратор), §7.3 (tools),
  §7.4 (RAG/P1-бар галлюцинаций), §7.5 (промпт — код), §7.7 (evals);
  P-5, P-9, P-12; R-2/R-7.
- **Статус:** черновик на согласование. Код не пишется до одобрения основателя.
- **Открывает:** 0016 (канал Telegram). Класс модели этого окна (пишет спеку/код) —
  сильная, effort high/xhigh. Класс модели рантайма (`LLM_MODEL`) — решается
  golden-set'ом внутри этой задачи (см. §«Модель рантайма»).

## Проблема

Платформе нужен первый реальный AI-вызов и **канонический паттерн
«инструмент = тонкая обёртка над сервисом ядра»** (P-5) — самый важный AI-паттерн
проекта. Оркестратор (§7.1) принимает нормализованное сообщение гостя + контекст
тенанта/гостя, зовёт LLM через `ai/gateway` с одним инструментом
`create_service_request` и превращает намерение гостя в заявку через модуль
`requests`. Промпт — версионируемый файл (§7.5). Всё, что видит гость, проходит
один golden-set с одним порогом до продакшена (§7.7, инвариант качества из
DISCUSSION_LOG).

**Технический крючок (обязателен, не опция):** нынешний `ai/gateway` **не умеет
tool-calling** — `LlmRequest` несёт только `messages/system/max_tokens`,
`LlmResponse` — только `text`, `AnthropicProvider` не передаёт `tools=` и не
разбирает `tool_use`. Оркестратор с инструментом невозможен без расширения
gateway. §7.2 требует «весь LLM-трафик только через gateway», поэтому tool-контракт
живёт в `ai/gateway`, а не в оркестраторе. Альтернатив, совместимых с §7.2, нет.

## Решение

| Файл | Что даёт |
| --- | --- |
| `ai/gateway/schemas.py` | Расширение: `ToolSpec`, `ToolCall` в `LlmRequest`/`LlmResponse` (R-6) |
| `ai/gateway/provider.py` | `LlmProviderResult` получает `tool_calls`, `stop_reason` |
| `ai/gateway/anthropic_provider.py` | Передаёт `tools=`, парсит `tool_use`-блоки |
| `ai/gateway/mock_provider.py` | Сценарный `tool_use` для CI-тестов оркестратора |
| `ai/orchestrator.py` | Оркестратор §7.1: сообщение → gateway → исполнение tool-call → исход |
| `ai/tools/registry.py` | Реестр инструментов (генерируемая документация, §7.3) |
| `ai/tools/create_service_request.py` | **CANONICAL tool**: обёртка над `requests.create_request` |
| `ai/prompts/concierge_v1.md` | Системный промпт с версией (§7.5); хешируется `compute_prompt_hash` |
| `ai/evals/` | Golden-set: CI-набор (mock) + офлайн бейк-офф моделей (реальные LLM) |
| `docs/adr/ADR-010-*.md` | Обобщение языкового правила §7.7 (см. §«Golden-set») |

### Расширение gateway под tool-calling (в пакете `ai/gateway`)

- `ToolSpec` (Pydantic): `name`, `description`, `input_schema` (JSON Schema),
  `confirmation_class` (`auto`/`confirm_guest`/`confirm_staff`, P-9).
  `input_schema` строится оркестратором на запросе (в т.ч. `enum` доступных
  категорий тенанта — см. ниже), поэтому tool-схема не хранится статически.
- `LlmRequest` += `tools: list[ToolSpec] = []`, `tool_choice` (auto по умолчанию).
- `LlmProviderResult`/`LlmResponse` += `tool_calls: list[ToolCall]`
  (`name`, `arguments: dict`, `id`) и `stop_reason`. `text` остаётся.
- `AnthropicProvider.complete()`: передаёт `tools=` в Messages API, собирает
  `tool_use`-блоки из `response.content` в `tool_calls`. Модель — из настроек,
  без маршрутизации (Non-Goal сохраняется).
- `MockLlmProvider`: настраиваемый сценарий — отдать `tool_use(name, arguments)`
  либо текст. Это Fake-адаптер (ADR-007), им пользуются CI-тесты оркестратора.
- `compute_prompt_hash` расширяется, чтобы включать сериализацию `tools`
  (набор инструментов — часть «версии промпта», §7.2).
- Стоимость/бюджет/журнал `LlmCallLog` не меняются: считаются на токенах, tool-use
  токены учитываются провайдером как обычно.
- Обновляется `ai/gateway/README.md` (в том же PR, R-4): раздел про tool-calling +
  сценарий изменения «новый инструмент».

### Контракт оркестратора (`ai/orchestrator.py`)

`handle_message(inbound, *, history, provider=None) -> OrchestratorTurn`,
вызывается внутри `tenant_context(...)` (P-4). Оркестратор **без бизнес-логики**
(P-5): только сборка запроса, вызов gateway, исполнение объявленного инструмента,
формирование исхода. `history` (прежние реплики диалога) передаёт вызывающая
сторона — оркестратор в Phase 0 не персистит (Conversation/Message появляются в
Task 0016).

Порядок:
1. Собрать `LlmRequest`: `system` из `concierge_v1.md`, `messages` = история +
   входящее, `tools` = реестр с `input_schema`, где категории заявок —
   `enum` ключей из `requests.list_categories()` текущего тенанта.
2. `gateway.complete(request, provider=provider)`.
3. Разобрать исход:
   - **нет tool-call** → текстовый ответ гостю (`OrchestratorTurn.reply`).
   - **tool-call класса `confirm_guest`** → пройти гейт подтверждения (ниже).
   - **эскалация** → исход `needs_human` (ниже).

`OrchestratorTurn` — типизированный результат (P-7): `kind`
(`reply`/`awaiting_confirmation`/`action_done`/`needs_human`), текст гостю,
опционально `created_request_id` и `pending_action`.

### CANONICAL tool `create_service_request` (§7.3, P-5)

- Контракт: имя, типизированный вход/выход (Pydantic), `confirmation_class =
  confirm_guest`, объявляет зависимость от модуля `requests`.
- **Вход инструмента — `category_key` (slug), а не `category_id` (UUID).** Гость
  говорит текстом; LLM не знает UUID и не должен его выдумывать. Оркестратор
  подставляет в `input_schema` `enum` реальных ключей категорий тенанта; обёртка
  резолвит key→id через `requests` и зовёт `create_request`. Это заземляет модель
  (§7.4: не галлюцинировать). Ключ не из списка → ошибка (см. §«Ошибки»), не тихое
  создание «не той» заявки.
- Поля: `category_key`, `summary`, `room_number?`, `details?` — маппятся в
  `ServiceRequestCreate`.

### Гейт подтверждения (P-9 `confirm_guest`) — структурно, не промптом

Класс подтверждения — свойство контракта инструмента; исполнение гейтит
оркестратор, а не текст промпта.

- На tool-call `confirm_guest` оркестратор **не исполняет** сервис на том же ходу.
  Возвращает `OrchestratorTurn(kind=awaiting_confirmation, pending_action=...)` и
  подтверждающий вопрос гостю («Оформить уборку номера 305?»).
- Заявка создаётся на последующем ходу, когда: (а) история показывает, что это же
  действие уже было предложено (подтверждение запрошено), и (б) гость подтвердил —
  модель, следуя промпту, повторяет tool-call. Детект «уже предложено» —
  структурный (прошлая реплика ассистента несла предложение); детект «гость
  согласился» — работа модели (при отказе она не повторит tool-call).
- Деньги/документы/изменение брони (`confirm_staff`) в Task 0015 не появляются —
  но класс в контракте заложен, чтобы будущие инструменты копировали канон (NG-4).

### Эскалация к человеку — исход оркестратора, а не второй инструмент

Карточка требует ровно один инструмент. Эскалация моделируется как исход
`needs_human`, не как tool. Для высокоставочного вопроса без источника знаний
(RAG ещё нет — Phase 1) корректное поведение = не выдумывать, а вернуть «позову
сотрудника». Это прямой P1-инвариант §7.4: экономия на модели не становится
экономией на честности. Golden-сценарий «нужна эскалация» проверяет именно отказ
от галлюцинации.

### Промпт как код (§7.5)

`ai/prompts/concierge_v1.md` — системный промпт с явной версией в имени.
Инструктирует: отвечать на языке гостя; для `confirm_guest`-действий сперва
подтвердить у гостя; при высокоставочном вопросе без опоры — эскалировать, не
выдумывать. Изменение промпта → новая версия файла + прогон evals (§7.7).

## Модель рантайма (`LLM_MODEL`)

- Дефолт **меняется с `claude-opus-4-8`** (избыточен/дорог для массового трафика,
  DISCUSSION_LOG). Кандидаты — **`claude-haiku-4-5`** ($1.00/$5.00 за Mtok, 200K
  контекст) и **`claude-sonnet-5`** ($3.00/$15.00; интро $2.00/$10.00 до
  2026‑08‑31), оба на существующем `AnthropicProvider` — смена `LLM_MODEL` + строка
  прайс-листа, ноль нового кода провайдера (асимметрия стоимости постройки,
  DISCUSSION_LOG). Opus 4.8 снят по себестоимости; GPT-4o-mini исключён (потребовал
  бы ревизии ADR-002 + новый адаптер/Fake — не окупается в скелете).
- В `MODEL_PRICING_USD_PER_MTOK` добавляются строки обоих кандидатов. Для Sonnet 5
  берётся **стандарт $3/$15** (не интро), чтобы COGS не занижался молча после
  2026‑08‑31; интро-скидка — комментарием.
- **Финальный выбор — по данным бейк-оффа (ниже), а не по интуиции.**
  Результат (2 прогона × 14 сценариев на реальных моделях, 2026-07-13):
  `LLM_MODEL = claude-haiku-4-5`. И Haiku 4.5, и Sonnet 5 в обоих прогонах на
  ВСЕХ 6 языках прошли P1-бар (§7.4): ни разу не выдумали цену позднего выезда —
  честно эскалировали (Sonnet на kk прямо: «бағаны ойдан айтпағым келеді» —
  «не хочу выдумывать цену»); оба дают корректный казахский; выбор категории при
  вызове инструмента всегда верный (`housekeeping`). При равном качестве Haiku
  втрое дешевле ($1/$5 vs $3/$15) — при самофинансируемом пилоте это решает.

## Golden-set (§7.7): два раздельных артефакта

Демографию пилота нельзя проверить mock-провайдером — он отдаёт заготовку и не
судит качество. Поэтому набор разделён:

1. **CI-evals (`golden set v0`, на mock):** структурные, детерминированные,
   бесплатные, гейтят merge. Проверяют канон пути «текст → выбор инструмента →
   confirm-гейт → сервис `requests` → заявка». Язык не значим (mock не смотрит
   содержание) — набор маленький, как в карточке.
2. **Model-selection bake-off (офлайн, на РЕАЛЬНЫХ `claude-haiku-4-5` и
   `claude-sonnet-5`):** golden-set на **6 языках пилота** (en/ru/kk/zh/tr/hi).
   Сценарии: несколько категорий заявок + высокоставочный контент (цены/правила/
   бронь — P1-бар §7.4) + «нужна эскалация человеку». Приоритет проверки —
   **казахский** (спящий низкоресурсный риск) и перевод знаний в zh/tr/hi.
   Не в CI (стоит токены, недетерминирован). Результат **фиксирует `LLM_MODEL`**.
   Метод оценки (LLM-judge с рубрикой vs ручная выборка), формат фикстур и порог —
   финализируются при реализации; результат заносится в спеку/DISCUSSION_LOG.
   Требует `ANTHROPIC_API_KEY` + токен-бюджет (пункт передачи смены).

**Оговорка по фазам:** полноценная проверка «перевод знаний RAG на лету без
искажения цены» требует RAG, а RAG — Phase 1. В Phase 0 источника знаний нет,
поэтому 6-язычный сет в 0015 проверяет tool-selection + confirm + отказ-от-
галлюцинации/эскалацию на всех 6 языках; fidelity-evals перевода знаний — задел на
Phase 1 (помечается хук). Это соблюдает фазировку (не строим Phase 1 в Phase 0).

**ADR-010 (обобщение §7.7 под демографию, R-11):** §7.7 сейчас говорит «на 3
языках». ADR-010 обобщает правило: golden-set покрывает **реальную демографию
тенантов**, а не фиксированное число; для первого пилота = 6 языков (en/ru/kk/zh/
tr/hi). §7.7 правится ссылкой на ADR-010. (ADR-009 занят — outbox-retention.)
Пишется в том же PR, что и код задачи.

## План тестов (R-7)

На `MockLlmProvider` (детерминированно, CI):
- «уберите номер 305» → tool-call `create_service_request(category_key=…)` →
  подтверждение гостя → заявка в БД (полный путь текст→LLM(mock)→инструмент→сервис).
- сообщение-вопрос без действия → текстовый ответ, инструмент не вызван.
- **отказ при неподтверждении:** предложение `confirm_guest` → `awaiting_
  confirmation`, заявка НЕ создана (гейт исполнения).
- `category_key` вне enum → ERR-AI-004 путь (переспрос/эскалация), заявки нет.
- изоляция тенанта: категории берутся тенантной сессией (чужие не видны).
- `golden set v0`: 3 сценария на mock, помечены как golden set v0 (карточка).

Офлайн (не CI): 6-язычный bake-off на реальных Haiku/Sonnet — прогон вручную,
результат фиксирует `LLM_MODEL` и заносится в DISCUSSION_LOG/спеку.

## Находки bake-off'а под prompt v2 (не блокируют Task 0015)

Golden-set сделал свою работу — вскрыл два дефекта, общих для обеих моделей
(значит, это про промпт, а не про выбор модели). Оба — кандидаты в правку
`concierge_v1.md` → `concierge_v2.md` до пилота (Task 0017/Phase 1), с повторным
прогоном bake-off'а:

1. **Утечка языка.** Обе модели иногда отвечают по-русски на английский/иной
   ввод. Явное правило про язык уже есть (`concierge_v1.md`, секция «Язык»), и
   всё равно течёт — значит, дело не в отсутствии инструкции, а в фоновом
   русском системного промпта, который перебивает правило на коротком/
   неоднозначном вводе. Для пилота, где 70% гостей — иностранцы, это дефект.
   **Фикс (prompt v2):**
   - весь промпт `concierge_v2.md` — на английском (совпадает с правилом
     «язык кода — английский» + «промпты — код»);
   - правило про язык — первой строкой и жёстче: «Reply in the exact same
     language as the guest's most recent message; detect it fresh every turn;
     never reply in the language of these instructions or of earlier turns»
     (при желании продублировать в конце — recency);
   - примеры-реплики (сейчас русский образец «Оформить уборку… — верно?») —
     пометить как illustrative-only, ответ производить на языке гостя;
   - **сделать измеримым:** в `evals/bakeoff.py` добавить ассерт «язык ответа
     == язык ввода» на всех 6 языках — этот прогон и подтвердит фикс, а не
     ручной осмотр. Бэкстоп (детерминированный детект языка на выходе +
     регенерация при рассинхроне) — только если после eval всё ещё течёт.
2. **Непостоянство протокола подтверждения.** Модель то вызывает инструмент (и
   оркестратор перехватывает по гейту P-9), то сперва переспрашивает гостя
   ТЕКСТОМ без вызова инструмента. Второе — тоже разумный UX, но текущий гейт
   (§«Гейт подтверждения») рассчитан на tool-first: при text-first `pending_action`
   не формируется. Варианты: (а) жёстче в промпте требовать tool-call; (б)
   научить оркестратор трактовать текстовый переспрос как ожидание подтверждения.
   Решается вместе с контрактом истории Task 0016.
