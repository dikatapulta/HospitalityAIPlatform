# Спека 0016 — Канал Telegram

- **Задача:** Task 0016 (PHASE0.md), FOUNDATION §7.1 (вход оркестратора), §8.4
  (подписи/секреты вебхуков), §11 (RLS/идентичность, аутентификация эндпоинта),
  §9 (Conversation/Message); P-4, P-7, P-8, P-12; R-2/R-4/R-7.
- **Статус:** реализовано в этом PR (спека фиксирует принятые решения и границу
  задачи для ревьюера).
- **Открывает:** 0017 (сквозная сборка). Класс модели этого окна — сильная,
  effort high.

## Проблема

Платформе нужен первый реальный вход гостя. Канал Telegram принимает вебхук Bot
API, приводит сообщение к общему контракту (P-7), идемпотентно сохраняет диалог
(P-8) и отвечает. Идемпотентность канала — канон для WhatsApp/Email в будущем
(P-12), поэтому паттерн важнее конкретного канала.

## Граница задачи (ключевое решение)

Task 0016 доводит канал до **сохранённого `Message` и ответа гостю**; вызов
оркестратора и создание заявки — **Task 0017** («соединить всё: Telegram →
оркестратор → инструмент → заявка → …»). Основания держать границу здесь:

- DoD карточки 0016 — ровно «сообщение появляется как `Message` в БД с
  correlation_id»; её тесты — секрет/идемпотентность/нормализация. Ни AI, ни
  заявок в списке 0016 нет.
- PHASE0: «если нужно ещё чуть-чуть добавить — это новая задача». 0017 существует
  именно для склейки.
- Оркестратор (0015) **уже спроектирован** под этот канал: в докстринге сказано,
  что историю и `pending_action` между ходами хранит «вызывающая сторона (канал,
  Task 0016)». Это форвард-контекст (что канал будет звать), а не требование
  звать сейчас; §7.8 (деградация при недоступности LLM — «забота канала») тоже
  прицепляется в 0017 вместе с вызовом.

Практическое следствие: на **текст** канал в 0016 отвечает молчанием (ответ —
AI в 0017), на **не-текст** — вежливым отказом (это и упражняет путь отправки, и
требуется карточкой). Никакого выбрасываемого «заглушка-ответа» на текст не
пишется.

### Контракт нормализованного сообщения (P-7)

Единый формат входящего для всех каналов. Поле `reply_to` **зарезервировано с
самого начала** (DISCUSSION_LOG): Telegram шлёт полный объект — заполняется сразу;
WhatsApp (Phase 1) даст только id и восстановит текст по `external_message_id` из
истории. Так WhatsApp-адаптеру не придётся менять контракт задним числом (P-12).
Это закрывает открытый вопрос №4 спеки 0015 (reply_to зарезервировано в контракте
входящего сообщения).

### Аутентификация вебхука (§8.4, §11)

Telegram не подписывает тело HMAC'ом: при `setWebhook` боту задаётся `secret_token`,
который Telegram шлёт в заголовке `X-Telegram-Bot-Api-Secret-Token` на каждом
запросе. Проверка — постоянного времени, **до** разбора тела (FastAPI решает
Depends раньше body → чужой запрос получает 403, а не 422 с формой payload).
Пустой секрет = вебхук закрыт (fail-closed, §11: «эндпоинт рождается
аутентифицированным»). Вебхук **не** использует сервисный токен `/api/v1/*` и
`require_authenticated_tenant` — тенанта ставит сам по маппингу чата.

### Маппинг чата на тенанта (Phase 0)

Один бот = демо-тенант, по slug из `TELEGRAM_TENANT_SLUG` (как
`service_token_tenant_slug`). Тенант не приходит от клиента (§11). Пер-чатовая
привязка (несколько отелей за одним ботом) — Phase 1.

### Идемпотентность (P-8)

`update_id` (уникальный у Telegram) namespace'ится в `idempotency_key`
(`telegram:update:<id>`) и ложится под уникальное ограничение
`messages(tenant_id, idempotency_key)`. Дедуп — через нарушение ограничения на
вставке, а не проверкой-перед-вставкой: между SELECT и INSERT возможна гонка двух
доставок, БД её закрывает. Исходящие имеют NULL-ключ (Postgres считает NULL-и
различными). Отправка ответа — best-effort: сбой сети не роняет вебхук и не
записывает недоставленный ответ; повторную доставку Telegram дедуплицирует по
входящему (второго ответа не будет) — компромисс Phase 0, задокументирован.

### Таблицы и RLS (P-4, §9)

`conversations`, `messages` — тенантные, RLS-блок скопирован из канона 0002
(ENABLE + FORCE + политика `tenant_isolation`). Изоляция — обязательный тест на
каждой новой таблице. `messages.correlation_id` — прямая опора DoD (строка
связана со следом запроса в логах, §10.2). Phase 0: гость — строка `Conversation`
(модуля `guests/` нет); `Guest`/`GuestIdentity` — Phase 1, отдельными таблицами.

`ignore_imports` (production-код `modules/` канал не видит).

## План тестов (R-7)

- нормализация: текст → TEXT; не-текст → UNSUPPORTED; reply_to из полного объекта;
  не-сообщение → None;
- вебхук (AsyncClient + БД): неверный/пустой секрет → 403 ERR-TELEGRAM-001, ничего
  не сохранено; текст → 200, один `Message` inbound с correlation_id, ответа нет;
  дубликат `update_id` → один `Message` (P-8); не-текст → входящее + исходящий
  отказ, отправитель вызван; не-сообщение → 200 без эффектов;
- store: `ensure_conversation` идемпотентна; повторный ключ → None;
- изоляция тенантов на `conversations`/`messages` (WITH CHECK + платформенная
  сессия не видит).

## DoD (карточка)

- Дубликат update не создаёт второй `Message`; неверный секрет → 403; нормализация
  форматов (текст; не-текст → вежливый отказ) — тестами.
- Зелёный CI (ruff, lint-imports, mypy strict, тесты, покрытие; README канала).
- На staging: сообщение реальному боту → `Message` в БД с correlation_id (требует
  зарегистрированного вебхука по HTTPS — см. `docs/runbooks/telegram.md`; шаг
  основателя, как секреты).
