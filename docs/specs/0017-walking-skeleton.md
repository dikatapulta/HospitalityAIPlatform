# Spec 0017 — Сквозная сборка Walking Skeleton

- **Статус:** accepted
- **Задача:** PHASE0 Task 0017 (открывает 0018, 0019)
- **Затрагивает:** `channels/telegram`, `hospitality/worker.py`, `shared/config.py`,
  миграция 0009. Доменные модули (`modules/requests`, `ai/*`) **не меняются** —
  проверка гипотезы, что их контракты стыкуются без правок.

## Проблема

Все кирпичи скелета существуют по отдельности (Task 0012–0016): заявки с событиями
(`request.created`, `request.status_changed`), оркестратор + инструмент
`create_service_request`, канал Telegram (вебхук → нормализация → Message), шина
событий (outbox → воркер). Ни разу они не проходили путь целиком. Task 0017 —
первый прогон скелета от сообщения гостя до подтверждения о выполнении.

Целевой поток:

```
гость пишет боту ──▶ вебхук ──▶ оркестратор ──▶ create_service_request
   (Telegram)                                          │
                                                        ▼
                                             requests.create_request
                                                        │  (P-6: событие в той же транзакции)
                                                        ▼
                                              outbox: request.created
                                                        │  (воркер, ADR-005)
                                                        ▼
                              подписчик notify_staff ──▶ уведомление в staff-чат Telegram
                                                        │
   сотрудник в staff-чате: /assign /start /done ◀───────┘
                    │
                    ▼
      requests.change_request_status(→ done)
                    │  (P-6: событие)
                    ▼
       outbox: request.status_changed(done)
                    │  (воркер)
                    ▼
   подписчик notify_guest ──▶ подтверждение гостю в его чат
```

## Ключевое правило (P-6) и проверка R-11

Уведомления — **подписчики событий**, а не прямые вызовы из `modules/requests`.
Модуль `requests` не знает ни про Telegram, ни про то, что кто-то уведомляет
службу или гостя: он лишь публикует факты. Подписчиков подключает composition
root воркера (`hospitality/worker.py`), как `include_router` в `app.py`.

**Проверка стыкуемости контрактов (R-11).** При сборке скелета проверено, что
существующие контракты соединяются без правок доменных модулей. Единственный
настоящий пробел — обратная адресация: событие `request.status_changed` несёт
`request_id`, но **не** знает, в какой чат какого канала вернуть подтверждение
гостю (в Phase 0 модуля `guests/` и идентичностей ещё нет — гость это строка
`Conversation`). Это не дефект контракта события и не повод менять схему: событие
описывает доменный факт, а сопоставление «заявка → куда доставить» — забота
композиционного слоя (канала). Решение — канал ведёт свою таблицу привязки
`request_origins` (см. ADR-011). Схемы события, интерфейс инструмента и
`channels/base.py` **не меняются** — стыкуется без хаков.

## Решение (что добавляем)

Всё новое — в композиционном слое (`channels/telegram`, `worker.py`) и конфиге.

### 1. Гостевой ход: канал зовёт оркестратор (`channels/telegram/guest.py`)

На **текстовое** сообщение из гостевого чата канал вызывает
`ai.orchestrator.handle_message` и отвечает `reply_text` гостю. Между ходами канал
хранит состояние диалога, которого требует оркестратор (Task 0015 спроектирован
под это):

- **История** (`history`) — восстанавливается из прежних `Message` диалога
  (inbound → `user`, outbound → `assistant`); текущее входящее исключается.
- **`pending_action`** — гейт подтверждения P-9 переживает два вебхука (гость
  просит → «оформить?» → гость «да»). Хранится в новой колонке
  `conversations.pending_action` (JSONB, NULL = нет ожидания). Ставится при
  `AWAITING_CONFIRMATION`, перезаписывается результатом каждого хода (при
  `ACTION_DONE`/`REPLY` → NULL, т.е. очищается сама).
- **Провайдер LLM** прокидывается зависимостью FastAPI (`get_orchestrator_provider`,
  по умолчанию `None` → боевой Anthropic; тесты подменяют scripted-фейком) —
  тот же приём подмены, что у отправителя (`get_telegram_sender`).

Исход хода:
- `REPLY` → отправить текст гостю.
- `AWAITING_CONFIRMATION` → отправить вопрос, сохранить `pending_action`.
- `ACTION_DONE` → заявка создана: записать привязку `request_origins(request_id →
  conversation_id)`, отправить текст. Уведомление службе идёт **не отсюда**, а
  подписчиком `request.created`.
- `NEEDS_HUMAN` → отправить текст эскалации (оркестратор уже перевёл ошибку
  инструмента в эскалацию).

Деградация (§7.8): `AppError` провайдера (ERR-AI-001/002/003) из
`handle_message` не роняет вебхук — канал ловит его, логирует и шлёт гостю
честный фолбэк-текст (структурные формы/кнопки — Phase 1).

### 2. Уведомление службы (`channels/telegram/notifications.py`, подписчик `request.created`)

Подписчик `notify_staff_on_request_created`: на `request.created` шлёт в staff-чат
(`TELEGRAM_STAFF_CHAT_ID`) сообщение с сутью заявки и готовыми командами закрытия.
Идемпотентность (P-8): исход фиксируется исходящим `Message` с ключом
`staff:request_created:<request_id>` — повторная доставка события не шлёт второе
уведомление. Staff-чат не настроен → подписчик логирует и выходит (скелету на
staging чат обязателен).

### 3. Staff-команды (`channels/telegram/staff.py`)

Если входящий чат == `TELEGRAM_STAFF_CHAT_ID`, канал трактует сообщение как команду
персонала, **не** зовёт оркестратор и не создаёт заявок. Команды — по одной на
переход жизненного цикла (карта `STATUS_TRANSITIONS` модуля requests не
обходится):

| Команда | Переход |
| --- | --- |
| `/assign <request_id>` | new → assigned |
| `/start <request_id>` | assigned → in_progress |
| `/done <request_id>` | in_progress → done |
| `/cancel <request_id>` | * → cancelled |

Обработчик зовёт `requests.change_request_status` и отвечает персоналу результатом
(новый статус) или понятной ошибкой (`ERR-REQUESTS-002` не найдена / `-003`
недопустимый переход). Guest-подтверждение при `done` идёт **не отсюда**, а
подписчиком `request.status_changed` — команда лишь публикует событие.

### 4. Подтверждение гостю (`notifications.py`, подписчик `request.status_changed`)

`notify_guest_on_request_done`: срабатывает только при `new_status == done`. По
`request_id` находит `request_origins → conversation_id → external_id` (чат гостя)
и шлёт подтверждение. Привязки нет (заявка создана не через Telegram, например
curl-ом) → логирует и выходит. Идемпотентность — исходящий `Message` с ключом
`guest:request_done:<request_id>`.

### 5. Регистрация подписчиков (`worker.py`)

`register_subscribers()` дополняется двумя парами (событие → обработчик) канала
Telegram; отправитель и staff-chat-id связываются `functools.partial` при
регистрации. Это единственное место склейки (P-12), аналог `include_router`.

## Контракты и данные

Новые артефакты — только композиционного слоя; доменные контракты не тронуты.

- **Колонка** `conversations.pending_action JSONB NULL` — состояние гейта P-9.
- **Таблица** `request_origins` (тенантная, RLS — канон 0002): `id`, `tenant_id`,
  `request_id` (уникален в паре с `tenant_id`), `conversation_id` (FK), `created_at`.
  `request_id` — **без FK** на `service_requests`: канал не связывает свою схему с
  таблицей чужого модуля (P-2), хранит id как непрозрачную ссылку из события.
- **Конфиг** `TELEGRAM_STAFF_CHAT_ID` (пусто = уведомления службы выключены).

## План тестов (R-7)

Интеграционный тест — на mock-провайдере (LLM не дёргается), через настоящий
`create_app` (ASGI) + инлайновый прогон воркера `deliver_pending_events()`:

1. **Сквозной поток на mock-провайдере (главный тест DoD):**
   вебхук гостя «уберите номер 305» → `AWAITING_CONFIRMATION` (заявки нет,
   `pending_action` сохранён) → вебхук «да» → заявка создана + `request_origins`
   записан → `deliver_pending_events()` доставляет `request.created` → staff-чат
   получил уведомление → вебхуки `/assign`,`/start`,`/done` в staff-чат →
   `deliver_pending_events()` доставляет `request.status_changed(done)` → гость
   получил подтверждение. Проверяется, что уведомление службе несёт **тот же
   correlation_id**, что и исходное сообщение гостя (outbox протаскивает его
   через async-границу).
2. **Идемпотентность подписчиков (P-8):** повторная доставка `request.created` /
   `request.status_changed` не шлёт второго уведомления.
3. **Staff-команды:** недопустимый переход → `ERR-REQUESTS-003` в ответ персоналу;
   неизвестная команда / кривой UUID → понятная подсказка; заявка не меняется.
4. **Гость, вопрос без действия:** модель вернула текст без инструмента → `REPLY`,
   заявка не создаётся.
5. **Деградация:** `AppError` провайдера → фолбэк-текст гостю, вебхук отвечает 200.
6. **store**: `pending_action` round-trip; `request_origins` идемпотентность записи;
   восстановление истории исключает текущее сообщение.

Плюс правка существующих тестов канала: текст теперь получает AI-ответ (Task 0016
на текст не отвечал) — вебхук-фикстура подменяет провайдер, ассерты обновлены.

## Границы (сознательно вне Task 0017)

- UI персонала (кабинет) — Phase 1; staff-команды в чате это его временная замена.
- Идентичность гостя (`guests/`, `GuestIdentity`) — Phase 1; `request_origins`
  будет вытеснен резолвом идентичности гостя (ADR-011, «Последствия»).
- Маршрутизация заявок в конкретную службу, SLA, эскалации — Phase 1.
- Полноценная §7.8-деградация (кнопки/формы) — Phase 1; сейчас фолбэк-текст.
- Богатое управление диалогом (сброс `pending_action` по смене темы, мультивызов
  инструментов за ход) — Phase 1.
