# Спека 0017.1 — Детерминированное исполнение подтверждения P-9 в оркестраторе

- **Задача:** Task 0017.1 (PHASE0.md), hotfix issue #31; FOUNDATION §7.1, §7.3,
  §7.5; P-5, P-7, P-9; R-2/R-7. Класс ревью — A (канон оркестратора +
  safety-гейт P-9).
- **Статус:** hotfix бага, найденного при верификации DoD Task 0017 на staging
  (2026-07-14).
- **Открывает:** 0018 (не блокирует, но должен лечь до аудита фазы 0020).

## Проблема

На ходе подтверждения гость говорит «да», бот отвечает «передаю заявку службе»,
но заявка не создаётся и уведомление в staff-чат не уходит. Причины две, обе в
`ai/orchestrator.py`:

1. Ветка `if not response.tool_calls: return REPLY` срабатывает **раньше**
   какой-либо проверки `pending_action` — сохранённый гейт молча теряется.
2. Сохранённые `pending_action.arguments` (лежат в БД,
   `conversations.pending_action`) для исполнения **не используются**: код
   полагается на то, что модель на «да» повторно выдаст `tool_use` с теми же
   аргументами. Haiku 4.5 делает это нестабильно (дефект «непостоянство
   протокола подтверждения», bake-off, `docs/DISCUSSION_LOG.md`).

Итог: исполнение подтверждённого действия зависит от каприза модели — прямое
нарушение духа P-9 («гейт — свойство контракта, не текста промпта»).

## Решение

Ход подтверждения (`pending_action is not None`) перестаёт быть «обычным ходом
диалога с надеждой на ре-эмиссию». Оркестратор разделяется на два пути:

### 1. Обычный ход (`pending_action is None`) — без изменений поведения

Промпт консьержа + инструменты реестра; tool-call класса `confirm_guest` →
`AWAITING_CONFIRMATION` + `pending_action`; `auto` → исполнить; нет tool-call →
`REPLY`.

### 2. Ход подтверждения — структурная классификация + сохранённое действие

Оркестратор делает **один вызов классификации** через gateway:

- `system` — новый версионируемый промпт `confirmation_gate_v1.md` (§7.5) +
  компактное описание ожидающего действия (`tool_name` + `arguments`);
- `messages` — та же история диалога + реплика гостя (язык гостя виден модели);
- `tools` — ровно один служебный инструмент `resolve_confirmation` со схемой
  `{decision: enum[confirm|decline|other], reply: string}`;
- `forced_tool="resolve_confirmation"` — **принудительный выбор инструмента**
  (`tool_choice: {type: "tool"}` Anthropic): модель обязана вернуть структурный
  вердикт, свободный текст невозможен. Это и есть «структурная классификация
  надёжнее ре-эмиссии tool_use» из issue #31.

Дальше — детерминированно, без участия модели:

| `decision` | Действие оркестратора | Исход |
| --- | --- | --- |
| `confirm` | `registry.execute(pending_action.tool_name, pending_action.arguments)` — **сохранённые** аргументы, не пересказ модели | `ACTION_DONE` (+`created_request_id`); текст гостю — `reply` классификатора (язык гостя), fallback — статичный |
| `decline` | ничего не исполнять | `REPLY`; гейт гаснет (канал уже очищает `pending_action` на всех исходах, кроме `AWAITING_CONFIRMATION`) |
| `other` (передумал, правка деталей, другая тема) | старое предложение снимается; сообщение обрабатывается **как новый запрос** — обычный путь №1 | исход пути №1 (`REPLY` / новое `AWAITING_CONFIRMATION` / …) |

Ошибка исполнения (`ERR-AI-004`, ERR-REQUESTS-*) — как раньше: `NEEDS_HUMAN`,
не 5xx. Нарушение протокола классификатором (нет вызова `resolve_confirmation`
в ответе — с боевым forced tool choice недостижимо, возможно только с
неправильно настроенным фейком) — безопасная сторона: предупреждение в лог +
трактовка как `other` (ничего не исполняется молча и ничего не создаётся).

`reply` классификатора решает и утечку языка на системных репликах: текст
«готово/отменил» пишет модель на языке гостя, а не русская константа.

### Расширение gateway: `forced_tool`

`LlmRequest` += `forced_tool: str | None = None` (имя инструмента, который
модель обязана вызвать). `AnthropicProvider` транслирует в
`tool_choice={"type": "tool", "name": ...}`; `None` — прежнее поведение (auto).
Поле входит в `compute_prompt_hash` (влияет на поведение — часть «версии
промпта», §7.2). Расширение аддитивно (политика совместимости §13.5) и было
заложено в спеке 0015 («`tool_choice` (auto по умолчанию)»), но не реализовано.
Fake-провайдеры сценарные — поле не интерпретируют, тесты видят его в
`provider.calls`.

### Промпт v2 (`concierge_v2.md`) — заодно 2 дефекта bake-off'а

По находкам bake-off'а (spec 0015, DISCUSSION_LOG «Модель для гостевого
диалога»):

1. **Утечка языка** — промпт целиком на английском (фоновый русский v1 перебивал
   правило); правило языка — первой строкой и жёстче («detect it fresh every
   turn; never reply in the language of these instructions»); примеры помечены
   illustrative-only.
2. **Переспрос как «уже сделано»** — явное требование: предложение действия —
   всегда вопрос; запрещено писать «передаю/оформил», пока система не исполнила
   действие. Структурно это подкреплено фиксом: текст «готово» теперь появляется
   только ПОСЛЕ фактического исполнения (из `reply` классификатора).

`PROMPT_NAME = "concierge_v2"`; `concierge_v1.md` остаётся в репозитории как
версия-предшественник (§7.5: версия — в имени файла).

## Контракты

- `OrchestratorTurn` и сигнатура `handle_message` — **без изменений**: канал
  (`channels/telegram/guest.py`) не трогается.
- Меняется провайдер-протокол хода подтверждения: вместо «консьерж-промпт +
  реестр инструментов» — «классификатор + forced `resolve_confirmation`».
  Существующие тесты, скриптовавшие ре-эмиссию `create_service_request` на
  втором ходе, переводятся на новый протокол (сами утверждения тестов — про
  создание/несоздание заявки — не меняются).
- `resolve_confirmation` — **служебный механизм гейта, не AI-способность**: в
  реестр инструментов (§7.3) не входит, класса подтверждения не имеет, сервисов
  ядра не вызывает.

## Уточнение к формулировке воспроизводящего теста в карточке (R-11)

Карточка требует тест «фейк возвращает утвердительный текст БЕЗ `tool_use` →
заявка всё равно создаётся». Формулировка написана против старого протокола,
где на ходе подтверждения провайдер отвечал за консьержа. В новом протоколе
утвердительный ответ гостя превращается не в свободный текст, а в структурный
вердикт `confirm` (forced tool choice это гарантирует на реальном API).
Воспроизводящий тест поэтому формулируется на уровне инварианта бага: **модель
НЕ ре-эмитит `create_service_request` (возвращает только вердикт) → заявка всё
равно создаётся из сохранённого `pending_action`**. На текущем коде тест
красный (оркестратор видит незнакомый инструмент → `NEEDS_HUMAN`, заявки нет) —
баг воспроизведён; после фикса — зелёный. Случай «фейк вернул свободный текст
вместо вердикта» покрыт отдельным тестом безопасного fallback'а (`other`).

## План тестов (R-7)

`ai/tests/test_orchestrator.py` (Fake-провайдер, детерминированно):

1. **Воспроизводящий (красный до фикса):** предложение → на ходе подтверждения
   провайдер возвращает только вердикт `confirm` (никакой ре-эмиссии
   `create_service_request`) → заявка создана из сохранённого `pending_action`;
   второй запрос к провайдеру — классификация (`forced_tool ==
   "resolve_confirmation"`, единственный инструмент — `resolve_confirmation`).
2. **Отказ:** вердикт `decline` → `REPLY`, заявки нет, `pending_action` исхода
   пуст (гейт гаснет на стороне канала).
3. **Передумал:** вердикт `other` → обычный путь: новый tool-call →
   новое `AWAITING_CONFIRMATION` с новыми аргументами, заявки нет.
4. **Нарушение протокола фейком:** ответ классификации без вызова
   `resolve_confirmation` → fallback `other` (обычный путь), ничего не создано.
5. Существующие тесты (golden v0, unknown-category, walking skeleton) —
   переведены на новый протокол, утверждения прежние.
