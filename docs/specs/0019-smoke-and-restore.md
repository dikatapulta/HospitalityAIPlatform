# Spec 0019 — Smoke-набор v0 и репетиция восстановления

- **Задача:** PHASE0.md, Task 0019. Правила: §13.6 (smoke на языке бизнеса),
  §10.10 (бэкапы), §10.11 (восстановимость, bus factor = 1), R-2 (эта спека).
- **База ветки:** `task-0018-observability` (PR #51, ещё не в `main`) — smoke
  использует `/metrics` из Task 0018.

## Проблема

1. У основателя нет инструмента приёмки: проверить «система жива и гость
   обслуживается» можно только чтением логов или ручной перепиской с ботом.
   §13.6 требует сквозные тесты на языке продукта с Фазы 0.
2. Staging-БД не бэкапится вообще (осознанный долг Task 0006). Восстановление
   никогда не репетировалось — §10.11 требует отрепетировать до появления
   ценных данных.

## Решение: smoke-набор

### Архитектура — чёрный ящик, только HTTP

Smoke-тесты живут в `tests/smoke/` и НЕ импортируют пакет `hospitality`:
они видят систему так же, как внешний мир — через HTTP. Поэтому один и тот же
набор без изменений гоняется против локальной среды, CI-окружения и staging.

Точки входа/наблюдения:

- `POST /channels/telegram/webhook` — действия гостя (синтетический `chat_id`,
  свой на каждый прогон: история диалога не накапливается между прогонами);
- `GET /api/v1/requests` (+ `/categories`) — наблюдение бизнес-результата
  «заявка у службы» и уборка за собой (`POST .../status` → `cancelled`);
- `GET /metrics` — наблюдение «AI ответил» (прирост `llm_calls_total`
  со `status="ok"`, Task 0018);
- `GET /health/ready` — preflight с понятной диагностикой «среда не поднята».

LLM — настоящий (ADR-007: поведение модели не симулируется; фейк-провайдер —
только сценарный стаб юнит-тестов). Стоимость одного прогона — 3–4 вызова
Haiku, доли цента.

### Сценарии (язык бизнеса, §13.6)

1. **Гость просит уборку → заявка появляется у службы.** Два хода вебхука:
   просьба об уборке с номером комнаты → подтверждение «да» (гейт P-9).
   Проверка: в `/api/v1/requests` появилась новая заявка категории
   `housekeeping` с этим номером комнаты. После проверки заявка отменяется
   (не мусорить в staging; уведомление в staff-чат при этом уходит — это
   штатно и видно персоналу как отменённая заявка).
2. **Гость задаёт вопрос → получает ответ.** Вопрос про завтрак → проверка:
   успешный вызов модели состоялся (`llm_calls_total` вырос), лишняя заявка
   не создана. Текст ответа гостю снаружи системы не наблюдаем (ответ уходит
   в Telegram-чат; API диалогов появится с кабинетом персонала в Phase 1) —
   это зафиксированное ограничение v0.
3. **Дубликат вебхука → одна заявка.** Повтор того же `update_id`
   подтверждения из сценария 1 — количество заявок не меняется (P-8).
   Если сценарий 1 не прошёл, сценарий 3 пропускается с пометкой.

Против флейка настоящей модели: если после первого «да» заявка не появилась
за 15 секунд (модель переспросила уточнение), smoke шлёт одно повторное
подтверждение и ждёт ещё раз. Второй провал — честное падение.

### Вывод для основателя (DoD)

`tests/smoke/conftest.py` печатает результат по-русски, по строке на сценарий:

```
✅ Гость просит уборку → заявка появляется у службы (11.8 с)
❌ Гость задаёт вопрос → получает ответ — AI не ответил: проверь ANTHROPIC_API_KEY в .env и логи app
```

Падения формулируются действием («что проверить»), а не стектрейсом
(`--tb=no` в make-цели; полный вывод — обычным `pytest tests/smoke -m smoke`).

### Конфигурация smoke

`SmokeSettings` (pydantic-settings, читает окружение и `.env`):

| Поле | Переменная | Fallback из `.env` | Умолчание |
|---|---|---|---|
| `base_url` | `SMOKE_BASE_URL` | — | `http://localhost:8000` |
| `webhook_secret` | `SMOKE_WEBHOOK_SECRET` | `TELEGRAM_WEBHOOK_SECRET` | пусто → preflight-ошибка |
| `service_token` | `SMOKE_SERVICE_TOKEN` | `SERVICE_TOKEN` | `dev-service-token` |

### Запуск

- **Локально:** `make smoke` (`pytest tests/smoke -m smoke`; `make dev`,
  `make migrate`, `make seed` уже выполнены — preflight подскажет, если нет).
  В `.env` нужны `ANTHROPIC_API_KEY` и непустой `TELEGRAM_WEBHOOK_SECRET`
  (в `.env.example` появился дефолт `dev-webhook-secret` — тот же паттерн,
  что `SERVICE_TOKEN=dev-service-token`; staging использует свой случайный).
- **Staging:** `make smoke-staging` → `ops/smoke-staging.sh`: берёт адрес из
  `STAGING_HOST` (локальный `.env`, в репозитории адреса нет), секреты
  подтягивает по SSH из `/opt/hospitality/.env` (на диск не пишет) и гоняет
  тот же pytest с `SMOKE_BASE_URL=http://$STAGING_HOST:8000`.
- **CI:** отдельный job `smoke`: поднимает compose-стек, применяет миграции,
  сидит демо-данные и гоняет набор. Если секрет `ANTHROPIC_API_KEY` в репо
  не задан — job грациозно пропускается с notice (канон guard'а
  deploy-staging). Секрет задаёт основатель (ручной шаг передачи смены).
  Job не входит в `needs` deploy-staging: флейк модели не должен блокировать
  выкатку; данные о флейках соберёт аудит 0020.

### Сопутствующие изменения

- `ops/docker-compose.yml`: `app` не получал `ANTHROPIC_API_KEY` и
  `TELEGRAM_*` вовсе — добавлен проброс тех же переменных, что на staging
  (зеркало `docker-compose.staging.yml`); `worker` — `TELEGRAM_BOT_TOKEN` и
  `TELEGRAM_STAFF_CHAT_ID` для паритета.
- `make smoke` больше не «пинг Postgres/Redis», а бизнес-набор (P-12: одно
  имя — один смысл). Старый скрипт переименован `ops/smoke.sh` →
  `ops/env-check.sh` и остаётся проверкой compose в CI job dev-environment.
- `pyproject.toml`: маркер `smoke`; в `addopts` добавлен `-m "not smoke"` —
  юнит-прогон (`make check`, CI check) smoke не запускает и порог покрытия
  не размывает.
- `docker-compose.staging.yml`: app переведён на один uvicorn-воркер
  (override `command`). Найдено при постройке smoke: реестр Prometheus живёт
  в процессе, при `--workers 2` из CMD образа `/metrics` отдаёт счётчики
  случайного воркера — сценарий «гость получает ответ» флейковал бы, а
  алертер (ERR-OPS-002, Task 0018) видел бы половину 5xx. Нагрузке staging
  одного воркера достаточно; multiprocess-реестр — Phase 1 (issue в трекере).

## Решение: бэкап и восстановление

### Бэкап (`ops/backup/backup.sh`)

Запускается на сервере (cron) или локально; параметры — окружением
(`COMPOSE_FILE`, `ENV_FILE`, `BACKUP_DIR`, `BACKUP_RETENTION_DAYS`):

1. `pg_dump -Fc` внутри контейнера `db` (канон deploy.sh: до БД staging можно
   достать только через docker) → `BACKUP_DIR/hospitality-<UTC-метка>.dump`;
2. валидация архива: `pg_restore --list` читает свежий дамп — битый бэкап
   обнаруживается в момент создания, а не в день аварии (§10.10 «проверка
   восстановления» в минимальной автоматической форме);
3. retention: дампы старше `BACKUP_RETENTION_DAYS` (14) удаляются.

Расписание: cron на сервере, ежедневно 03:00 UTC (RPO ≤ 24ч, §10.10). Строка
crontab — в runbook restore.md; deploy CI кладёт свежий `backup.sh` на сервер
при каждом деплое (тот же scp-шаг, что compose+deploy.sh).

Offsite-копия (сервер — единственная машина с данными): `make backup-fetch`
забирает последний дамп по SSH в локальный `backups/` (в `.gitignore`).
Еженедельный ручной запуск — шаг основателя, зафиксирован в runbook.

### Восстановление (`docs/runbooks/restore.md`)

Пошаговая процедура: остановить app/worker/alerter → развернуть чистый том БД
(или новый сервер по deploy.md часть A) → `pg_restore --clean --if-exists`
внутри контейнера db → `alembic upgrade head` (дамп может быть старше кода) →
поднять стек → `make smoke-staging` + сверка количества заявок до/после.

**Репетиция — по-настоящему, один раз, на staging** (§10.11): с настоящим
удалением docker-тома БД и восстановлением из свежего дампа, с участием
основателя (деструктивный шаг — только после его «go»). Фактические тайминги
и находки фиксируются в runbook после репетиции. Побочная ценность: пока БД
лежит, алертер (Task 0018) обязан прислать ERR-OPS-001 в канал — репетиция
заодно проверяет тракт алертов на staging.

### Отступления от FOUNDATION (фиксируются честно)

- §10.10 «еженедельная проверка восстановления»: в Phase 0 — валидация
  каждого дампа `pg_restore --list` + разовая настоящая репетиция +
  квартальная повторная (вместе с аудитом конституции §13.7). Полностью
  автоматическая еженедельная репетиция — Phase 1 (нужен второй контур БД).
- §10.10 «цель RPO ≤ 1ч через WAL» — не в этой задаче: ежедневный дамп даёт
  RPO ≤ 24ч, этого достаточно до первого пилота (NG-8: не строить впрок).
- §11 «шифрование бэкапов»: дампы лежат нешифрованными на VPS и (offsite)
  на машине основателя — в Phase 0 в БД только демо-данные. До пилота с
  реальными данными гостей — шифровать дампы (решение вместе с хостингом
  прода и резидентностью данных, §11).

## План тестов

Сам smoke-набор и есть тесты задачи (карточка). Дополнительно: юнит-тестов
нет — в `tests/smoke/` нет логики приложения, только HTTP-клиент; `make check`
подтверждает, что smoke не попадает в юнит-прогон, mypy strict и ruff — что
код набора типизирован и чист. DoD-прогоны: `make smoke` локально зелёный;
репетиция восстановления на staging проведена и вписана в restore.md.
