# Spec 0021 — Обработка заявки после создания: язык гостя, ход персонала, частичное выполнение

- **Статус:** proposed (ревью до кода, R-2)
- **Дата:** 2026-07-18
- **Источник:** живой прогон на staging (гость пишет боту на казахском; заявка
  проходит new → … → done). Наблюдения зафиксированы в issues #77, #38, #75, #76.
- **Связанные решения:** ADR-013 (жизненный цикл без `assigned`, proposed),
  ADR-011 (доставка уведомлений), Task #38 заход 2а (дневной номер `#N`, PR #69),
  баг #71 (урок: два языка в одном LLM-вызове ненадёжны; отдельный вызов
  с одной задачей — надёжен), issue #66 (`default_language` — мёртвое поле).

Спека покрывает четыре проблемы одного пользовательского пути и режется на три
код-PR (см. «План реализации»). Общие ограничения: P-9 (смена статуса — действие
человека), P-11 (языки — конфигурация тенанта, не хардкод отеля), P-12 (один
канонический паттерн), P-6 (уведомления — подписчики событий), наблюдаемость P-10.

## Проблема 1 (#77): гость получает статусные сообщения не на своём языке

**Наблюдение.** Гость писал на казахском; при завершении заявки бот прислал
«Ваша заявка «305 бөлмені тазалау» выполнена. Спасибо! / Your request is done,
thank you!» — RU/EN-заглушка с казахской сутью внутри. Причина: константа
`GUEST_DONE_CONFIRMATION` в `channels/telegram/notifications.py` — двуязычный
хардкод, язык гостя системе неизвестен.

**Решение.**

1. **Язык — свойство заявки.** Колонка `service_requests.guest_language`
   (VARCHAR(2), NULL) — ISO 639-1 код языка, на котором гость просил заявку.
   Заполняется новым обязательным аргументом инструмента
   `create_service_request.guest_language` («код языка последнего сообщения
   гостя»). Модель определяет язык в момент, когда он достоверно известен;
   дальше он течёт по конвейеру детерминированно — без повторной детекции.
   Классификация кода языка — не «двуязычная генерация», ловушку бага #71 не
   задевает (проверяется evals, см. ниже). Невалидное значение от модели
   (не два латинских символа) не валит заявку: сохраняется NULL + warning-лог —
   заявка ценнее языковой метки (тот же принцип, что деградация перевода сути).
2. **Канонический текст + один перевод.** Уведомление гостю собирается на
   русском (канонический платформенный текст: шаблон статуса + `summary` +
   примечание персонала, см. Проблему 4). Целевой язык:
   `request.guest_language`, при NULL — `default_language` конфига тенанта
   (поле оживает, issue #66), при отсутствии конфига — «ru». Если целевой язык
   «ru» — отправляем канонический текст без LLM. Иначе — ОДИН вызов
   `ai.translation.translate_for_guest(text, language_code)` (промпт
   `translate_to_guest_v1`, версия в имени файла §7.5): единственная задача,
   единственный целевой язык — форма, надёжность которой замерена при фиксе #71.
3. **Деградация.** Ошибка провайдера (`ERR-AI-*`) — отправляем канонический
   русский текст с warning-логом: гость получает хоть и не идеальное, но
   связное сообщение, в котором его собственная формулировка (`summary`) — на
   его языке. Уведомление не пропадает никогда.
4. **Покрываемые сообщения.** Все места, где система пишет гостю о заявке сама:
   `done` (есть), `cancelled` (НОВОЕ: сейчас отменённая заявка исчезает молча —
   гость ждёт бесконечно), `done` с примечанием (Проблема 4). Канонические
   формулировки:
   - done: `Ваша заявка «{summary}» выполнена. Спасибо!`
   - done + примечание: `… выполнена.\nОт персонала: {note}`
   - cancelled: `К сожалению, вашу заявку «{summary}» пришлось отменить.`
     (+ `\nОт персонала: {note}` при наличии; о примечании — Проблема 4).
5. **Наблюдаемость.** Лог `guest_notified_*` дополняется полями
   `guest_language`, `translated` (bool), `translation_fallback` (код причины).

**Альтернативы (отклонены).**
- *Статический каталог локализаций (ru/kk/en).* Детерминированно и бесплатно,
  но гости пишут на любых языках (ADR-010) — LLM-путь нужен всё равно, а два
  механизма для одной задачи нарушают P-12. Каталог платформенных строк станет
  уместен вместе со слоем локализации (Phase 1, FOUNDATION §3.1.3) — спека ему
  не противоречит: канонический текст уже отделён от механизма доставки.
- *Детекция языка в момент уведомления по `summary`.* Ненадёжно для коротких
  или цифровых сутей («2 кофе», «305»), лишний вызов на каждое уведомление.
- *Язык на `Conversation`.* В Phase 0 у диалога нет владельца-модуля с языком;
  язык «на заявке» точнее: гость мог сменить язык между двумя заявками одного
  диалога — каждая уведомляется на языке, на котором её просили. С появлением
  `guests/` (Phase 1) язык гостя может стать `GuestFact` — колонка заявки
  останется снимком на момент создания.

**Влияние.** `modules/requests` (колонка + схемы), `ai/tools/create_service_request`
(контракт инструмента — расширение), `ai/translation.py` + новый промпт,
`channels/telegram/notifications.py`. Событие `request.status_changed` НЕ
расширяется — подписчик дочитывает заявку (прецедент #37/#38).

## Проблема 2 (#38, заходы 2b/2c): персоналу неудобно адресовать заявку

**Наблюдение.** Полный UUID в каждой команде; ERR-REQUESTS-003 от путаницы
(S-3 чек-листа staff-UX). Заход 2а (PR #69) уже дал дневной номер `#N` в
уведомлении и командах. Остались пункты 2 и 3 карточки #38: inline-кнопки и
команда реплаем.

**Решение.**

1. **Inline-кнопки под уведомлением о заявке** — ноль ручного ввода:
   - статус `new`: `[🏃 Взять в работу]` `[❌ Отменить]`;
   - статус `in_progress` (после «Взять»): `[✅ Готово]`
     `[⚠️ Готово частично]` `[❌ Отменить]`;
   - терминальный статус: клавиатура снимается, в чат уходит короткий итог
     («Заявка #5 «…» → done»).
   `callback_data` = `req:<uuid>:<start|done|done_note|cancel>` (46 байт < лимита
   64). Нажатие → `requests.change_request_status(...)` — та же карта переходов,
   что у команд (P-5, DoD #38); повторное/устаревшее нажатие → короткий тост
   `answerCallbackQuery` («уже в работе» — человекочитаемый ERR-REQUESTS-003).
   После перехода бот обновляет клавиатуру уведомления под новый статус
   (`editMessageReplyMarkup`, best-effort: сбой редактирования не откатывает
   переход — устаревшие кнопки безопасны, их гасит карта переходов).
2. **Команда реплаем на уведомление** (#38 п.3): `/start`, `/done [текст]`,
   `/cancel [текст]` ответом на сообщение-уведомление — заявка резолвится по
   `external_message_id` уведомления: исходящие уведомления уже хранятся в
   `messages` с ключом `staff:request_created:<request_id>` — обратный поиск
   по `external_message_id` даёт request_id без новой таблицы и без парсинга
   текста. Явный аргумент (`/done 5`) продолжает работать и имеет приоритет
   над реплаем.
3. **Идемпотентность** — общий путь канала: `update_id` вебхука (в т.ч. для
   `callback_query`) дедуплицируется существующим ключом
   `telegram:update:<update_id>` (P-8). Два разных нажатия — два update — второй
   гасится картой переходов, не дедупликацией.
4. **Кто нажал.** Telegram передаёт `from.id`/`username` — пишутся в
   структурный лог (`staff_command_applied.actor`). Привязка к `User`/RBAC —
   Phase 1 (§17.7), не здесь.

**Альтернативы (отклонены).**
- *Резолв реплая парсингом `#N` из текста уведомления* — хрупко (текст меняется,
  номер повторяется за сутки); ключ идемпотентности — уже готовый указатель.
- *Отдельная таблица «сообщение → заявка»* — не нужна, `messages` уже хранит
  и ключ, и `external_message_id`.

**Влияние.** `channels/telegram`: `schemas.py` (+`callback_query`),
`normalize.py` (+`MessageKind.CALLBACK`, поля `callback_id`/`actor_external_id` —
аддитивное расширение `NormalizedMessage`, P-7), `client.py` (порт: reply_markup,
`answer_callback_query`, `edit_message_reply_markup`), `staff.py`,
`notifications.py` (кнопки у уведомления), `store.py` (поиск заявки по
`external_message_id`). Домен не меняется.

## Проблема 3 (#75): статус `assigned` — лишний шаг

**Наблюдение.** Персонал не различает «назначено» и «в работе»; серия
ERR-REQUESTS-003 из-за пропуска `/assign`. Служба пилота — 1–2 человека.

**Решение.** Убрать `assigned` из жизненного цикла:
`new → in_progress → done/cancelled`. «Кто взял» в Phase 1 станет атрибутом
`assignee` (+его смена — обычное действие), а не проходным статусом. Полное
обоснование, варианты и последствия — **ADR-013** (в этом же PR); правка
FOUNDATION §5.2 — в PR реализации вместе со сменой статуса ADR на accepted.
P-9 сохраняется: `new → in_progress` — явное действие человека («Взять в
работу» / `/start`).

Механика: значение уходит из `RequestStatus` и `STATUS_TRANSITIONS`; миграция
данных `UPDATE … SET status='in_progress' WHERE status='assigned'` (колонка —
VARCHAR, `native_enum=False` — ALTER TYPE не нужен); `/assign` отвечает
подсказкой «этот шаг упразднён — используйте /start N» (переучивание, не
молчание); HTTP API перестаёт принимать `assigned` — осознанное исключение из
политики совместимости §13.5, зафиксировано в ADR-013 (Phase 0, внешних
потребителей API нет).

**Влияние.** `modules/requests` (enum, переходы, миграция, README),
`channels/telegram/staff.py`, FOUNDATION §5.2 (через ADR-013), тесты обоих.

## Проблема 4 (#76): частичное выполнение без обратной связи гостю

**Наблюдение.** «Полотенца принесли, кофе нет» — сообщить некуда, гость в
неведении.

**Решение.**

1. **Домен.** Колонка `service_requests.resolution_note` (VARCHAR(500), NULL) —
   примечание персонала к терминальному переходу (done/cancelled), по-русски
   (язык персонала). `change_request_status(request_id, new_status, *,
   resolution_note=None)`; сервис пишет примечание только на терминальных
   переходах (на прочих — игнорирует с warning-логом: контракт не расширяет
   карту переходов). Новый статус НЕ вводится: «частично» — свойство
   завершения, а не отдельная стадия; терминальная развилка умножила бы карту
   переходов, коды ошибок и путаницу персонала — ровно то, с чем боремся в
   Проблеме 3. HTTP API: опциональное поле `resolution_note` в теле
   `POST /{id}/status` (аддитивно, §13.5).
2. **Каналы ввода персонала** (оба — Проблема 2 их уже строит):
   - кнопка `[⚠️ Готово частично]` → бот задаёт вопрос с ForceReply
     («Заявка #5: что не сделано и почему? Ответьте на это сообщение»);
     ответ-реплай персонала (обычный текст) → `done` + примечание. Вопрос
     записывается с ключом `staff:note_prompt:<request_id>` — реплай резолвится
     тем же обратным поиском, что в Проблеме 2. Ожидание не персистится в
     отдельном состоянии: не ответили — заявка просто осталась in_progress,
     повторное нажатие задаёт вопрос снова;
   - хвост команды: `/done 5 кофе не принесли — закончился` (и `/cancel 5 …`).
3. **Гостю** примечание доставляется частью уведомления о завершении/отмене,
   переведённое на его язык вместе со всем текстом (Проблема 1, один вызов).
   В staff-чат итог дублируется по-русски («#5 → done. Примечание: …»).

**Альтернативы (отклонены).**
- *Статус `done_partial`* — см. выше: свойство, а не стадия; для отчётности
  Phase 1 достаточно `resolution_note IS NOT NULL`.
- *Свободный диалог с ботом («напишите, что не сделано» без reply)* — требует
  машины состояний staff-чата; ForceReply даёт то же UX без состояния.

**Влияние.** `modules/requests` (колонка + сервис + схемы + README),
`channels/telegram` (staff.py, notifications.py), HTTP API (аддитивно).

## План реализации (после merge #68 → #69 → #74)

| PR | Содержимое | Миграция | Закрывает |
|---|---|---|---|
| B | ADR-013 → accepted, FOUNDATION §5.2, enum/переходы, staff.py | 0011 (данные: assigned→in_progress) | #75 |
| C | `guest_language` + аргумент инструмента + `translate_for_guest` + промпт + уведомления done/cancelled на языке гостя | 0012 (колонка) | #77, #66 |
| D | callback_query сквозняком, кнопки, reply-команды, ForceReply, `resolution_note` | 0013 (колонка) | #38, #76 |

Порядок B → C → D (D строит клавиатуры уже без `assigned` и шлёт примечание
через механизм C). Каждый PR: тесты (юнит + сквозной walking-skeleton путь),
обновлённые README модулей, зелёный `make check`/CI (R-9).

**Тесты и evals.** Юнит-тесты — на mock-провайдере gateway (сценарные ответы).
Для аргумента `guest_language` — ассерт в bake-off (`ai/evals/bakeoff.py`):
модель заполняет код языка корректно на языках golden-набора (en/ru/kk/zh/tr/hi,
ADR-010) — тем же прогоном, что уже меряет создание заявки (#62).

**Вне скоупа спеки** (зафиксировано, чтобы не потерять): язык персонала как
конфиг тенанта (P-11, сейчас — платформенный дефолт «ru» в промптах перевода);
локализация ответов канала без LLM (`UNSUPPORTED_REPLY`/`DEGRADED_REPLY` —
осознанная двуязычная деградация §7.8 до слоя локализации Phase 1); уведомление
staff-чата о переходах, сделанных через HTTP API мимо чата (Phase 1, кабинет).
