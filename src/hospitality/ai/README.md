# ai/ — AI-слой: оркестратор, инструменты, промпты, evals (Task 0015/0017.1)

## Назначение

Композиционный слой (FOUNDATION §5.1, §7): превращает сообщение гостя в ответ
или действие ядра. Бизнес-логики не содержит (P-5): каждая способность AI —
тонкая обёртка над сервисом доменного модуля. Весь LLM-трафик — только через
`ai/gateway` (§7.2, см. его README).

## Состав

| Пакет/файл | Что даёт |
| --- | --- |
| `orchestrator.py` | §7.1: сообщение → gateway → гейт P-9 → исполнение инструмента → `OrchestratorTurn` |
| `translation.py` | Перевод сути заявки на язык персонала для staff-чата (баг #71): один вызов gateway, надёжнее «двух языков в одном tool-call» |

| `evals/` | Офлайн bake-off моделей на 6 языках пилота (§7.7, ADR-010; не CI) |
| `tests/` | Тесты оркестратора + golden set v0 (CI-гейт канона на Fake-провайдере) |

## Публичный API

- `orchestrator.handle_message(*, message, history=None, pending_action=None,
  provider=None) -> OrchestratorTurn` — единая точка обработки сообщения гостя;
  вызывается внутри `tenant_context(...)` (P-4). Историю диалога и
  `pending_action` хранит вызывающая сторона (канал; ADR-011 —
  `conversations.pending_action`).
- `OrchestratorTurn` (P-7): `kind` (`reply` / `awaiting_confirmation` /
  `action_done` / `needs_human`), `reply_text`, `pending_action?`,
  `created_request_id?`.
- Ошибки провайдера (ERR-AI-001/002/003) пробрасываются — деградация §7.8
  забота канала. Ошибка исполнения инструмента (ERR-AI-004 и т.п.) — исход
  `needs_human`, не 5xx.

## Гейт подтверждения P-9 (`confirm_guest`) — структурный, детерминированный

Класс подтверждения — свойство контракта инструмента (`ai/tools`), гейтит
оркестратор, не текст промпта. Два хода (spec 0017.1):

1. **Предложение.** Модель вызывает инструмент класса `confirm_guest` →
   оркестратор НЕ исполняет его: возвращает `awaiting_confirmation` +
   `pending_action` (`tool_name` + `arguments`). Канал сохраняет его между
   вебхуками и задаёт гостю подтверждающий вопрос.
2. **Ход подтверждения** (`pending_action` передан). Оркестратор не полагается
   на повторный tool_use модели (issue #31: Haiku ре-эмитит нестабильно —
   заявка молча терялась). Вместо этого — структурная классификация ответа
   гостя: один вызов gateway с единственным служебным инструментом
   `resolve_confirmation` и `forced_tool` (модель обязана вернуть вердикт,
   свободный текст невозможен). Дальше детерминированно:
   - `confirm` → исполнить **сохранённый** `pending_action` → `action_done`;
   - `decline` → ничего не исполнять → `reply`; гейт гаснет (канал очищает
     `pending_action` на всех исходах, кроме `awaiting_confirmation`);
   - `other` (передумал/правка/другая тема) → старое предложение снимается,
     сообщение обрабатывается как новый запрос (обычный ход).

   Реплику «готово/отменил» пишет классификатор на языке гостя (`reply`);
   текст «передал в службу» появляется только ПОСЛЕ фактического исполнения.
   `resolve_confirmation` — механизм гейта, не AI-способность: в реестре §7.3
   его нет.

## Промпты (§7.5)

- `concierge_v3.md` — активный системный промпт обычного хода
  (`orchestrator.PROMPT_NAME`). v3 закрыл баг #71: v2 учил модель, что вызов
  инструмента = отправка заявки службе («the request is only submitted after
  the guest confirms»), поэтому Haiku придерживал `tool_use` до «да» — а на ходе
  «да» `pending_action` уже не было, и заявка не создавалась **никогда** (на
  английском гейт не вооружался в 0/4 прогонов, kk 3/4). v3 переформулирует:
  вызов инструмента — лишь ЧЕРНОВИК заявки, ничего не отправляет (подтверждение и
  отправку делает система), поэтому модель обязана звать инструмент на том же
  ходу, где предлагает заявку. Замер на Haiku 4.5: v3 — 24/24 на 6 языках пилота.
- `concierge_v2.md` — предыдущая версия (баг #71, оставлена как история). v2
  закрыла два дефекта bake-off'а (DISCUSSION_LOG): промпт целиком на английском
  с жёстким правилом «отвечай на языке последнего сообщения гостя» первой строкой
  (утечка языка) и требованием формулировать предложение действия вопросом.
- `confirmation_gate_v1.md` — промпт классификатора хода подтверждения
  (`orchestrator.CONFIRMATION_PROMPT_NAME`).
- Смена промпта = новый файл `<name>_v<N+1>.md` + смена константы + прогон
  evals (§7.7). Старые версии остаются в репозитории как история.

## События

Не публикует и не потребляет доменных событий (события заявок публикует
`modules/requests` при исполнении инструмента).

## Таблицы

Своих нет. `llm_call_log` — у `gateway` (его README).

## Зависимости

Внутренние: `modules/requests` (публичный API — категории, создание заявки),
`shared` (errors, logging, tenancy). Направление слоёв (R-5): `ai/` зависит от
доменных модулей, обратное запрещено; каналы зависят от `ai/`.

## Типовые сценарии изменения

- **Новый AI-инструмент** — копировать канон `tools/create_service_request.py`
  (контракт: имя, Pydantic-вход, класс подтверждения P-9) + запись в
  `tools/registry.py` + тест. Инструмент класса `confirm_guest` автоматически
  проходит гейт — отдельного кода подтверждения писать не нужно.
- **Правка промпта** — новая версия файла + evals; см. §«Промпты».
- **Смена модели рантайма** — `LLM_MODEL` + прайс-лист gateway; выбор — только
  по bake-off (`python -m hospitality.ai.evals.bakeoff`, нужен
  `ANTHROPIC_API_KEY`).
