# Smoke-набор (Task 0019)

Инструмент приёмки основателя (FOUNDATION §13.6): сквозные сценарии на языке
бизнеса против **живой** среды. Результат читается без чтения кода:

```
✅ Гость просит уборку → заявка появляется у службы (11.8 с)
✅ Гость задаёт вопрос → получает ответ, лишняя заявка не создаётся (4.1 с)
✅ Дубликат вебхука → заявка не дублируется (0.6 с)

Итог: система жива, сценарии гостя работают.
```

## Запуск

| Команда | Против чего | Что нужно заранее |
|---|---|---|
| `make smoke` | локальная среда | `make dev`, `make migrate`, `make seed`; в `.env` — `ANTHROPIC_API_KEY` и `TELEGRAM_WEBHOOK_SECRET` |
| `make smoke-staging` | staging | `STAGING_HOST`/`STAGING_SSH_KEY` в `.env`; секреты подтянутся по SSH |
| CI job `smoke` | compose-стек в CI | секрет `ANTHROPIC_API_KEY` в GitHub Actions (нет секрета → job пропускается) |

Полный вывод с трейсами (для разработки): `.venv/bin/pytest tests/smoke -m smoke --no-cov`.

## Устройство

- **Чёрный ящик:** тесты не импортируют пакет `hospitality` — только HTTP
  (вебхук Telegram, API заявок, `/metrics`, `/health/ready`), поэтому один
  набор работает против любой среды (spec `docs/specs/0019-smoke-and-restore.md`).
- **Настоящая модель:** LLM не подменяется (ADR-007). Прогон стоит 3–4 вызова
  Haiku. Сценарии терпимы к свободе модели (одно повторное подтверждение),
  но не имитируют её.
- **Следы в среде:** smoke-заявки отменяются сразу после проверки; на staging
  в staff-чат при этом штатно приходит уведомление о (тут же отменённой)
  заявке. Диалоги гостя остаются в БД — по одному синтетическому чату на прогон.
- В юнит-прогон (`make check`) набор не входит: маркер `smoke` отфильтрован
  в `addopts` (pyproject.toml).

## Ограничение v0

Текст ответа гостю снаружи системы не наблюдаем (уходит в Telegram-чат);
«гость получил ответ» проверяется по успешному вызову модели в `/metrics`.
Полная проверка текста — когда появится API диалогов (кабинет персонала, Phase 1).
